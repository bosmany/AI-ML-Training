Chapter 31:
Model Deployment

A model sitting in a notebook helps nobody. Deployment turns it into something other code (or other people) can actually call: an API endpoint with validated inputs, packaged so it runs identically on any machine, with a way to save/reload the trained model itself. FastAPI and Docker are the industry-standard tools for this — this chapter reads their code (real syntax, straight from production practice) and runs the parts that don't need a real server or container: request validation logic, model serialization, and health/metrics tracking.

🕐 ~55 min read 💻 5 exercises 🏗️ 1 mini project 🧪 Quiz to complete
31.1 — Why Deploy? From Notebook to Production

A trained model is just a function: input in, prediction out. "Deploying" it means wrapping that function behind something other systems can call reliably — over HTTP, on a schedule, or embedded in an app — with validated inputs (garbage in, garbage predictions out, silently), a stable environment (the exact library versions it was trained with), and observability (is it actually still working, right now, in production).

31.2 — Building an API With FastAPI

FastAPI turns a Python function into an HTTP endpoint with a decorator, and uses Pydantic models to validate incoming request data automatically — malformed input gets rejected with a clear error before your prediction code ever runs.

🎬
Reference code — FastAPI can't run inside this browser sandbox
Serving real HTTP requests needs a real network socket, which a WASM sandbox doesn't have. Read this for the real syntax; the runnable exercises below reproduce the VALIDATION LOGIC in plain Python so you can practice the concept here.
Terminal / real script — reference only, no Run button
from fastapi import FastAPI from pydantic import BaseModel app = FastAPI() class CandidateRequest(BaseModel): # Pydantic validates types automatically score: float city: str @app.post('/predict') def predict(request: CandidateRequest): prediction = 'qualified' if request.score >= 0.8 else 'not qualified' return {'prediction': prediction} # run with: uvicorn main:app --reload # then: curl -X POST localhost:8000/predict -d '{"score":0.9,"city":"Hubli"}'

The exact same VALIDATION IDEA, without needing FastAPI installed — this is what actually runs below:

The same validation logic, plain Python
def predict_request_handler(data): required = ['score', 'city'] missing = [k for k in required if k not in data] if missing: return {'status': 400, 'error': f'Missing fields: {missing}'} if not isinstance(data['score'], (int, float)): return {'status': 400, 'error': 'score must be numeric'} prediction = 'qualified' if data['score'] >= 0.8 else 'not qualified' return {'status': 200, 'prediction': prediction} print(predict_request_handler({'score': 0.92, 'city': 'Hubli'})) print(predict_request_handler({'city': 'Pune'})) # missing 'score' -> validation error
31.3 — Docker: The Same Environment, Everywhere

"Works on my machine" usually means a specific combination of OS, Python version, and library versions that isn't guaranteed anywhere else. A Dockerfile scripts the ENTIRE environment — base image, dependencies, code — into a single reproducible artifact ("image") that runs identically on your laptop, a teammate's laptop, and a production server.

Terminal / real project — reference only, no Run button
# Dockerfile FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install -r requirements.txt # Chapter 6.1's requirements.txt, inside the container COPY . . CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"] # build and run: # docker build -t candidate-api . # docker run -p 8000:8000 candidate-api
31.4 — Model Serialization: Saving the Trained Object Itself

Training happens once (or on a schedule); serving happens on every request. pickle (stdlib) or joblib (faster for large NumPy arrays, common for sklearn models) save a trained Python object to a file, and load it back byte-for-byte identical — no retraining needed to serve predictions.

Serializing and restoring a model with pickle
import pickle model_stub = {'weights': [0.5, -0.3, 0.2], 'bias': 0.1, 'name': 'toy_model'} serialized = pickle.dumps(model_stub) # -> bytes, ready to write to a .pkl file restored = pickle.loads(serialized) # bytes -> the exact same object back print(f'Restored name: {restored["name"]}') print(f'Round-trip identical: {restored == model_stub}')
🚫
Never unpickle data from an untrusted source — unlike JSON, unpickling can execute arbitrary code embedded in the byte stream. Only load pickle files you (or your own training pipeline) created.
31.5 — Health Checks & Basic Monitoring

A deployed model needs a /health endpoint (a real production system pings it every few seconds to know if the service is alive) and basic metrics — request count, error rate, latency — so a silent failure doesn't go unnoticed for days.

A minimal health/metrics tracker
class HealthMonitor: def __init__(self): self.request_count = 0 self.error_count = 0 self.latencies = [] def log_request(self, latency_ms, is_error=False): self.request_count += 1 if is_error: self.error_count += 1 self.latencies.append(latency_ms) def health_report(self): avg_latency = sum(self.latencies) / len(self.latencies) if self.latencies else 0 error_rate = self.error_count / self.request_count if self.request_count else 0 return { 'status': 'healthy' if error_rate < 0.1 else 'degraded', 'requests': self.request_count, 'error_rate': error_rate, 'avg_latency_ms': avg_latency, } mon = HealthMonitor() mon.log_request(120); mon.log_request(95); mon.log_request(200, is_error=True) report = mon.health_report() print(f'Status: {report["status"]}') print(f'Error rate: {report["error_rate"]:.4f}')
Chapter 31 Summary
  • FastAPI + Pydantic turn a Python function into a validated HTTP endpoint.
  • Docker packages the entire runtime environment into one reproducible artifact.
  • pickle/joblib save a trained model object to disk and restore it byte-for-byte, without retraining.
  • Health checks and metrics (request count, error rate, latency) catch silent production failures.
