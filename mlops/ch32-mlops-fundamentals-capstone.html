Chapter 32:
MLOps Fundamentals

This is the Module 7 — and core curriculum — capstone. A model that's deployed once and never touched again silently rots: the world changes, the data changes, and accuracy quietly degrades. MLOps is the discipline of treating a model like a living system — tracked experiments so you know what you actually trained, drift detection so you know when reality has moved on, and CI/CD so retraining and redeploying is a routine, not an emergency.

🕐 ~60 min read 💻 5 exercises 🏗️ Core Curriculum Capstone 🧪 Quiz to complete 💼 Interview Prep included
32.1 — Experiment Tracking

Training a model 20 different ways (different features, hyperparameters, random seeds) without recording what you tried is how "which run was the good one?" becomes unanswerable a week later. MLflow is the standard tool: log parameters, metrics, and the model artifact itself for every run, then compare runs in a UI.

Terminal / real project — reference only, no Run button
import mlflow with mlflow.start_run(): mlflow.log_param('n_estimators', 100) mlflow.log_param('max_depth', 5) # ... train the model ... mlflow.log_metric('accuracy', 0.87) mlflow.sklearn.log_model(model, 'model') # mlflow ui -- opens a local dashboard comparing every run you've logged
32.2 — Model Versioning & Registries

A model registry tracks which exact model artifact is in Staging, which is in Production, and every version that came before — so "roll back to last week's model" is a one-line operation, not a scramble to find an old pickle file on someone's laptop.

32.3 — Drift Detection: Has the World Changed?

A model trained on last year's data can quietly become wrong as user behavior, prices, or populations shift — data drift. A simple, real detector: compare the production data's mean to the training data's mean, in units of the training data's own standard deviation. A shift of many standard deviations is a strong signal something changed.

A simple mean-shift drift detector
import numpy as np training_scores = np.array([0.72, 0.75, 0.68, 0.80, 0.77, 0.74, 0.71, 0.79]) production_scores = np.array([0.55, 0.60, 0.58, 0.62, 0.57, 0.61, 0.59, 0.56]) train_mean, train_std = training_scores.mean(), training_scores.std() prod_mean = production_scores.mean() mean_shift = abs(prod_mean - train_mean) / train_std # shift, in training std-devs print(f'Mean shift (in training std devs): {mean_shift:.4f}') print(f'Drift detected: {mean_shift > 1.0}')
⭐
This is the same z-score idea from Chapter 9, applied to a whole distribution's mean instead of a single value. Real production systems use more rigorous statistical tests (Kolmogorov-Smirnov, population stability index) — but the core idea is identical: is this new data more than a few standard deviations from what the model was trained on?
32.4 — CI/CD for ML: Automated Retraining

Traditional CI/CD runs tests and deploys code on every push. ML CI/CD adds a step: on a schedule (or when drift crosses a threshold), automatically retrain on fresh data, re-evaluate against a held-out set, and only promote the new model to production if it beats the current one.

Terminal / real project — reference only, no Run button
# .github/workflows/retrain.yml name: Weekly Retrain on: schedule: - cron: '0 0 * * 0' # every Sunday at midnight jobs: retrain: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: pip install -r requirements.txt - run: python train.py - run: python evaluate.py --gate-on-accuracy 0.85 # fails the job if the new model is worse - run: python promote_to_registry.py

💼 MLOps & Production ML Interview Prep

12 questions commonly asked in ML engineer / MLOps interviews, covering everything in this chapter plus the deployment concepts from Chapter 31. Click a question to reveal the answer. This section isn't graded — it's here so you walk into an interview room prepared, not just quizzed.

What is MLOps, and how is it different from regular DevOps?▶
MLOps applies DevOps practices (CI/CD, automation, monitoring) to the ML lifecycle, but adds concerns DevOps doesn't have: the model's behavior depends on DATA as well as code, so you also need to version datasets, track experiments, detect data/concept drift, and periodically retrain — a correctly-deployed model can still silently become wrong as the world changes, which isn't true of ordinary deployed code.
What's the difference between data drift and concept drift?▶
Data drift: the distribution of INPUT features changes (e.g. average customer age shifts), but the relationship between inputs and the correct output stays the same. Concept drift: the relationship itself changes (e.g. what used to predict "fraud" no longer does, because fraud patterns evolved) — the same input now warrants a different output. Concept drift is generally harder to detect and more damaging.
Why track experiments instead of just remembering your best model?▶
Because "remembering" doesn't scale past a handful of runs, isn't reproducible by a teammate, and gives you no audit trail when a stakeholder asks "why did we pick this model." Tools like MLflow or Weights & Biases log every run's hyperparameters, metrics, code version, and the resulting artifact, so any run can be compared or exactly reproduced later.
What is a feature store and why would a team adopt one?▶
A feature store is a centralized system for computing, storing, and serving features consistently to both training and serving pipelines. Without one, teams often compute features slightly differently in training (batch, offline) vs. serving (real-time, online) — causing "training-serving skew," where a model behaves differently in production than its offline evaluation predicted.
What is training-serving skew, and how do you prevent it?▶
It's the gap between a model's offline evaluation performance and its real production performance, usually caused by inconsistencies between how features were computed during training vs. how they're computed live at inference time (different code paths, different data freshness, different preprocessing). Prevented by sharing the exact same feature-computation code/logic (or a feature store) between training and serving, and by monitoring production performance against the offline baseline continuously.
Explain canary deployment and blue-green deployment for ML models.▶
Canary: route a small percentage of production traffic (e.g. 5%) to the new model, monitor its metrics against the old model, and gradually increase traffic if it performs well — limits the blast radius of a bad model. Blue-green: run the new model ("green") fully in parallel with the current one ("blue"), then switch all traffic over at once, keeping "blue" ready for an instant rollback if something's wrong.
What metrics would you monitor for a model already in production?▶
Two categories: (1) system health — request volume, latency, error rate, uptime (same as any deployed service); (2) model health — prediction distribution shifts, input feature drift, and — when ground-truth labels eventually arrive (e.g. did the customer actually churn) — the real-world accuracy/precision/recall, compared against the offline evaluation numbers.
What is A/B testing for ML models, and how does it differ from a typical product A/B test?▶
Same core idea — split traffic between model A and model B, measure a business metric (not just offline accuracy) for each, and pick the winner with statistical confidence. The ML-specific wrinkle: you often need to watch BOTH the business metric AND model-specific metrics (drift, latency, confidence calibration) simultaneously, since a model can win on the business metric short-term while degrading in ways that only show up over more time or more data.
How would you design a CI/CD pipeline for a retraining ML model?▶
On a schedule or drift-triggered event: pull fresh data → run the training script → evaluate against a fixed, held-out validation set → GATE the pipeline on the new model beating the current production model on the key metric (fail the job otherwise) → if it passes, register the new model version and promote it (often via canary rollout, not an instant full swap) → keep monitoring afterward.
How do you handle rolling back a bad model deployment?▶
This is exactly what a model registry and versioned deployments are for: keep the previous production model readily available (not deleted), route traffic back to it (a config change, not a retrain), and only then investigate what went wrong with the new version. Rollback should be measured in minutes, not "let's retrain and redeploy," which is why blue-green/canary patterns matter.
Why can't you just retrain a model constantly instead of monitoring for drift first?▶
Retraining costs compute, time, and carries its own risk (a new model can regress in ways the old one didn't, especially if recent data is noisy or has quality issues). Monitoring for drift and gating retraining on it (or on a schedule with an evaluation gate) targets retraining at when it's actually needed, and forces every new model to prove it's better before replacing a working one.
What's the difference between online (real-time) and batch inference, and when would you use each?▶
Online inference: a live request comes in (e.g. a user loads a page) and the model predicts in real time within the response — needs low latency, usually served behind an API. Batch inference: predictions are computed for many inputs at once on a schedule (e.g. nightly), stored, and looked up later — no latency pressure, much more compute-efficient per-prediction. Use online when the input isn't known until request time (fraud check on this specific transaction); use batch when you can precompute (tomorrow's recommended products for every user).
