Systems Fundamentals 1:
APIs & HTTP Fundamentals

You've called APIs throughout this whole course — requests.get(...), an LLM API, a REST endpoint. This chapter goes one level deeper: what an API actually IS underneath the convenience wrapper, what every HTTP request/response is actually made of, why status codes aren't just a UI detail, and how to inspect a real API call happening on the wire instead of just trusting that it worked. This is the exact vocabulary interviewers use to check whether you've operated real systems, not just imported a client library.

🕐 ~60 min read 💻 5 exercises 🏗️ Capstone Project 🧪 5-question quiz 💼 12 Interview Prep Q&A
1.1 — What an API Actually Is: A Contract, Not Magic

An API (Application Programming Interface) is an agreed-upon interface that lets one piece of software ask another to do something, without either side needing to know how the other is built inside. A REST API — the dominant style for web APIs — organizes that interface around resources (nouns: a candidate, an order, a document) identified by a URL path, and lets the HTTP method carry the verb: what you want done to that resource. The whole exchange is a request/response cycle — a client sends a request (method + path + headers + optional body), the server sends back a response (status code + optional body) — and NOTHING happens outside that cycle. There's no hidden magic: it's a contract both sides agree to honor.

Modeling a request and a response as plain Python dicts
def make_request(method, path, headers=None, body=None): return {'method': method, 'path': path, 'headers': headers or {}, 'body': body} def make_response(status, body=None): return {'status': status, 'body': body} # A client wants the candidate with id 5 req = make_request('GET', '/candidates/5') # The server looks it up and responds res = make_response(200, {'id': 5, 'name': 'Priya', 'score': 0.91}) print(f"Request: {req['method']} {req['path']}") print(f"Response: {res['status']} {res['body']}")
🧠
Why "contract" is the right word. The provider promises that a given request shape produces a given response shape. As long as both sides honor it, either side's internals can be rewritten completely without breaking the other. Break the contract silently (rename a field, change what a status code means) and every client relying on it breaks too — which is exactly why API versioning (/v1/, /v2/) exists.
1.2 — HTTP Methods & Idempotency

Five methods cover almost everything: GET (read), POST (create / trigger an action), PUT (replace an entire resource), PATCH (partially update a resource), DELETE (remove). The property that matters most in practice is idempotency: an idempotent request produces the same end state no matter how many times it's repeated. This course's Chapter 7 introduced this exact definition for a reason — it's the single most useful lens for reasoning about retries, which is the whole subject of Systems Fundamentals 2.

MethodSafe (no side effects)?Idempotent?Typical use
GETYesYesRetrieve a resource
POSTNoNo (usually)Create a new resource / trigger an action
PUTNoYesReplace a resource entirely
PATCHNoDepends on the update (often not)Partially update a resource
DELETENoYesRemove a resource
Idempotent PUT vs non-idempotent POST — same repeat, different end state
db = {} next_id = [1] def handle_put(resource_id, body): db[resource_id] = body # last write wins; repeating this keeps the SAME end state return 204 def handle_post(collection, body): new_id = next_id[0] next_id[0] += 1 db[new_id] = body # every call creates a NEW record return 201, new_id # PUT /users/5 {'name': 'Alice'} sent twice (e.g. the first response got lost) handle_put(5, {'name': 'Alice'}) handle_put(5, {'name': 'Alice'}) print(f'After 2x PUT: {db}') db.clear(); next_id[0] = 1 # POST /users {'name': 'Alice'} sent twice — same intent, very different result _, id1 = handle_post('users', {'name': 'Alice'}) _, id2 = handle_post('users', {'name': 'Alice'}) print(f'After 2x POST: {db}') print(f'Idempotent PUT gave one record; non-idempotent POST gave two: ids {id1} and {id2}')
⚠️
This is why blind retries are dangerous on plain POST. If a client doesn't get a response (network blip) and retries a PUT, worst case the server does identical work twice — no harm done. Retry a bare POST the same way and you can double-charge a card or double-send an email. Real systems fix this with an idempotency key (a client-generated unique ID attached to the request) so the server can recognize and safely ignore a duplicate POST — more on this in SF 2.
1.3 — HTTP Status Codes: The Real Taxonomy

Status codes aren't decoration — they're the single most information-dense part of a response. Five ranges: 2xx success, 3xx redirect, 4xx the client did something wrong, 5xx the server did something wrong. Memorizing the RANGE is easy; the interview-caliber skill is knowing the specific codes that actually show up in practice and the differences interviewers use to probe for real understanding.

CodeNameWhat it actually means
200OKSuccess, here's the data
201CreatedSuccess, a new resource now exists
204No ContentSuccess, nothing to return (common for DELETE/PUT)
301 / 302Moved Permanently / FoundGo to a different URL — 301 permanently (update bookmarks/caches), 302 just for now
400Bad RequestThe request itself is malformed (bad JSON, wrong types)
401Unauthorized"I don't know who you are" — missing/invalid credentials
403Forbidden"I know who you are, but you can't do this"

409ConflictRequest conflicts with the resource's current state (e.g. duplicate create)
422Unprocessable EntityWell-formed request, but the data fails validation rules
429Too Many RequestsRate-limited — slow down, often with a Retry-After header

504Gateway TimeoutProxy waited for upstream and got NO response in time
⭐
The pairs interviewers actually probe: 401 vs 403 — 401 is an authentication failure ("who are you?"), 403 is an authorization failure ("I know who you are, you're just not allowed"). 502 vs 503 vs 504 — 502 means the gateway got a broken response FROM upstream; 503 means upstream is deliberately/currently unable to serve (often paired with Retry-After); 504 means the gateway got NO response at all before its own timeout expired. All three sound like "something's broken upstream" but point a debugging session in three completely different directions.
A status-code classifier — the range logic, made concrete
def classify_status(code): if 200 <= code < 300: return 'success' if 300 <= code < 400: return 'redirect' if 400 <= code < 500: return 'client error' if 500 <= code < 600: return 'server error' return 'unknown' for code in [200, 201, 301, 400, 401, 403, 404, 429, 500, 502, 503, 504]: print(f'{code}: {classify_status(code)}')
1.4 — Headers: Content-Type, Authorization, Accept

Headers are metadata about the request or response, separate from the body. Three you'll touch constantly: Content-Type (what format the BODY you're sending is — e.g. application/json), Accept (what format you want the response IN), and Authorization (your credentials, e.g. Bearer <token>). A shocking number of real API bugs are "the body was fine but a header was missing" — which is exactly why building and validating request headers as data is worth practicing directly.

Building and validating a request dict with proper headers
def build_request(method, path, content_type=None, auth_token=None, body=None): headers = {'Accept': 'application/json'} if content_type: headers['Content-Type'] = content_type if auth_token: headers['Authorization'] = f'Bearer {auth_token}' return {'method': method, 'path': path, 'headers': headers, 'body': body} def validate_request(req): errors = [] if req['method'] in ('POST', 'PUT', 'PATCH') and 'Content-Type' not in req['headers']: errors.append('Missing Content-Type for a body-carrying request') if 'Authorization' not in req['headers']: errors.append('Missing Authorization header') return len(errors) == 0, errors req1 = build_request('POST', '/candidates', content_type='application/json', auth_token='abc123', body={'name': 'Sam'}) ok1, errs1 = validate_request(req1) print(f'Request 1 headers: {req1["headers"]}') print(f'Request 1 valid: {ok1}') req2 = build_request('POST', '/candidates', body={'name': 'Sam'}) ok2, errs2 = validate_request(req2) print(f'Request 2 valid: {ok2}, errors: {errs2}')
1.5 — Inspecting a Real API Call: curl -v and Browser DevTools

Everything above is the mental model. Here's how you actually LOOK at a real request on the wire, on your own machine, outside this browser sandbox.

🎬
Reference only — a real outbound network call can't run inside this browser sandbox. No real socket, no real internet the way a terminal has. Read this to know exactly what to type and what to look at; try it for real in your own terminal.
Terminal / real command — reference only, no Run button
$ curl -v https://api.example.com/candidates/5 * Trying 93.184.216.34:443... * Connected to api.example.com (93.184.216.34) port 443 > GET /candidates/5 HTTP/1.1 > Host: api.example.com > User-Agent: curl/8.4.0 > Accept: */* > < HTTP/1.1 200 OK < Content-Type: application/json < Content-Length: 87 < {"id":5,"name":"Priya","score":0.91}

-v (verbose) is the flag that makes curl show its work. Lines starting with * are curl's OWN connection diagnostics (DNS resolution, TCP connect, TLS handshake) — not part of the HTTP exchange itself. Lines starting with > are the raw request curl actually SENT: the request line, then every header, byte for byte. Lines starting with < are the raw response curl RECEIVED: the status line, then every response header. The blank line after the last header marks the end of headers; whatever follows is the body. A huge class of API bugs are "I thought I sent header X" vs "the server says it never saw header X" — curl -v settles that argument by showing you the literal bytes that went out, not what your client code claims it's doing.

🧠
Other curl flags worth knowing: -X POST sets the method, -H "Authorization: Bearer xyz" adds a header (repeatable), -d '{"name":"Sam"}' sends a body (and implicitly switches the method to POST if you didn't set one), -i shows response headers without the full verbose connection log.
Terminal / real command — reference only, no Run button
$ curl -X POST https://api.example.com/candidates \ -H "Content-Type: application/json" \ -H "Authorization: Bearer abc123" \ -d '{"name": "Sam", "score": 0.87}'

Browser DevTools gives you the same view for anything the browser itself sent. Open it with F12 (or right-click a page → Inspect), click the Network tab, then reload the page or trigger the action you're debugging — every request the page makes appears in the list as it happens. Click one to see: the Headers sub-tab (request URL, method, status code, all request/response headers), Payload/Request (the body you sent), Response/Preview (the body you got back, pretty-printed if it's JSON), and Timing (a breakdown of DNS lookup, connection, "Waiting (TTFB)" — how long the server took to start responding — and content download). A slow TTFB points at a slow backend; a slow content-download on a small payload points at the network, not the server.

1.6 — Auth Basics: API Keys, Bearer Tokens & Decoding a JWT

An API key is typically a long-lived, static credential identifying WHICH application is calling — common for server-to-server integrations. A Bearer token (often a JWT — JSON Web Token) usually identifies a specific logged-in user's session, is short-lived by design, and gets refreshed regularly. Both travel the same way, in the Authorization header, which is why the header's exact format (Bearer <token> vs a custom X-API-Key:) is what actually tells you which scheme is in play.

A JWT has three dot-separated, base64url-encoded parts: header.payload.signature. The header and payload are just base64url-encoded JSON — readable by anyone, with no secret required — while the signature is what an issuer's secret/public key can use to prove the token wasn't tampered with. This is genuinely decodable with nothing but the Python standard library:

Decoding a JWT's header and payload — stdlib only, no crypto
import base64 import json # A sample JWT — the signature below is a fake/dummy value, not a real HMAC. # We are decoding structure here, NOT verifying authenticity. token = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsZXggUml2ZXJhIiwicm9sZSI6ImNhbmRpZGF0ZSIsImlhdCI6MTcwMDAwMDAwMCwiZXhwIjoxNzAwMDAzNjAwfQ.dGhpc19pc19hX2Zha2Vfc2lnbmF0dXJlX25vdF9yZWFs" def decode_jwt_part(segment): # JWT base64url segments have their '=' padding stripped; add it back before decoding padded = segment + '=' * (-len(segment) % 4) decoded_bytes = base64.urlsafe_b64decode(padded) return json.loads(decoded_bytes) header_b64, payload_b64, signature_b64 = token.split('.') header = decode_jwt_part(header_b64) payload = decode_jwt_part(payload_b64) print(f'Header: {header}') print(f'Payload: {payload}') print(f'Algorithm: {header["alg"]}') print(f'Subject: {payload["sub"]}') print(f'Role: {payload["role"]}')
🚫
Decoding is NOT verifying. Anyone can decode a JWT — it's encoded, not encrypted. The signature is what proves the claims haven't been tampered with, and checking it requires the issuer's secret (HMAC) or public key (RSA/EC), which the code above never touches. Never put genuinely secret data inside a JWT payload assuming it's hidden — it isn't. And never trust a JWT's claims for anything security-sensitive (like "role": "admin") until the signature has actually been cryptographically verified against the issuer's key.
Systems Fundamentals 1 Summary
  • An API is a contract: a request shape promises a response shape, and REST models that around resources (nouns) with the HTTP method carrying the verb.
  • Idempotency — same end state no matter how many repeats — determines whether blindly retrying a failed request is safe. GET/PUT/DELETE are idempotent by convention; plain POST usually isn't.
  • Status codes are diagnostic, not decorative — 401 vs 403 (authentication vs authorization) and 502 vs 503 vs 504 (bad response / overloaded / no response in time) point a debugging session in different directions.
  • Headers (Content-Type, Accept, Authorization) carry metadata the body doesn't — a missing header is one of the most common silent API bugs.
  • curl -v and DevTools' Network tab show you the literal bytes on the wire — the fastest way to settle "what did I actually send / actually get back."
  • A JWT's header and payload are just readable, unencrypted JSON — decoding them proves nothing about authenticity; only signature verification does.

💼 APIs & HTTP Fundamentals — Interview Prep

12 questions covering the exact angles interviewers use to check whether you've actually operated real APIs, not just called requests.get() from a tutorial. Click a question to reveal the answer.

What is an API, in one sentence, and why do people call it a "contract"?▶
An API is an agreed-upon interface that lets one piece of software ask another to do something, without either side needing to know how the other is implemented internally. It's called a "contract" because both sides commit to it: the provider promises a given request shape (method, path, headers, body) will produce a given response shape (status code, body structure), and as long as both sides honor that, either side's internals can change completely without breaking the other. Break the contract silently (rename a field, change what a status code means) and every client relying on it breaks too — which is exactly why API versioning exists.
What's the actual difference between PUT and PATCH, beyond "PUT replaces, PATCH updates"?▶
PUT is defined to replace the ENTIRE resource with whatever you send — a strict implementation should remove any field the resource had before that you didn't include. PATCH is a PARTIAL update — you send only what changed, and everything else stays as it was. That's why PUT is idempotent by definition (sending the same full replacement twice yields the same end state), while PATCH's idempotency depends entirely on how it's implemented: "set status to active" is idempotent, but "increment view_count by 1" as a PATCH body is NOT — applying it twice doubles the increment. Naming that idempotency gap explicitly is a stronger answer than just reciting "partial vs full."
Why is idempotency such a big deal for anything that might get retried?▶
Networks fail silently and often: a client sends a request, the server processes it successfully, and then the RESPONSE gets lost on the way back — from the client's point of view that looks identical to the request never arriving at all. If the operation is idempotent, blindly retrying is always safe: worst case the server does the same work twice and lands in the same state either way. If it's NOT idempotent (a bare POST that charges a card, sends an email, increments a counter), a naive retry can cause a real duplicate side effect — which is exactly why production systems use an idempotency key (a client-generated unique ID attached to the request) on non-idempotent endpoints, rather than just trusting the HTTP method.
You get a 401 on a request you're sure has a valid API key attached correctly. What's actually going wrong, and how is that different from a 403?▶
If you're genuinely sending valid credentials and still getting 401, the common real causes are: the header name/format is wrong (missing the "Bearer " prefix, or using a header the server doesn't check), the credential has expired, or it's not actually reaching the server on this specific request (lost across a redirect, or a proxy stripping it). 401 always means "the server doesn't know who you are, or doesn't believe your claimed identity" — an authentication problem. 403 means the opposite: the server knows exactly who you are, but that identity isn't allowed to do this specific thing — an authorization problem. Conflating the two in a debugging conversation is a fast way to signal you don't actually understand the auth flow.
Your service calls another service through a load balancer/proxy and gets a 502. What does that specifically tell you, versus a 503 or a 504?▶
502 Bad Gateway means the proxy DID get a response from the upstream service, but that response was malformed, empty, or otherwise not valid HTTP it could forward — usually points at the upstream crashing mid-response or returning garbage, not at being unreachable. 503 Service Unavailable means the service is deliberately or currently unable to handle the request (overloaded, in maintenance, failing its own health check) — often paired with a Retry-After header. 504 Gateway Timeout means the proxy sent the request upstream and waited its configured timeout, but got NO response at all — the upstream might still be working, just too slowly. All three read as "something's broken upstream," but they send a debugging session in three different directions: 502 → check upstream's own crash logs, 503 → check upstream capacity/health, 504 → check upstream latency and the proxy's timeout setting.
What's the difference between curl's > and < lines in verbose mode, and why does that distinction matter when debugging?▶
In curl -v output, > lines are the raw request curl actually SENT — the request line then every header, exactly as bytes went out. < lines are the raw response curl RECEIVED — the status line then every response header. * lines are curl's own connection diagnostics (DNS, TCP connect, TLS handshake) and aren't part of the HTTP exchange at all. The distinction matters because a huge class of API bugs are "I thought I sent header X" vs "the server says it never saw header X" — verbose curl output settles that definitively by showing the literal bytes, instead of trusting what your client code claims it's doing.
In the browser DevTools Network tab, where do you look to find out how long a request actually took, broken down by phase?▶
Open DevTools (F12 or right-click → Inspect), go to Network, trigger the request, then click it in the list. The Timing sub-tab breaks total time into phases — DNS lookup, connection/TLS setup, "Waiting (TTFB — Time To First Byte)" which is how long the SERVER took to start responding, and Content Download which is how long the body took to arrive. A high TTFB points at a slow backend or database query; a high Content Download time on a small payload points at network conditions, not the server.
What's the actual difference between an API key and a Bearer token?▶
An API key is typically a long-lived, static credential issued once per application/project — it identifies WHICH application is calling, and usually doesn't expire or rotate automatically. A Bearer token (commonly a JWT or OAuth access token) usually identifies a specific user's authenticated session, is short-lived by design, and is meant to be refreshed rather than hard-coded. In practice: API keys are common for server-to-server or third-party integrations; Bearer tokens are common for logged-in-user flows. Both travel the same way — in the Authorization header — which is exactly why the header's format is what tells you which scheme is actually in play.
If you can base64-decode a JWT's payload without needing any secret, is the JWT insecure? Why or why not?▶
No — that's expected, not a flaw. A JWT's header and payload are base64url-ENCODED, not encrypted, so anyone holding the token can read the claims; the security guarantee a JWT provides is INTEGRITY, not secrecy. The signature (the third segment) lets a party who knows the issuer's secret (HMAC) or public key (RSA/EC) verify the header+payload haven't been tampered with since issuance. The practical consequence: never put genuinely secret data in a JWT payload (it's world-readable to anyone who intercepts it), and never trust a JWT's claims for anything security-sensitive until you've verified the signature — decoding it and seeing role: admin proves nothing if the signature was never checked.
Why does response.raise_for_status() matter, and what does skipping it actually cost you?▶
Without it, a 4xx/5xx response from requests doesn't raise anything — response.json() happily parses whatever error body the server sent back (which might not even be JSON), and your code proceeds as if the call succeeded, often crashing much later with a confusing, unrelated error (like a KeyError on a field that only exists in the success shape). raise_for_status() converts a bad status into an immediate, clear exception at the exact point the real problem occurred — right next to the request that failed, where you actually want to catch and handle it, not three functions downstream.
What does 429 mean, and what header should a well-behaved API include alongside it?▶
429 Too Many Requests means the client has been rate-limited — it's sent too many requests in a given window and the server is refusing this one to protect itself. A well-behaved API pairs a 429 with a Retry-After header telling the client how many seconds (or a timestamp) to wait before retrying, so a well-written client can back off automatically instead of hammering the server in a tight retry loop — which is exactly the kind of behavior that triggered the rate limit in the first place.
Why does a REST API design resources as nouns (/candidates/5) rather than verbs (/getCandidate?id=5)?▶
REST models the API around RESOURCES (nouns) and lets the HTTP METHOD carry the verb — GET /candidates/5 reads it, PUT /candidates/5 replaces it, DELETE /candidates/5 removes it — so the same URL consistently means "the candidate with id 5" regardless of what you're doing to it. A verb-in-the-URL design duplicates information HTTP already gives you for free via the method, and tends to sprawl into an ad-hoc, inconsistent set of endpoint names as the API grows. It's also what makes HTTP-level tooling (caching GETs, rate-limiting by method, generic API gateways) work cleanly — none of that reasons about a URL path's verb, only about the actual HTTP method.
