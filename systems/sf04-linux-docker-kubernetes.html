4.1 — Linux Error Literacy

Every process that exits leaves an exit code behind — 0 means success, anything else means failure, and a handful of non-zero codes have conventional meanings worth memorizing: 1 is a generic error, 126 means the command was found but isn't executable (a permissions problem), 127 means the command wasn't found at all (a typo or missing install), 137 means the process was killed by SIGKILL — signal 9 — which is what the kernel's OOM killer uses when a process exhausts available memory, and 139 means a segmentation fault (SIGSEGV). You can check the last command's exit code yourself with echo $? immediately after it runs.

🧠
The single most useful distinction in all of networking debugging: Connection refused means your request DID reach the target machine, but nothing was listening on that port — the machine is up, the service isn't. A Connection timed out means your request never got a response AT ALL — usually a firewall silently dropping the packet, or the machine being unreachable. "Refused" tells you the service crashed or never started; "timed out" tells you to look at the network path, not the service.

A few more real error messages worth recognizing instantly: "Permission denied" means the process's user doesn't have the file-mode/ownership rights it needs — fixed with chmod (change permission bits) or chown (change owner), never by just running everything as root, which trades a debugging problem for a security one. "No such file or directory" is almost always a path bug — very often code that assumed it would run from a specific working directory and didn't, so a relative path resolved somewhere unexpected. "Address already in use" means something else is already bound to that port on this machine — you either kill that process or pick a different port.

log_lines = [ "app[3021]: ERROR PermissionError: Permission denied: '/var/log/app/output.log'", "kernel: Out of memory: Killed process 4521 (worker) total-vm:2048000kB", "app[3021]: ERROR ConnectionRefusedError: Connection refused", "app[3021]: INFO Starting worker pool", "nginx: ERROR bind() to 0.0.0.0:8080 failed (98: Address already in use)", "app[3021]: ERROR FileNotFoundError: No such file or directory: 'config.yaml'", ] def classify_line(line): if 'Permission denied' in line or 'PermissionError' in line: return 'permission' if 'Out of memory' in line or 'Killed process' in line: return 'oom' if 'Connection refused' in line: return 'connection_refused' if 'Address already in use' in line: return 'port_in_use' if 'No such file or directory' in line: return 'missing_file' return None classified = {} for line in log_lines: cat = classify_line(line) if cat: classified.setdefault(cat, []).append(line) # (append the LINE, not just count -- keeps the evidence, not just a tally) for cat, lines in classified.items(): print(f'{cat}: {len(lines)}') print(f'Total classified: {sum(len(v) for v in classified.values())} of {len(log_lines)} lines')
⭐
This is a simplified stand-in for what real observability tooling does at scale. A production system doesn't have a human reading every log line — it has pattern-matching rules (much more sophisticated than this, but the same idea) that classify and count errors automatically, so an on-call engineer sees "OOM errors spiked 5x in the last 10 minutes" instead of scrolling through raw text.
4.2 — Docker Error Literacy

A Dockerfile builds an image as a stack of layers, one per instruction — each layer is cached, which is why reordering a Dockerfile (e.g. copying dependency files and installing them BEFORE copying your actual source code) can make rebuilds dramatically faster. But the errors that actually bite in practice happen at run time, not build time:

SymptomReal cause
"port is already allocated"Something else on the HOST already has that port bound — pick a different host port, or stop the other process
Container exits immediately after `docker run`Almost always: the app crashed on startup (check docker logs first), OR the main process wasn't kept in the foreground — a container stays alive only as long as its PID 1 process keeps running
"OOMKilled": true in `docker inspect`The container hit its configured memory limit and the kernel killed it — same underlying mechanism as Linux's OOM killer, just scoped to the container's cgroup
Volume mount permission errorsThe UID running inside the container doesn't match the UID that owns the files on the host-mounted volume
"image not found" / pull errorsTypo in the image name, wrong tag, or missing registry authentication

A real, runnable piece: diagnosing containers from their inspect state
def diagnose_container(inspect): state = inspect['State'] if state.get('OOMKilled'): return 'OOMKilled - hit its memory limit' if state['ExitCode'] != 0 and not state['Running']: return f"Exited with code {state['ExitCode']} - check docker logs" if state['Running'] and state.get('Health', {}).get('Status') != 'healthy': return 'Running but unhealthy - check healthcheck + docker logs' return 'Healthy' containers = { 'web-1': {'State': {'Running': False, 'ExitCode': 1, 'OOMKilled': False}}, 'worker-2': {'State': {'Running': False, 'ExitCode': 137, 'OOMKilled': True}}, 'cache-3': {'State': {'Running': True, 'ExitCode': 0, 'OOMKilled': False, 'Health': {'Status': 'healthy'}}}, } for name, info in containers.items(): print(f'{name}: {diagnose_container(info)}')
⚠️
Notice the check ORDER in diagnose_container(): OOMKilled is checked FIRST, before the generic non-zero-exit-code check. An OOM-killed container also has a non-zero exit code (typically 137), so if the generic check ran first it would report a misleadingly vague "exited with code 137" instead of the actually-useful "OOMKilled." Same discipline as SF 3's firewall rule ordering — specific, more-informative checks belong before general fallback ones.
4.3 — Kubernetes Pod Lifecycle & the Real Diagnostic Workflow

A pod moves through real, named phases: Pending (not yet scheduled onto a node — often a resource-request issue, or no node matches its constraints), ContainerCreating (node assigned, image being pulled / volumes being mounted), Running, CrashLoopBackOff (the container keeps starting and dying, and Kubernetes is backing off how fast it retries — a rapidly-crashing app, not a scheduling problem), ImagePullBackOff (Kubernetes cannot pull the image at all — almost always a typo in the image name/tag, or missing registry credentials), and Completed/Error for one-off Jobs.

🎯
Liveness vs. readiness probes — one of the single most common real interview questions in this space. A failing liveness probe tells Kubernetes "this container is stuck/dead" — Kubernetes RESTARTS the container. A failing readiness probe tells Kubernetes "this container is alive but not ready to receive traffic yet" — Kubernetes stops SENDING it traffic, but does NOT restart it. Get this backwards and you'll either restart a container that just needs more warm-up time (readiness's actual job), or keep sending live traffic to something that's actually broken (liveness's actual job).

The real diagnostic workflow engineers actually use, in order:

  • kubectl get pods — spot the phase. Is it Pending, CrashLoopBackOff, ImagePullBackOff, or something else?
  • kubectl describe pod <name> — read the Events section at the very bottom. This is where the real reason usually lives (a scheduling failure, a failed probe, an image pull error) — skipping straight to logs and missing this section is a very common early-career mistake.
  • kubectl logs <pod> (or kubectl logs <pod> --previous if it already crashed and restarted — the CURRENT logs of a freshly-restarted container won't contain the crash that already happened).
  • kubectl get events --sort-by=.lastTimestamp — cluster-wide recent events, useful when the problem might not even be about this one pod (e.g. a node running out of resources).
A real, runnable piece: mapping pod state to the correct next command
POD_ACTIONS = { 'Pending': 'kubectl describe pod <name>', 'ImagePullBackOff': 'kubectl describe pod <name>', 'CrashLoopBackOff': 'kubectl logs <name> --previous', } def next_command(pod): phase = pod['phase'] if phase == 'Running' and not pod.get('ready', True): return 'kubectl describe pod <name>' # readiness probe likely failing return POD_ACTIONS.get(phase, 'kubectl get events --sort-by=.lastTimestamp') pods = [ {'name': 'api-7f9', 'phase': 'CrashLoopBackOff', 'restarts': 5}, {'name': 'worker-2c1', 'phase': 'ImagePullBackOff', 'restarts': 0}, {'name': 'cache-9a0', 'phase': 'Running', 'ready': False}, {'name': 'gateway-3e2', 'phase': 'Pending', 'restarts': 0}, ] for pod in pods: print(f"{pod['name']} ({pod['phase']}): {next_command(pod)}")
Systems Fundamentals 4 Summary
  • Exit codes have conventional meanings (0=success, 137=killed/OOM, 127=command not found) — check with echo $?.
  • "Connection refused" = reachable, nothing listening. "Connection timed out" = never got a response, usually a network/firewall path issue. Different problems, different fixes.
  • A container exiting immediately? Check docker logs FIRST. OOMKilled in docker inspect means it hit its memory limit.
  • Liveness probe failure → restart. Readiness probe failure → removed from traffic, not restarted. Different mechanisms for different problems.
  • Real K8s workflow: kubectl get pods → describe (read Events!) → logs (or --previous) → cluster-wide get events.

💼 Linux/Docker/Kubernetes Interview Prep

12 questions on operational troubleshooting — the "have you actually run this in production" questions that separate candidates who've read documentation from candidates who've been paged at 2am. Click a question to reveal the answer. Not graded — this is here so you walk into an interview prepared.

What's the difference between a liveness probe and a readiness probe, precisely?▶
A failing liveness probe means "this container is stuck or dead" — Kubernetes restarts it. A failing readiness probe means "this container is alive but not ready for traffic yet" — Kubernetes removes it from service endpoints (stops routing traffic to it) WITHOUT restarting it. Confusing the two causes real incidents: using only a liveness probe for a slow-starting app can trigger a restart loop during normal startup; using only a readiness probe for a genuinely deadlocked process means it silently stays up forever, never restarted, never serving traffic.
What's the practical difference between CrashLoopBackOff and ImagePullBackOff?▶
ImagePullBackOff means Kubernetes can't even GET the image — the container has never actually started, so `kubectl logs` will show nothing useful; the fix is almost always in the image name/tag/registry-auth, found via `kubectl describe pod`. CrashLoopBackOff means the image WAS pulled and the container DID start, but it keeps crashing shortly after — the fix is in the application itself, found via `kubectl logs --previous`. Different failure point, different diagnostic command, different fix.
What does OOMKilled mean, and how would you actually fix it?▶
The container exceeded its configured memory limit and the kernel's cgroup OOM killer terminated it (exit code 137, SIGKILL). Real fixes: raise the memory limit if the usage is legitimate and just underprovisioned; find and fix an actual memory leak if usage climbs without bound over time (very different problems requiring very different fixes — profiling before you touch the config is the right first step, not just raising the limit and hoping).
"Connection refused" vs. "Connection timed out" — why does the distinction matter operationally?▶
"Refused" means your packet reached the target machine and got an active rejection — nothing is listening on that port. You look at the SERVICE (is it running? did it crash? wrong port?). "Timed out" means you got no response at all — the packet likely never arrived. You look at the NETWORK PATH (firewall rule, security group, routing, or the host being down entirely). Debugging the wrong side wastes real time — restarting a healthy service won't fix a firewall rule, and rewriting a firewall rule won't fix a crashed process.
Why does `kubectl describe pod`'s Events section matter more than people expect?▶
Because a lot of real failures happen BEFORE your application code ever runs — failed scheduling (no node has enough resources), failed image pulls, failed volume mounts, failed probes. None of that shows up in `kubectl logs`, because your container's own logging never started. The Events section is where Kubernetes itself reports what IT did and why — skipping straight to `logs` and finding nothing is a very common early mistake that wastes real debugging time.
Why would a container exit immediately even though the app "should" work?▶
A container stays alive only as long as its PID 1 process keeps running in the FOREGROUND. A very common mistake: a Dockerfile's CMD starts a process that immediately backgrounds itself (daemonizes), so Docker sees PID 1 exit right away and stops the container — even though the "real" process is technically still trying to run. The fix is running the process in the foreground (many tools have a `--no-daemon`-style flag for exactly this reason).
What does exit code 137 specifically tell you, versus a generic exit code 1?▶
137 = 128 + 9, where 9 is the signal number for SIGKILL — the process was forcibly killed from the outside (very often the OOM killer), not something that failed on its own via a normal error path. Exit code 1 is just "the program itself decided to exit with an error" — the process controlled its own shutdown. Seeing 137 should immediately point you toward memory limits and `dmesg`/`docker inspect`'s OOMKilled flag, not toward reading application error-handling code.
A pod is stuck in Pending for 10 minutes. What's your first hypothesis?▶
Pending almost always means it hasn't been SCHEDULED onto any node yet — most commonly because no node currently has enough free CPU/memory to satisfy the pod's resource requests, or a node-affinity/taint rule excludes every available node. `kubectl describe pod` Events will say exactly this (e.g. "0/5 nodes are available: insufficient cpu"). It's rarely an application bug at this stage, since the application container hasn't even started.
How would you approach a completely unfamiliar production incident, in general?▶
Same shape every time: (1) what changed recently — a deploy, a config change, a traffic spike; (2) what's the blast radius — one service or many, one region or all; (3) check the cheapest signals first (dashboards, `kubectl get pods`, error rates) before deep-diving into one theory; (4) form a specific, falsifiable hypothesis and check it directly rather than guessing broadly; (5) mitigate first if you can (rollback, restart, scale up), root-cause fully afterward — restoring service and finding the root cause are different goals with different urgency.
What's the actual difference between `chmod` and `chown`, and when do you need each?▶
chmod changes WHAT a file's owner/group/others are allowed to do to it (read/write/execute bits). chown changes WHO owns the file in the first place. A "Permission denied" error can come from either direction: the right user doesn't have the right BITS set (fix with `chmod`), or the process is running as the WRONG user entirely for a file it should own (fix with `chown`) — this second case is exactly what causes Docker volume-mount permission errors when the container's UID doesn't match the host file owner's UID.
Why check `ss -tulpn` (or `netstat -tulpn`) specifically, instead of just trying to connect and seeing what happens?▶
It answers the exact question directly — "is ANYTHING actually listening on this port, on this machine, right now" — rather than inferring it indirectly from a failed connection attempt, which conflates "nothing is listening here" with "a firewall somewhere between me and there is blocking me." Checking directly on the machine itself rules out the network-path variable entirely, narrowing the search space fast.
Why does `journalctl -u -f` matter for a systemd-managed service, specifically?▶
Many real services run as systemd units rather than inside a container, and systemd captures their stdout/stderr into the structured system journal rather than a plain log file — `journalctl -u ` is the correct, complete way to see that output (including startup failures that happened before any application-level log file was even opened), and `-f` follows it live the same way `tail -f` would for a plain file.
