Systems Fundamentals 2:
Networking & Microservices Communication

"My service says it called the other service, but did it actually get there?" is one of the most common real production questions — and answering it well requires the minimum useful networking model (not a full OSI-layer deep-dive), the sync-vs-async tradeoff, and a genuinely runnable toolkit: health checks, correlation IDs, retries with backoff, and a circuit breaker. This chapter builds all four for real, plus a runnable simulation that traces a request across a chain of services and reports exactly where it broke.

🕐 ~60 min read 💻 5 exercises 🏗️ Capstone Project 🧪 5-question quiz 💼 12 Interview Prep Q&A
2.1 — The Minimum Useful Networking Model

You don't need all seven OSI layers memorized to reason about microservices — you need four ideas. An IP address identifies a machine on a network. A port (0-65535) identifies WHICH process on that machine to talk to — a single server can run a web app on port 443 and a database on port 5432 at the same time, distinguished purely by port. DNS resolves a human-readable domain name (api.example.com) to the IP address a machine can actually route packets to — that's the entire idea, nothing more mystical than a giant lookup table. And two protocol names you'll hear constantly: TCP operates around what's informally called "Layer 4" (it deals in raw connections between IP:port pairs, with no idea what HTTP or a "request" even is), while HTTP operates around "Layer 7" (it's built ON TOP of a TCP connection, and understands methods, paths, headers). That Layer 4 / Layer 7 vocabulary matters directly in SF 3, where it's the whole difference between two kinds of load balancer.

2.2 — TCP vs UDP: Guaranteed Delivery vs Speed

TCP guarantees delivery and ordering — every packet is acknowledged, lost packets get retransmitted, and the receiving application always sees bytes in the order they were sent. UDP makes none of those guarantees — packets can arrive out of order or simply vanish, and UDP won't retransmit them. That single difference explains a very concrete real-world split: a file download needs TCP (a file missing even one byte is corrupted — you need the guarantee). A live video call can use UDP (a packet from half a second ago that arrives late is USELESS by the time TCP would finish retransmitting it — you'd rather skip the frame and keep the call moving in real time than freeze everything waiting for a guaranteed-perfect but stale packet). TCP optimizes for correctness; UDP optimizes for timeliness.

2.3 — Synchronous vs Asynchronous Service Communication

Synchronous communication — Service A calls Service B over HTTP and waits for the response before continuing — is simple to write and reason about, but it COUPLES the two services together in real time: if B is slow or down, A is slow or down too, and a chain of sync calls compounds latency (A waits for B, who waits for C...). Asynchronous communication — A publishes a message to a queue and moves on immediately, B consumes it whenever it's ready — decouples the two services' uptime: B being down for five minutes doesn't block A at all, the message just waits in the queue. The real cost of async is eventual consistency: there's a window of time where A's state has changed but B hasn't caught up yet, plus real complexity around message ordering and duplicate delivery. Neither is "better" — sync is right when the caller genuinely needs an immediate answer to keep going (e.g. "is this login valid"); async is right when the caller can move on before the work is fully done (e.g. "send this welcome email eventually").

2.4 — Verifying Service A Actually Reaches Service B

Three real tools answer "is my service actually talking to the other one?" First: a /healthz endpoint that a service exposes, reporting whether it's actually able to serve traffic right now — infrastructure polls this constantly. Second: correlation IDs — a unique identifier generated once (usually at the entry point) and passed unchanged through every downstream call and every log line that request touches, so you can grep logs for ONE id and see the entire path a single request took, in order, across every service. Third: for a whole chain of services, it helps to think of the system as a call graph — which service calls which — and simulate tracing a request through it to see exactly which hop (if any) fails.

A health-check pass over multiple services
def check_health(services): report = {} for name, info in services.items(): report[name] = 'healthy' if info['up'] and info['latency_ms'] < 500 else 'unhealthy' return report services = { 'auth-service': {'up': True, 'latency_ms': 45}, 'candidate-service': {'up': True, 'latency_ms': 620}, # up, but dangerously slow 'scoring-service': {'up': False, 'latency_ms': 0}, } print(check_health(services))
⭐
Notice "healthy" isn't just "up". A service that's technically running but taking 620ms to respond is a real production problem a boolean up/down check would completely miss — a good health check considers latency (and often error rate) alongside raw reachability.
Correlation IDs: tracing ONE request across services via logs
def log_event(correlation_id, service, event, **extra): return f'[{correlation_id}] {service}: {event} {extra}' cid = 'req-8842' # generated ONCE, at the entry point (e.g. the API gateway) logs = [ log_event(cid, 'api-gateway', 'received request', path='/apply'), log_event(cid, 'auth-service', 'token validated', user_id=501), log_event(cid, 'candidate-service', 'profile loaded', candidate_id=501), ] for line in logs: print(line) # "trace this one request" = grep every log line for its correlation ID matching = [l for l in logs if cid in l] print(f'Log lines for {cid}: {len(matching)}')

Zoom out from single log lines to the whole system, and you can model it as a call graph: a dict mapping each service to the calls it makes downstream, each with a simulated latency and a flag for whether that specific hop fails. Tracing a request through it — stopping and reporting the instant something fails — is exactly the debugging instinct "which service actually broke," made runnable.

Simulating and tracing a microservice call graph
CALL_GRAPH = { 'api-gateway': [{'to': 'auth-service', 'latency_ms': 40, 'fails': False}], 'auth-service': [{'to': 'candidate-service', 'latency_ms': 60, 'fails': False}], 'candidate-service': [{'to': 'scoring-service', 'latency_ms': 150, 'fails': True}], # <- this hop fails 'scoring-service': [{'to': 'notification-service', 'latency_ms': 30, 'fails': False}], 'notification-service': [], } def trace_request(graph, start): total_latency = 0 path = [start] node = start while graph.get(node): call = graph[node][0] total_latency += call['latency_ms'] if call['fails']: return {'ok': False, 'failed_at': call['to'], 'path': path + [call['to']], 'total_latency_ms': total_latency} path.append(call['to']) node = call['to'] return {'ok': True, 'failed_at': None, 'path': path, 'total_latency_ms': total_latency} result = trace_request(CALL_GRAPH, 'api-gateway') print(f'Trace result: {result}')
🧠
This is the "which hop failed" question, made concrete. Instead of guessing, the trace walks the graph hop by hop, adding up latency as it goes, and stops exactly at the failing call — telling you both WHERE it broke and how much time was spent before the failure. Real distributed tracing tools (Jaeger, Zipkin, OpenTelemetry) do exactly this, using correlation IDs (called "trace IDs" there) to stitch spans from every service back into one picture.
2.5 — Timeouts, Retries With Backoff & Circuit Breakers

A timeout is a promise you make to yourself: "I will not wait for this call forever." Without one, a single slow downstream call can hang your service indefinitely. A retry re-attempts a failed call — but retrying immediately, in a tight loop, can hammer an already-struggling service and make things worse (a "retry storm"). Exponential backoff fixes that by waiting progressively longer between attempts (0.1s, 0.2s, 0.4s...), giving the downstream service breathing room to recover.

Retry with exponential backoff — genuinely runnable
def call_with_retry(fn, max_retries=3, base_delay=0.1): delays = [] for attempt in range(max_retries + 1): result = fn(attempt) if result['ok']: return result, delays if attempt < max_retries: delay = base_delay * (2 ** attempt) # 0.1, 0.2, 0.4, 0.8, ... delays.append(delay) return result, delays # a flaky call that fails on attempts 0 and 1, succeeds on attempt 2 def flaky_call(attempt): return {'ok': attempt >= 2, 'attempt': attempt} result, delays = call_with_retry(flaky_call) print(f'Result: {result}') print(f'Backoff delays used: {delays}')

But retries alone assume the failure is brief. If the downstream is genuinely down for a while, retries just mean EVERY caller keeps hammering it — a circuit breaker fixes that with a small state machine: CLOSED (normal — calls flow through, failures are counted), OPEN (after too many failures — calls are short-circuited immediately, no network call even attempted, protecting both the caller and the struggling service), and HALF_OPEN (after a cooldown — let a trial call through; succeed and it closes again, fail and it reopens).

A circuit breaker state machine — CLOSED / OPEN / HALF_OPEN
class CircuitBreaker: def __init__(self, failure_threshold=3, reset_after=5): self.state = 'CLOSED' self.failure_count = 0 self.failure_threshold = failure_threshold self.reset_after = reset_after # calls-while-open before trying a HALF_OPEN test self.calls_since_open = 0 def call(self, fn, *args): if self.state == 'OPEN': self.calls_since_open += 1 if self.calls_since_open >= self.reset_after: self.state = 'HALF_OPEN' # allow ONE trial call through else: return {'ok': False, 'error': 'circuit open - call short-circuited, no network call made'} try: result = fn(*args) if not result['ok']: raise ValueError('downstream failure') if self.state == 'HALF_OPEN': self.state = 'CLOSED' # trial call succeeded -> fully recovered self.failure_count = 0 return result except Exception: self.failure_count += 1 if self.state == 'HALF_OPEN': self.state = 'OPEN' # trial call failed -> still broken, reopen self.calls_since_open = 0 elif self.failure_count >= self.failure_threshold: self.state = 'OPEN' self.calls_since_open = 0 return {'ok': False, 'error': 'downstream failure'} def always_fail(): return {'ok': False} def always_succeed(): return {'ok': True} cb = CircuitBreaker(failure_threshold=3, reset_after=2) for i in range(3): cb.call(always_fail) print(f'State after 3 failures: {cb.state}') r1 = cb.call(always_succeed) # circuit is OPEN, still short-circuited (only 1 call since open) print(f'r1: {r1}, state: {cb.state}') r2 = cb.call(always_succeed) # 2nd call since open -> HALF_OPEN trial -> succeeds -> CLOSED print(f'r2: {r2}, state: {cb.state}')
⚠️
Retries and circuit breakers solve different halves of the same problem. Retries absorb brief transient blips. A circuit breaker protects against a sustained outage — without it, every caller's retries keep pounding a service that's already struggling, which can prevent it from ever recovering. And crucially: NEVER retry a non-idempotent operation without an idempotency key — SF 1 covered exactly why (a naive retry on a bare POST can double-charge a card or double-send an email).
Systems Fundamentals 2 Summary
  • IP + port + DNS is the whole minimum networking model — an address, which process on that address, and a name-to-address lookup.
  • TCP guarantees delivery/order; UDP doesn't — which is exactly why file downloads need TCP and live video calls can use UDP.
  • Sync communication couples uptime/latency across services; async decouples it at the cost of eventual consistency and added complexity.
  • Health checks, correlation IDs, and call-graph tracing are the real toolkit for answering "is Service A actually reaching Service B?"
  • Retry with exponential backoff absorbs brief blips; a circuit breaker (CLOSED/OPEN/HALF_OPEN) protects against sustained outages — and neither should ever apply blindly to a non-idempotent operation.

💼 Networking & Microservices — Interview Prep

12 questions covering the exact angles interviewers use to check whether you've genuinely reasoned about distributed systems reliability, not just called an HTTP client. Click a question to reveal the answer.

What's the real tradeoff between synchronous and asynchronous service communication?▶
Sync (direct HTTP call, wait for the response) is simpler to write and reason about, but it couples the caller's uptime and latency to the callee's — if B is slow or down, A is too, and a chain of sync calls compounds latency (A waits for B, who waits for C...). Async (a message queue, an event) decouples services — A can publish and move on even if B is temporarily down, since messages queue up until B recovers — but adds real complexity: eventual consistency (a window where A "finished" but B hasn't processed yet), message ordering, and what happens if a message gets processed twice.
Why do health-check endpoints matter, beyond "nice to have"?▶
A /healthz endpoint gives infrastructure (a load balancer, an orchestrator like Kubernetes) a fast, cheap way to answer "can this instance serve traffic right now" without guessing from application-level signals. Without one, a crashed or degraded instance can keep receiving traffic indefinitely, silently failing every request it gets. With one, the failing instance gets pulled from rotation (or restarted) automatically, often before anyone even notices.
What is a correlation ID for, and why can't you just use timestamps to trace a request across services?▶
A correlation ID is a unique identifier generated once (usually at the entry point, like the API gateway) and passed along unchanged through every downstream call and every log line that request touches. It lets you grep logs for one ID and see the entire path a single request took across every service, in order. Timestamps can't do this reliably: multiple requests happen concurrently, clocks can drift slightly between machines, and nothing about a timestamp tells you WHICH request a given log line belongs to — a correlation ID is an explicit, unambiguous key for exactly that.
When do retries make things WORSE instead of better?▶
When the operation being retried is NOT idempotent. Retrying a failed "charge this card" POST without an idempotency key can charge the customer twice; retrying a failed "send notification email" can spam the user. The fix isn't "never retry" — it's "only retry operations that are safe to repeat" (idempotent by nature, or made safe via an idempotency key), combined with a circuit breaker so a genuinely down service doesn't get hammered by every caller retrying simultaneously — a "retry storm" that can turn a partial outage into a full one.
Walk through the three circuit breaker states and what each one actually does.▶
CLOSED is normal — requests flow through directly, failures are counted. After enough failures cross a threshold, the breaker trips to OPEN — calls are short-circuited immediately without even attempting the network call, protecting both the caller (fast failure instead of hanging) and the struggling downstream service (no added load while it's already in trouble). After a cooldown period, it moves to HALF_OPEN — it lets a small number of trial requests through to test recovery; success closes it again, failure reopens it and the cooldown starts over.
Why can video calls get away with UDP while a file download needs TCP?▶
TCP guarantees delivery and ordering — every packet is acknowledged, lost packets get retransmitted — exactly what a file download needs (a file missing even one byte is corrupted). UDP makes no such guarantee — packets can arrive out of order or simply not arrive, and UDP won't retransmit. For a live call, a packet from half a second ago that arrives late is USELESS by the time a guaranteed retransmit would finish — you'd rather skip it and keep the stream moving in real time than freeze waiting for a perfect-but-stale packet. TCP optimizes for correctness; UDP optimizes for timeliness — which one you want depends entirely on whether a late-but-correct packet is still useful.
In one sentence, what does DNS actually do, and why does that matter for debugging?▶
DNS resolves a human-readable domain name to the IP address a machine can actually route packets to — that's the whole idea. It matters for debugging because "can't reach the server" can come from at least two different places: DNS resolution failing (the name never turns into an IP at all) vs. the resolved IP being unreachable (the name resolves fine, but nothing responds) — mixing those up sends you down the wrong troubleshooting path.
Why does "connection refused" point at something different than a timeout?▶
"Connection refused" means a packet reached the target machine and the target machine ACTIVELY said no — usually because nothing is listening on that port (the process isn't running, or is bound to a different port). A timeout means your packet never got any response at all, which usually means something in between is silently dropping it (a firewall rule, a security group, a network partition) or the target machine itself is unreachable. That's a genuinely useful diagnostic split covered in more depth in SF 4: "refused" tells you the network path IS open and something IS there, just not listening; a timeout tells you the network path itself is the suspect.
Why would you choose a message queue over a direct HTTP call between two services?▶
A direct HTTP call couples the two services' availability together in real time — if the downstream is down or slow, the caller is blocked or failing too. A message queue (Kafka, RabbitMQ, SQS) decouples that: the caller publishes and moves on immediately, and the queue holds the message until the consumer is ready — even if the consumer is down for five minutes, nothing is lost, it just catches up later. The tradeoff is added complexity (message ordering, at-least-once vs exactly-once delivery, eventual consistency instead of an immediate answer), and it's a bad fit when the caller genuinely needs an immediate answer to keep going (e.g. "is this login valid" can't be answered asynchronously).
What is "eventual consistency," concretely, and why is it the price of async communication?▶
When Service A publishes an event and Service B consumes it asynchronously, there's a real window of time — milliseconds to seconds, sometimes longer under load — during which A's state has changed but B hasn't caught up. Anything reading from B during that window sees stale data relative to A. That's eventual consistency: the system WILL become consistent, just not INSTANTLY. It's the price of decoupling — a synchronous call gives immediate consistency but couples uptime; async gives resilience to the downstream being temporarily unavailable, at the cost of a window where different parts of the system briefly disagree.
How would you actually verify Service A is reaching Service B, beyond "it looks fine in the dashboard"?▶
Layer the evidence: (1) Check B's own health-check endpoint directly — if it's unhealthy, nothing reaching it matters yet. (2) Check A's outbound logs for the exact call it made to B and B's inbound logs for a matching entry with the SAME correlation ID — if A logged sending it but B never logged receiving it, the break is somewhere in between (DNS, firewall, wrong URL/port). (3) If both logged it but something's still wrong, compare timestamps for unusually high latency and check the actual response body/status B sent back, not just "did a response come back." Health, then logs matched by correlation ID, then latency/response detail — that layered approach is a much stronger answer than "I'd check the logs."
You added retries AND a circuit breaker to a downstream call. Why do you need both, rather than just one?▶
Retries alone handle brief, transient blips (a dropped packet, a momentary GC pause) — but if the downstream is GENUINELY down for a while, retries alone just mean every caller hammers it repeatedly, which can prevent it from ever recovering (the "retry storm" problem). A circuit breaker alone handles the sustained-outage case (stop calling entirely once it's clearly failing) but without retries, a single transient blip that would have succeeded on the very next attempt gets treated as a hard failure. Together: retries absorb short-lived transient failures, and the circuit breaker only kicks in once failures are frequent/sustained enough to indicate a real outage — protecting the downstream from pile-on while staying resilient to one-off blips.
