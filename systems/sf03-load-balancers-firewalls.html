Systems Fundamentals 3:
Load Balancers & Firewalls

Two questions that come up in almost every backend/DevOps interview: "explain Layer 4 vs Layer 7 load balancing" and "how does a firewall actually decide what to block?" Both get real, runnable answers here — you'll build and run an L4-style router next to an L7-style router on the SAME requests to see the difference concretely, implement round robin and least-connections side by side, and write a first-match-wins firewall rule evaluator using nothing but the Python standard library.

🕐 ~60 min read 💻 5 exercises 🏗️ Capstone Project 🧪 5-question quiz 💼 12 Interview Prep Q&A
3.1 — What a Load Balancer Does, and Why You Need One

A load balancer sits in front of multiple backend instances and gives clients ONE stable entry point, then distributes incoming traffic across the backends behind it. Three concrete reasons this matters: it enables horizontal scaling (add more backend instances under load, remove them when load drops, without clients ever knowing how many there are), it enables zero-downtime deploys (bring up new-version instances, wait for them to pass health checks, shift traffic over, THEN retire the old ones — at no point is there a moment with zero healthy backends), and it gives you a single place to handle cross-cutting concerns like TLS termination.

3.2 — Layer 4 vs Layer 7 Load Balancing, Made Concrete

This is the vocabulary from SF 2.1, applied directly: an L4 (Layer 4) load balancer operates at the TCP level — it only ever sees IP addresses and ports, a raw connection, with zero visibility into what's actually inside the HTTP request. An L7 (Layer 7) load balancer operates at the HTTP level — it can actually read the request (path, headers, cookies) and route based on what it finds, e.g. /api/* to one backend pool and /static/* to a completely different one, or route based on a header for A/B testing. The tradeoff: L7 has to actually parse the HTTP request to make that decision (more CPU cost), L4 is faster and simpler but structurally can't make content-aware decisions.

L4 (blind to path) vs L7 (routes by path) — same requests, different routers
class L4RoundRobin: # L4 only ever sees a "connection" — no path, no headers, nothing above TCP def __init__(self, backends): self.backends = backends self.i = 0 def route(self, connection): backend = self.backends[self.i % len(self.backends)] self.i += 1 return backend l4 = L4RoundRobin(['backend-1', 'backend-2', 'backend-3']) connections = [{'src_ip': '10.0.0.1'}, {'src_ip': '10.0.0.2'}, {'src_ip': '10.0.0.3'}, {'src_ip': '10.0.0.4'}] for c in connections: print(f'{c["src_ip"]} -> {l4.route(c)}') class L7Router: # L7 can see the actual HTTP request — route on its path def __init__(self, path_rules): # [(prefix, [backends]), ...] — first match wins self.path_rules = path_rules self.counters = {prefix: 0 for prefix, _ in path_rules} def route(self, request): for prefix, backends in self.path_rules: if request['path'].startswith(prefix): i = self.counters[prefix] backend = backends[i % len(backends)] self.counters[prefix] += 1 return backend return None l7 = L7Router([ ('/api/', ['api-backend-1', 'api-backend-2']), ('/static/', ['cdn-backend-1']), ]) requests_l7 = [{'path': '/api/candidates'}, {'path': '/static/logo.png'}, {'path': '/api/scores'}, {'path': '/static/app.js'}] for r in requests_l7: print(f'{r["path"]} -> {l7.route(r)}')
⭐
Look at what each router's input actually contains. L4's route() takes a bare "connection" — it structurally has no path field to inspect, so it can only ever round-robin blindly across ALL backends. L7's route() takes a full request dict with a path, and uses it to send /api/ traffic and /static/ traffic to entirely different, purpose-built backend pools. That's not a design choice within L4 — it's a hard ceiling on what information is even available at that layer.
3.3 — Load Balancing Algorithms: Round Robin vs Least Connections

Round robin cycles through backends in a fixed order, assuming every request costs about the same and every backend is equally free — simple, but blind to real, uneven load. Least connections tracks how many requests are CURRENTLY in flight per backend and always sends the next one to whichever backend is least busy right now — it adapts to real load instead of blindly cycling. (A third option, weighted round robin, assigns each backend a weight so a bigger/more capable server gets proportionally more traffic — useful when backends aren't identical, or during a gradual canary rollout.)

Round robin vs least connections — same request count, different choices under uneven load
def round_robin(backends, n_requests): choices = [] i = 0 for _ in range(n_requests): choices.append(backends[i % len(backends)]) i += 1 return choices def least_connections(backends_load, n_requests): choices = [] for _ in range(n_requests): backend = min(backends_load, key=lambda b: backends_load[b]) backends_load[backend] += 1 # it's now handling one more request choices.append(backend) return choices backends = ['A', 'B', 'C'] rr_choices = round_robin(backends, 6) print(f'Round robin: {rr_choices}') load = {'A': 4, 'B': 0, 'C': 2} # A is already busy, B is idle lc_choices = least_connections(load, 6) print(f'Least connections: {lc_choices}') print(f'Final load: {load}')
🧠
Round robin sends A a request even though it started at 4 already-in-flight requests — it has no idea A is busy, it just knows "it's A's turn." Least connections never sends A anything in this run: B and C stay lower than A's starting point the entire time, so the algorithm correctly keeps steering new work away from the already-loaded backend. That's the concrete difference, not just an assertion about it.
3.4 — Health Checks Feeding the Routing Decision

A load balancer's routing algorithm only chooses among backends it believes are healthy. A backend that fails its health check gets pulled OUT of the rotation automatically — no human intervention needed — so new requests stop being sent to it until it recovers (or gets replaced).

A backend pool that stops routing to unhealthy backends
class HealthAwarePool: def __init__(self, backends): self.backends = backends # dict: backend -> {'healthy': bool} self.i = 0 def healthy_backends(self): return [b for b, info in self.backends.items() if info['healthy']] def route(self): pool = self.healthy_backends() if not pool: return None backend = pool[self.i % len(pool)] self.i += 1 return backend def mark_unhealthy(self, backend): self.backends[backend]['healthy'] = False pool = HealthAwarePool({'A': {'healthy': True}, 'B': {'healthy': True}, 'C': {'healthy': True}}) routed = [pool.route() for _ in range(3)] print(f'Before failure: {routed}') pool.mark_unhealthy('B') # B fails its health check -> removed from rotation routed2 = [pool.route() for _ in range(3)] print(f'After B fails health check: {routed2}')
3.5 — Firewalls: Allow/Deny Rules, Stateful vs Stateless, and WAFs

A firewall allows or denies traffic based on rules — typically matching source IP/CIDR range, destination port, and protocol. A stateless firewall evaluates every packet independently with no memory of prior traffic, so allowing an outbound connection requires a SEPARATE explicit rule to allow its return traffic back in. A stateful firewall tracks active connections — once it sees an outbound request get allowed, it automatically permits the matching response without a separate rule, because it remembers that connection is legitimate. (In AWS vocabulary, which this course's bonus/interview-mastery.html cloud cheat-sheet also touches on: Security Groups are the stateful, per-resource version of this; Network ACLs are the stateless, per-subnet version — used together as layered defense, not as a choice between one or the other.)

A WAF (Web Application Firewall) adds something neither kind of plain firewall can do: it operates at Layer 7 and inspects the actual HTTP CONTENT — path, headers, query string, body — looking for attack patterns like SQL injection strings or XSS payloads. A request from an allowed IP on an allowed port can still get blocked by a WAF if its BODY contains something like ' OR 1=1-- — a plain IP/port firewall structurally cannot see that far, because it never looks past the network/transport layer.

First-match-wins firewall rule evaluator — real semantics, real stdlib
import ipaddress FIREWALL_RULES = [ {'src': '203.0.113.0/24', 'port': None, 'action': 'DENY'}, # known-bad range, ANY port — checked FIRST {'src': '10.0.0.0/8', 'port': 22, 'action': 'ALLOW'}, # internal SSH {'src': '0.0.0.0/0', 'port': 443, 'action': 'ALLOW'}, # public HTTPS {'src': '0.0.0.0/0', 'port': 80, 'action': 'ALLOW'}, # public HTTP {'src': '0.0.0.0/0', 'port': None, 'action': 'DENY'}, # default deny — catches everything else ] def evaluate(rules, src_ip, dest_port): ip = ipaddress.ip_address(src_ip) for rule in rules: # evaluated top to bottom — FIRST match wins network = ipaddress.ip_network(rule['src']) port_matches = rule['port'] is None or rule['port'] == dest_port if ip in network and port_matches: return rule['action'] return 'DENY' # implicit default deny if somehow nothing matched at all tests = [ ('10.0.0.5', 22), # internal SSH -> ALLOW ('8.8.8.8', 443), # public HTTPS -> ALLOW ('8.8.8.8', 22), # public SSH attempt -> falls through to default deny ('203.0.113.5', 443), # known-bad range -> DENY even on an otherwise-open port ] for src, port in tests: print(f'{src}:{port} -> {evaluate(FIREWALL_RULES, src, port)}')
⚠️
Rule ORDER is not cosmetic — it changes the outcome. The known-bad-range DENY rule is placed FIRST, before the broad "allow port 443 from anywhere" rule. If that order were reversed, 203.0.113.5:443 would match the broad ALLOW rule first and never even reach the DENY rule — the specific block would silently never fire. Real firewalls (iptables, security groups) evaluate top-to-bottom and stop at the first match, exactly like evaluate() does — which is why "put specific exceptions before general rules" is a real operational discipline, not a style preference.
Systems Fundamentals 3 Summary
  • A load balancer gives clients one stable entry point and enables horizontal scaling plus zero-downtime deploys.
  • L4 routes on IP/port only (blind to HTTP content); L7 can read the actual request and route by path/header/cookie — at the cost of more CPU to parse it.
  • Round robin ignores current load; least connections adapts to it by tracking in-flight requests per backend; weighted round robin handles unequal backend capacity.
  • Health checks feed the routing decision directly — an unhealthy backend gets pulled from rotation automatically.
  • Firewalls match rules top-to-bottom, first match wins — stateful firewalls (Security Groups) remember connections, stateless ones (NACLs) don't. A WAF adds Layer 7 content inspection a plain firewall structurally cannot do.

💼 Load Balancers & Firewalls — Interview Prep

12 questions covering the exact L4/L7 and firewall angles interviewers use to check whether you've genuinely reasoned about traffic routing and network security, not just memorized the terms. Click a question to reveal the answer.

What can an L7 load balancer see and decide on that an L4 load balancer fundamentally cannot?▶
L4 operates below the HTTP layer — it only sees IP addresses and ports (a raw TCP connection), so it can only route on things like source IP or destination port; it has no idea what path or headers are inside the request. L7 sits at the HTTP layer and can actually read the request — path, headers, cookies, even body in some cases — so it can route /api/* to one pool and /static/* to another, route by header for A/B testing, or use a cookie for session affinity. The tradeoff: L7 has to actually terminate/parse the HTTP request to decide, costing more CPU than L4's "just look at the IP/port" approach — L4 is faster and simpler, L7 is smarter and more flexible.
Why would you choose least-connections over round robin?▶
Round robin assumes every request costs roughly the same and every backend is equally free — it cycles through backends in order regardless of current load. That breaks the moment requests have wildly different costs (a fast health check vs. a slow report-generation call) or backends already carry different loads. Least-connections actively tracks how many requests are CURRENTLY in flight per backend and sends the next one to whichever is least busy right now — adapting to real, uneven load instead of blindly cycling, at the cost of needing to track live connection counts.
What's the actual difference between a stateful and a stateless firewall?▶
A stateless firewall evaluates every packet independently against its rule list — it has no memory of what came before, so allowing a TCP connection's response traffic back in needs an explicit rule for the return path too. A stateful firewall tracks the state of active connections — once it sees an outbound request get initiated and allowed, it automatically permits the matching return traffic without a separate rule, because it "remembers" that connection is legitimately open. Most real firewalls in practice (AWS Security Groups included) are stateful; the stateless model still matters conceptually because it's closer to how a Network ACL behaves — rules apply in both directions independently.
What does a WAF add on top of a regular firewall?▶
A regular firewall (or security group) decides allow/deny based on IP, port, and protocol — it has zero visibility into what's actually inside an allowed HTTP request. A WAF (Web Application Firewall) operates at Layer 7 and inspects the actual HTTP content — path, headers, query string, body — looking for patterns that indicate an attack: SQL injection strings, XSS payloads, known-bad user agents, abnormal request shapes. A request from an allowed IP on an allowed port can still be blocked by a WAF if the BODY contains something like ' OR 1=1-- — something a regular firewall structurally cannot see, because it never looks past the IP/port/protocol layer.
Why do health checks matter specifically for load balancing, beyond general monitoring?▶
Health checks feed the load balancer's actual ROUTING decision in real time — a backend that fails its health check is automatically pulled out of rotation, so new requests stop going to it without a human intervening. Without this feedback loop, a load balancer would keep sending a fraction of traffic to a broken backend indefinitely (if it's doing round robin blindly), guaranteeing a steady stream of failed user requests instead of just routing around the problem automatically until the backend recovers or gets replaced.
How does a load balancer plus health checks enable zero-downtime deploys?▶
Bring up new-version instances alongside the old ones, wait for the new instances to pass their health checks (confirming they're actually ready to serve traffic, not just "the process started"), start routing traffic to them, then drain and terminate the old instances only once the new ones are confirmed healthy and successfully taking traffic. At no point is there a moment with zero healthy backends — the load balancer's health-aware routing IS the mechanism that makes this safe; without it, you'd have to blindly guess when the new version is "ready" and hope for the best.
What is "sticky session" (session affinity), and why does it require L7?▶
Sticky session means routing all of a given user's requests to the SAME backend instance for their session — usually because that backend holds in-memory state (a shopping cart, a login session) other backends don't have. Implementing this requires reading something that identifies the user — typically a cookie — which only an L7 load balancer can see, since L4 has no visibility into HTTP-level data. It's often treated as a workaround rather than a best practice: the more scalable fix is usually moving that session state out of backend memory into a shared store (Redis, a database) so ANY backend can serve ANY request, removing the need for stickiness entirely.
Why does TLS termination usually happen at the load balancer rather than at each backend?▶
Terminating TLS (decrypting incoming HTTPS) at the load balancer means the certificate and private key only need to be managed and rotated in ONE place instead of on every backend instance — a real operational simplification at scale. It also lets backend-to-load-balancer traffic (inside a trusted private network) skip the CPU cost of encryption/decryption on every hop, and the load balancer needs L7 visibility into the decrypted HTTP request anyway (to route by path/header), so it has to decrypt regardless.
When would you actually reach for weighted round robin instead of plain round robin?▶
When your backends aren't identical — one server has twice the CPU/memory of another, or you're doing a gradual canary rollout and want only a small percentage of traffic hitting a new version while it's being validated. Weighted round robin lets you assign each backend a weight so the bigger/preferred one gets proportionally more requests instead of the exact same share as a weaker or newer, unproven machine — plain round robin has no way to express "this backend can handle more" at all.
What's the difference between a Security Group and a Network ACL, at the concept level (not AWS-specific trivia)?▶
A security-group-style firewall is typically STATEFUL and applied at the instance/resource level — you write ALLOW rules (nothing else is usually needed since anything not allowed is implicitly denied), and it automatically permits return traffic for connections it approved. A network-ACL-style firewall is typically STATELESS and applied at the subnet/network level — it evaluates rules in order, supports both explicit ALLOW and DENY, and requires thinking about both directions of traffic separately since it has no connection memory. In practice they're layered together: the ACL is a coarser, network-wide filter, and the security group is a finer, per-resource filter — defense in depth, not a choice between one or the other.
Firewall rules are usually evaluated top to bottom, first match wins. Why does rule ORDER matter so much?▶
The first rule that matches a given packet determines the outcome — everything after that point is never even evaluated. A broad ALLOW rule placed before a more specific DENY rule can silently swallow traffic that was supposed to be blocked, since the ALLOW matches first and the DENY is never reached. The practical discipline: put your most specific rules (especially DENY rules for known-bad sources) EARLIER, and your broad catch-all rules (including the default-deny fallback) LAST — the same "specific exceptions before general rules" logic that shows up in CSS specificity or exception-handling order.
A backend behind your load balancer starts timing out under load. Walk through how the load balancer, health checks, and backend pool interact to handle this automatically — versus what happens if none of this is set up.▶
With it set up: the backend starts failing its health checks → the load balancer marks it unhealthy and stops routing new traffic to it → remaining traffic redistributes across the still-healthy backends → if part of an auto-scaling group, a new healthy instance may spin up to replace lost capacity → once the struggling backend recovers (or gets replaced), it rejoins the health-checked pool automatically. Without any of this: the load balancer keeps blindly sending it a share of every request (naive round robin with no health awareness), each of those requests times out from the user's perspective, and the problem doesn't self-heal — a human has to notice and intervene manually while users keep experiencing failures the whole time.
