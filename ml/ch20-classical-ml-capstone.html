Chapter 20:
Classical ML Capstone

This is the Module 4 capstone. Chapters 15-19 built the pieces one at a time: the train/test workflow and metrics, linear and logistic regression, trees and forests, clustering and PCA, and finally pipelines and cross-validation for honest tuning. A real ML project uses ALL of them together, in order: load data, split it, scale it, try more than one model, cross-validate to pick a winner honestly, tune it, and only THEN evaluate on a test set you haven't touched until the very end. By the end of this chapter you'll have built exactly that — then Module 5 turns to deep learning, where the workflow is the same shape but the models get much bigger.

20.2 — Loading a Realistic Dataset

A candidate hiring dataset — thirty applicants, three features, a computed label with a few realistic "surprises" (candidates who didn't perform the way their scores predicted, exactly like real hiring outcomes never perfectly match the numbers on paper).

Building the DataFrame and the label
import pandas as pd data = pd.DataFrame({ 'name': ['Alice','Bob','Carol','Dave','Eve','Frank','Grace','Heidi','Ivan','Judy', 'Karl','Liam','Mia','Noah','Olive','Pete','Quinn','Ruth','Sam','Tina', 'Umar','Vera','Walt','Xena','Yuri','Zoe','Adam','Beth','Cody','Dana'], 'interview_score': [92,75,81,60,95,55,88,70,90,58, 85,65,93,62,78,52,84,71,89,66, 57,80,73,91,63,86,59,94,68,77], 'yrs_experience': [5,2,3,1,6,1,4,2,5,1, 4,2,6,1,3,1,4,2,5,2, 1,3,2,5,1,4,1,6,2,3], 'technical_score': [88,70,79,65,91,60,84,68,85,62, 80,63,89,58,77,55,82,69,86,64, 59,78,71,90,61,83,58,92,67,74], }) data['qualified'] = ((data['interview_score'] + data['technical_score']) / 2 >= 75).astype(int) # a few realistic surprises: candidates who didn't perform the way their scores predicted for idx in [5, 12, 19, 23]: data.loc[idx, 'qualified'] = 1 - data.loc[idx, 'qualified'] print(f'Rows: {len(data)}') print(f'Qualified count: {data["qualified"].sum()}') print(f'Columns: {list(data.columns)}')
20.3 — Split, Then Build the Preprocessing + Model Pipeline

The test set gets carved off FIRST and stays untouched until 20.6 — everything in between (comparing models, tuning) only ever sees the training portion, exactly like Chapter 15 warned.

Split, then a scaler + LogisticRegression pipeline
from sklearn.model_selection import train_test_split from sklearn.preprocessing import StandardScaler from sklearn.pipeline import Pipeline from sklearn.linear_model import LogisticRegression # data, X, y already built in 20.2 X = data[['interview_score','yrs_experience','technical_score']].values y = data['qualified'].values X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.25, random_state=42) print(f'Train size: {len(X_train)}, Test size: {len(X_test)}') pipe_lr = Pipeline([('scaler', StandardScaler()), ('clf', LogisticRegression())]) print('Pipeline built — the test set has NOT been touched yet.')
20.4 — Comparing Two Models Honestly, With Cross-Validation

Never pick a model from one lucky train/test split (Chapter 19's whole lesson). Cross-validate BOTH candidates on the training data only, and compare their mean scores.

LogisticRegression vs RandomForestClassifier, 5-fold CV
from sklearn.model_selection import cross_val_score from sklearn.preprocessing import StandardScaler from sklearn.pipeline import Pipeline from sklearn.linear_model import LogisticRegression from sklearn.ensemble import RandomForestClassifier # X_train, y_train already built in 20.3 pipe_lr = Pipeline([('scaler', StandardScaler()), ('clf', LogisticRegression())]) scores_lr = cross_val_score(pipe_lr, X_train, y_train, cv=5) print(f'LogisticRegression CV mean: {scores_lr.mean():.4f}') pipe_rf = Pipeline([('scaler', StandardScaler()), ('clf', RandomForestClassifier(n_estimators=100, random_state=42))]) scores_rf = cross_val_score(pipe_rf, X_train, y_train, cv=5) print(f'RandomForest CV mean: {scores_rf.mean():.4f}') winner = 'LogisticRegression' if scores_lr.mean() >= scores_rf.mean() else 'RandomForest' print(f'Winner: {winner}')
⭐
LogisticRegression (0.8200) edges out RandomForest (0.7800) on this dataset — a simpler model beating a more complex one is common when the underlying relationship is close to linear, and a reminder that "more powerful model" doesn't automatically mean "better model" for every dataset. Cross-validation, not intuition, made that call.
20.5 — Tuning the Winner With GridSearchCV

Now that LogisticRegression has won the comparison, search over its regularization strength C to find the best-performing setting — still using only the training data.

GridSearchCV over clf__C
from sklearn.model_selection import GridSearchCV from sklearn.preprocessing import StandardScaler from sklearn.pipeline import Pipeline from sklearn.linear_model import LogisticRegression # X_train, y_train already built in 20.3 param_grid = {'clf__C': [0.01, 0.1, 1, 10]} grid = GridSearchCV(Pipeline([('scaler', StandardScaler()), ('clf', LogisticRegression())]), param_grid, cv=5) grid.fit(X_train, y_train) print(f'Best params: {grid.best_params_}') print(f'Best CV score: {grid.best_score_:.4f}')
20.6 — Final Evaluation on the Held-Out Test Set

Only NOW does the test set get touched — one time, for the final honest number. grid.best_estimator_ is the already-tuned pipeline, ready to predict.

One evaluation, on data the whole process never saw
from sklearn.metrics import accuracy_score, precision_score, recall_score, f1_score, confusion_matrix # grid already fit in 20.5; X_test, y_test already built in 20.3 final_model = grid.best_estimator_ final_preds = final_model.predict(X_test) print(f'Test accuracy: {accuracy_score(y_test, final_preds):.4f}') print(f'Test precision: {precision_score(y_test, final_preds):.4f}') print(f'Test recall: {recall_score(y_test, final_preds):.4f}') print(f'Test F1: {f1_score(y_test, final_preds):.4f}') print(f'Confusion matrix: {confusion_matrix(y_test, final_preds).tolist()}')
🚫
Precision (0.6667) is noticeably lower than recall (1.0000) — the model catches every truly qualified candidate in the test set, but also flags some who weren't. Whether that tradeoff is acceptable depends entirely on the real-world cost of a false positive (extra interview time) versus a false negative (missing a great hire) — a business decision, not a modeling one.
20.7 — Interpreting the Model: Feature Importance

Before shipping a model, it's worth asking what it's actually relying on. A RandomForest fit on the same training data gives an interpretable feature ranking, even though LogisticRegression was the chosen model.

Which features actually drove the decision
from sklearn.ensemble import RandomForestClassifier # X_train, y_train already built in 20.3 rf_check = RandomForestClassifier(n_estimators=100, random_state=42) rf_check.fit(X_train, y_train) print(f'Feature importances (interview, experience, technical): {[round(f,4) for f in rf_check.feature_importances_]}')
📌
interview_score and technical_score matter roughly equally (0.42 and 0.32), yrs_experience less so (0.26) — sensible, since the label itself was built mostly from the two score columns. In a real project, this is exactly the kind of check that catches a model relying on something it shouldn't (like a proxy for a protected attribute) before it ever ships.
Chapter 20 Summary — All of Module 4
  • Split before anything else, and don't touch the test set again until the very end.
  • Scale + Pipeline keep preprocessing and modeling bundled and leak-proof.
  • Cross-validate to COMPARE models — don't trust a single split's verdict on which one is better.
  • GridSearchCV tunes the winner, still using only training-side cross-validation.
  • One final test-set evaluation, with the full metric picture (accuracy, precision, recall, F1) — not just accuracy.
  • Interpret before you ship — feature importances are a cheap sanity check on what the model actually learned.

💼 Classical ML Interview Prep

12 questions commonly asked in ML engineer / data scientist interviews, covering everything in Chapters 15-20. Click a question to reveal the answer. Not graded — this is here so you walk into an interview prepared, not just quizzed.

Why do you split data into train/validation/test sets instead of just train/test?▶
If you tune hyperparameters by repeatedly checking performance on the "test" set, you're indirectly fitting to it — its score stops being an honest estimate of real-world performance. The validation set (or cross-validation folds) is what you tune against; the test set is touched exactly ONCE, at the very end, to report a final, unbiased performance number.
When would you use precision over recall, or vice versa?▶
Prioritize precision (minimize false positives) when a false positive is costly — e.g. flagging a legitimate transaction as fraud and blocking a real customer. Prioritize recall (minimize false negatives) when missing a true case is the costly outcome — e.g. failing to flag an actual cancer case. Most real systems pick a threshold that balances the two based on the actual business/human cost of each error type, often summarized with an F1 score or a full precision-recall curve.
Why does logistic regression use the sigmoid function instead of just outputting a raw linear combination?▶
A raw linear combination (w·x + b) can output any real number, but a probability needs to be constrained to (0, 1). Sigmoid squashes any real number into that range while preserving order (higher linear score → higher probability), making the output directly interpretable as "probability of the positive class."
Why does an unconstrained decision tree tend to overfit, and how does a random forest fix that?▶
An unconstrained tree keeps splitting until every leaf is pure, which means it eventually carves out individual noisy training points rather than genuine patterns — perfect training accuracy, poor generalization. A random forest trains many trees, each on a random bootstrap sample of rows and a random subset of features per split, then averages/votes across them — each tree overfits DIFFERENTLY, and that noise mostly cancels out in the average, while genuine shared patterns reinforce each other.
What's the difference between bagging (e.g. Random Forest) and boosting (e.g. Gradient Boosting)?▶
Bagging trains many models INDEPENDENTLY and in parallel (on different bootstrap samples), then averages them — reduces variance. Boosting trains models SEQUENTIALLY, where each new model focuses specifically on correcting the errors the previous models made — reduces bias, but is more prone to overfitting if not carefully regularized (via learning rate, tree depth, number of rounds).
Why is k-fold cross-validation preferred over a single train/test split for evaluating a model?▶
A single split gives one performance number that depends heavily on which specific rows happened to land in the test set — an unlucky (or lucky) split can make a model look worse (or better) than it really is. K-fold cross-validation rotates through k different train/test splits and averages the results, giving a more stable, reliable performance estimate and, as a bonus, a sense of the VARIANCE of that estimate across folds.
Why must feature scaling (StandardScaler) be fit on the training set only, then applied to the test set — never fit on the whole dataset first?▶
Fitting the scaler on the full dataset (including test data) lets information about the test set's distribution (its mean/std) leak into how the training data gets scaled — a subtle form of data leakage that makes offline evaluation look better than real-world performance will be. Always: fit the scaler on X_train only, then use that SAME fitted scaler to transform both X_train and X_test.
What is a sklearn Pipeline, and why use one instead of calling steps manually?▶
A Pipeline chains preprocessing steps (scaling, encoding) and a final model into one object with a single .fit()/.predict() interface. Beyond convenience, it prevents data leakage automatically — when used inside cross-validation, the Pipeline refits the scaler on only each fold's training portion, which is easy to forget if you scale the data manually before splitting into folds.
What does GridSearchCV actually do, and what's the main downside?▶
It exhaustively tries every combination of hyperparameter values you specify, evaluating each combination with cross-validation, and returns the best-performing combination. The downside is cost: the number of combinations (and therefore training runs) grows multiplicatively with each additional hyperparameter and value tried — for large search spaces, RandomizedSearchCV (sampling a fixed number of random combinations) or Bayesian optimization methods are often used instead.
What is K-Means clustering, and what's a key limitation of it?▶
K-Means partitions data into K clusters by repeatedly assigning each point to its nearest cluster center, then recomputing each center as the mean of its assigned points, until convergence. Key limitations: you must choose K in advance (often via the "elbow method" or silhouette score), it assumes roughly spherical, similarly-sized clusters, and it's sensitive to the initial random placement of centers (mitigated in practice by running it multiple times with different initializations, which is scikit-learn's default behavior).
What does PCA do, and what does "explained variance ratio" mean?▶
PCA (Principal Component Analysis) finds new axes (principal components), each a linear combination of the original features, ordered so the first axis captures the most variance in the data, the second captures the most REMAINING variance orthogonal to the first, and so on. "Explained variance ratio" tells you what fraction of the total variance each component captures — letting you reduce to fewer dimensions while quantifying exactly how much information you're keeping vs. discarding.
If two models have similar accuracy, what else would you consider before choosing one for production?▶
Inference latency and cost (a Random Forest with 500 trees is slower to serve than a single Logistic Regression), interpretability requirements (some domains, like credit decisions, may legally require explainable models), training/retraining cost, robustness to slightly different data distributions, and how each model's errors are DISTRIBUTED (a model with the same overall accuracy but worse errors on a critical subgroup may be the wrong choice despite matching aggregate metrics).
