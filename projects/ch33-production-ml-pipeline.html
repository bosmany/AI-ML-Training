33.2 — Loading and Cleaning Genuinely Messy Data

A subscription-churn export: 24 raw rows, two exact duplicate rows (the same customer record submitted twice), two missing satisfaction-style values, and a text column with three different casings of the same three plan names. None of this is exaggerated — it's the normal condition of a CSV that came out of a real system.

The raw export — before anyone has touched it
import numpy as np import pandas as pd raw = pd.DataFrame({ 'customer_id': ['C001','C002','C003','C004','C005','C006','C007','C008','C009','C010', 'C011','C012','C013','C014','C015','C016','C017','C018','C019','C020', 'C021','C022','C009','C014'], 'tenure_months': [2,34,12,5,48,9,60,3,15,22, 7,40,4,18,55,10,2,27,6,33, 45,8,15,18], 'monthly_charges': [85.5,42.0,60.0,90.0,35.0,75.0,30.0,95.0,58.0,50.0, 80.0,38.0,np.nan,55.0,32.0,70.0,88.0,45.0,np.nan,40.0, 33.0,72.0,58.0,55.0], 'plan_type': ['Basic','standard','BASIC','basic','Standard','Basic','PREMIUM','basic','Standard','standard', 'basic','Premium','BASIC','Standard','premium','Basic','basic','standard','Basic','Standard', 'PREMIUM','Basic','Standard','Standard'], }) # label built from a rule (short tenure + high charges tends to churn), plus two realistic # "surprises" flipped in — exactly like Ch 20's hiring-capstone label raw['churned'] = ((raw['tenure_months'] < 20) & (raw['monthly_charges'].fillna(56.5) > 50)).astype(int) for idx in [4, 17]: raw.loc[idx, 'churned'] = 1 - raw.loc[idx, 'churned'] print(f'Raw rows: {len(raw)}') print(f'Missing monthly_charges: {raw["monthly_charges"].isna().sum()}') print(f'Duplicate rows: {raw.duplicated().sum()}') print(f'Unique plan_type spellings: {sorted(raw["plan_type"].unique())}')
⚠️
Two duplicate rows, two missing values, eight spellings of three plan names — if this DataFrame went straight into train_test_split, the duplicates would let the same customer's exact row appear in BOTH halves of the split (leaking test information into training), the missing values would crash most sklearn estimators, and 'Basic'/'basic'/'BASIC' would be treated as three unrelated categories. Cleaning isn't optional busywork — skipping it breaks the entire pipeline that follows.
Cleaning: dedupe, impute, normalize casing, encode
# raw already built above data = raw.drop_duplicates().copy() print(f'Rows after dropping duplicates: {len(data)}') median_charge = data['monthly_charges'].median() data['monthly_charges'] = data['monthly_charges'].fillna(median_charge) print(f'Median used to fill missing charges: {median_charge}') data['plan_type'] = data['plan_type'].str.lower().str.strip() print(f'Plan types after normalizing: {sorted(data["plan_type"].unique())}') plan_map = {'basic':0, 'standard':1, 'premium':2} data['plan_code'] = data['plan_type'].map(plan_map) print(f'Rows after cleaning: {len(data)}') print(f'Any remaining NaNs: {bool(data.isna().any().any())}')
⭐
Every cleaning decision is made ONCE, deliberately, and is repeatable — the median comes from the data itself, the casing rule (.str.lower().str.strip()) is a single line that generalizes to any future casing variant, and plan_map is an explicit, auditable encoding rather than a silent pd.factorize() that could assign different codes on a different run.
33.3 — Feature Engineering & the Train/Test Split

Exactly like Chapter 20's capstone: the test set is carved off FIRST, using the cleaned features, and stays untouched until 33.5's final evaluation.

Numeric feature matrix, then split
from sklearn.model_selection import train_test_split # data already cleaned in 33.2 X = data[['tenure_months','monthly_charges','plan_code']].values y = data['churned'].values X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.25, random_state=42) print(f'Train size: {len(X_train)}, Test size: {len(X_test)}')
33.4 — Comparing Two Models Honestly, With Cross-Validation

Same discipline as Chapter 20: cross-validate both candidates on the training data ONLY, and let the mean CV score — not a gut feeling — pick the winner.

LogisticRegression vs RandomForestClassifier, 5-fold CV
from sklearn.preprocessing import StandardScaler from sklearn.pipeline import Pipeline from sklearn.linear_model import LogisticRegression from sklearn.ensemble import RandomForestClassifier from sklearn.model_selection import cross_val_score # X_train, y_train already built in 33.3 pipe_lr = Pipeline([('scaler', StandardScaler()), ('clf', LogisticRegression())]) scores_lr = cross_val_score(pipe_lr, X_train, y_train, cv=5) print(f'LogisticRegression CV mean: {scores_lr.mean():.4f}') pipe_rf = Pipeline([('scaler', StandardScaler()), ('clf', RandomForestClassifier(n_estimators=100, random_state=42))]) scores_rf = cross_val_score(pipe_rf, X_train, y_train, cv=5) print(f'RandomForest CV mean: {scores_rf.mean():.4f}') winner = 'LogisticRegression' if scores_lr.mean() >= scores_rf.mean() else 'RandomForest' print(f'Winner: {winner}')
⭐
LogisticRegression (0.8167) beats RandomForest (0.7500) on this small, mostly-linear dataset — the same lesson as Ch 20: more model complexity isn't automatically better, and cross-validation, not intuition, is what makes that call defensible in front of a team.
33.5 — Tuning the Winner & the Final Held-Out Evaluation

Tune the winner's hyperparameters with GridSearchCV (still training-data only), then touch the test set exactly once for a full, honest metric picture.

GridSearchCV, then ONE test-set evaluation
from sklearn.model_selection import GridSearchCV from sklearn.metrics import accuracy_score, precision_score, recall_score, f1_score, confusion_matrix # X_train, y_train, X_test, y_test already built in 33.3 param_grid = {'clf__C': [0.01, 0.1, 1, 10]} grid = GridSearchCV(Pipeline([('scaler', StandardScaler()), ('clf', LogisticRegression())]), param_grid, cv=5) grid.fit(X_train, y_train) print(f'Best params: {grid.best_params_}') print(f'Best CV score: {grid.best_score_:.4f}') final_model = grid.best_estimator_ final_preds = final_model.predict(X_test) print(f'Test accuracy: {accuracy_score(y_test, final_preds):.4f}') print(f'Test precision: {precision_score(y_test, final_preds):.4f}') print(f'Test recall: {recall_score(y_test, final_preds):.4f}') print(f'Test F1: {f1_score(y_test, final_preds):.4f}') print(f'Confusion matrix: {confusion_matrix(y_test, final_preds).tolist()}')
🚫
Recall (1.0000) is higher than precision (0.8333) — the model catches every customer who actually churns in the test set, but also flags one who didn't. For a churn model, that's usually the right side to err on (a false-positive "at risk" flag costs a cheap retention email; a false negative costs a lost customer nobody reached out to) — but that's a business call the model itself can't make.
33.6 — Serialization & a Validating Serve Function

Training happens once; serving happens on every request. pickle the fitted pipeline, then wrap it in a function that validates its input BEFORE ever calling .predict() — exactly Chapter 31's pattern — and log every call through a small health monitor.

Serialize, validate, serve, and monitor
import pickle # final_model already fit in 33.5 model_blob = pickle.dumps(final_model) class HealthMonitor: def __init__(self): self.request_count = 0 self.error_count = 0 def log_request(self, is_error=False): self.request_count += 1 if is_error: self.error_count += 1 def health_report(self): error_rate = self.error_count / self.request_count if self.request_count else 0 return {'status': 'healthy' if error_rate < 0.1 else 'degraded', 'requests': self.request_count, 'error_rate': error_rate} monitor = HealthMonitor() def serve(record): required = ['tenure_months','monthly_charges','plan_code'] missing = [k for k in required if k not in record] if missing: monitor.log_request(is_error=True) return {'status': 400, 'error': f'missing fields: {missing}'} loaded_model = pickle.loads(model_blob) x = np.array([[record['tenure_months'], record['monthly_charges'], record['plan_code']]]) pred = int(loaded_model.predict(x)[0]) monitor.log_request(is_error=False) return {'status': 200, 'churn_prediction': pred} r1 = serve({'tenure_months': 3, 'monthly_charges': 90.0, 'plan_code': 0}) r2 = serve({'tenure_months': 40, 'monthly_charges': 35.0, 'plan_code': 1}) r3 = serve({'monthly_charges': 80.0, 'plan_code': 0}) # missing 'tenure_months' print(f'Request 1: {r1}') print(f'Request 2: {r2}') print(f'Request 3: {r3}') report = monitor.health_report() print(f'Health status: {report["status"]}') print(f'Requests logged: {report["requests"]}') print(f'Error rate: {report["error_rate"]:.4f}')
📌
Validation happens BEFORE the model ever sees the input — a malformed request never reaches .predict(), it gets a clean 400 and an error log entry instead. This is the difference between a service that fails loudly and safely, and one that throws an opaque stack trace at whoever is calling it.
33.7 — The Deployment Gate: Should This Model Actually Ship?

A tuned model with good test metrics still shouldn't automatically replace what's already running. Chapter 32's gating idea, applied here: compare the new model's held-out metric against the CURRENT production model's known metric, and only ship if it's actually better.

A simple ship/hold gate
# test_f1 (via f1_score(y_test, final_preds)) already computed in 33.5 CURRENT_PROD_F1 = 0.70 # the F1 of the model currently serving traffic new_model_f1 = f1_score(y_test, final_preds) decision = 'SHIP' if new_model_f1 >= CURRENT_PROD_F1 else 'HOLD' print(f'Current production F1: {CURRENT_PROD_F1}') print(f'New model test F1: {new_model_f1:.4f}') print(f'Gate decision: {decision}')
🚀
0.9091 beats 0.70 — SHIP. A gate this simple (one metric, one threshold) is a real, defensible pattern for a first production version. As the project matures, a real gate usually adds more conditions: a minimum sample size before trusting the comparison, a check that the new model isn't dramatically worse on any single subgroup even if the aggregate improves, and a rollback plan if post-deployment monitoring disagrees with the offline number.
Chapter 33 Summary
  • Clean first, always — duplicates, missing values, and inconsistent categories break everything downstream if skipped.
  • Split before comparing or tuning, and touch the test set exactly once, at the very end.
  • Cross-validate to compare, GridSearchCV to tune — both use only the training data.
  • Serialize with pickle, then validate before you predict — a serving function should reject bad input before the model ever sees it.
  • Monitor every request — request count and error rate are the cheapest possible early-warning system.
  • Gate the ship decision on a real number, compared against what's already in production — never ship "because it's newer."

💼 End-to-End ML Project Interview Prep

12 questions that probe real project experience — the "walk me through a project" and system-design-lite questions that separate candidates who've read about ML from candidates who've shipped it. Click a question to reveal the answer. Not graded — this is here so you walk into an interview prepared.

Walk me through how you'd take a model from a notebook to production.▶
Start from the notebook's messy exploratory code and turn it into a repeatable pipeline: wrap preprocessing and the model in a single Pipeline object so there's one .fit()/.predict() surface, not scattered cells. Move the final training script out of the notebook into a version-controlled file with fixed random seeds. Serialize the fitted pipeline (pickle/joblib) rather than re-running training to get predictions. Wrap it in a validating request handler with clear error responses. Add health/monitoring hooks. Gate the actual deployment on the new model beating a real baseline on held-out data, not just "it ran without errors." The notebook is for exploration; production code needs to be boring, reproducible, and testable.
How do you decide if a new model is ready to replace the current one in production?▶
Compare the new model against the CURRENT production model's real metric on the same held-out evaluation set — not against an old baseline from months ago. Require it to beat that number by a meaningful margin (not just numerically, since a tiny improvement inside noise isn't real). Check it isn't worse on any critical subgroup even if the aggregate improved. If possible, roll it out gradually (canary/shadow deployment) and compare live behavior before a full cutover, since offline metrics don't always predict production behavior perfectly.
What's your process when a stakeholder asks why the model made a specific prediction?▶
Start with global feature importances to explain what the model generally relies on, then move to a per-prediction explanation if needed — for tree-based models, SHAP or the individual decision path; for linear models, the weighted contribution of each feature for that specific input. Translate the technical explanation into the stakeholder's language (e.g. "this customer was flagged because of low tenure and high monthly charges, which historically correlate with churn" rather than raw coefficient values). If the answer reveals the model is relying on something inappropriate (a proxy for a protected attribute, a data leak), that's worth surfacing immediately, not just answering the immediate question.
How do you handle messy, real-world data before it ever reaches a model?▶
Profile it first — count missing values per column, check for exact and near-duplicate rows, and list unique values for every categorical column to catch casing/spelling inconsistencies. Make every cleaning decision explicit and documented (why median vs. mean imputation, why this casing rule) so it's auditable and repeatable on the next data pull. Never silently drop rows without knowing how many and why. And critically: any imputation statistic (median, mean) must be computed from the TRAINING split only once splitting happens, otherwise it's a subtle form of data leakage.
Why scale features inside a Pipeline instead of scaling the whole dataset before splitting?▶
Fitting a scaler on the full dataset lets statistics from the test set (its mean/std) leak into how the training data gets transformed — the model effectively gets a sneak peek at the test distribution. A Pipeline refits the scaler on only each split's (or each CV fold's) training portion automatically, which is easy to get wrong if scaling is done manually before splitting.
How would you design a deployment gate for a retrained model?▶
At minimum: a fixed held-out evaluation set the new model is scored against, a clear metric and threshold (often "beats current production by at least X"), and an automated pass/fail that blocks promotion on failure (this chapter's SHIP/HOLD pattern, or a CI job that fails the pipeline). A more mature gate adds: minimum sample size before trusting the comparison, per-subgroup checks so an aggregate win doesn't hide a regression for an important segment, and a fast rollback path in case production behavior disagrees with the offline evaluation.
What do you monitor once a model is live, and why?▶
Two categories: system health (request volume, error rate, latency — the same as any deployed service) and model health (prediction distribution shifts, input feature drift versus training data, and — once ground-truth labels eventually arrive — real-world accuracy compared against the offline evaluation numbers). A model can be "up" (200 responses, low latency) while quietly making worse predictions than it did at launch; system health alone won't catch that.
How do you decide which metric (accuracy, precision, recall, F1) matters most for a given project?▶
Start from the real-world cost of each error type. If a false negative is expensive (missing an actual churner, missing a fraud case), prioritize recall. If a false positive is expensive (wrongly flagging a good customer, blocking a legitimate transaction), prioritize precision. F1 is a reasonable single-number compromise when both error types matter roughly equally and classes are imbalanced. Accuracy alone is often misleading on imbalanced data — a model that always predicts "no churn" can have high accuracy and be completely useless.
What's the risk of skipping cross-validation and just using a single train/test split to choose between models?▶
A single split's result depends heavily on which specific rows happened to land in the test set — an unlucky or lucky split can make a genuinely worse model look better, or vice versa, especially on small datasets. Cross-validation rotates through multiple splits and averages the result, which is far more resistant to that kind of noise, and additionally reveals the VARIANCE of the estimate across folds — a model with a high mean but wildly different fold scores is a red flag single-split evaluation would completely hide.
How would you roll back a bad deployment?▶
Keep the previous model version readily available rather than deleting it the moment a new one ships — a model registry (even a simple versioned S3/blob path) makes "what was running yesterday" a solved problem. Route traffic back to the previous version via a config change, not a retrain — rollback should be measured in minutes. Only after traffic is safely back on the known-good version should the investigation into what went wrong with the new one begin.
What would you do if your model's test performance looked great but real production performance was much worse?▶
First suspect data leakage in the offline evaluation (a feature that wouldn't actually be available at prediction time, or test-set contamination from duplicates/preprocessing done before the split — exactly this chapter's cleaning warnings). Second, check for distribution drift — is production input data actually similar to what the model trained on? Third, check the serving code itself for a subtle bug (wrong feature order, a preprocessing step applied in training but forgotten in serving, or vice versa) — "training/serving skew" is one of the most common real causes of this exact symptom.
How do you balance model complexity (e.g. Random Forest) against simplicity (e.g. Logistic Regression) when both perform similarly?▶
When cross-validated performance is close, prefer the simpler model — it's cheaper and faster to serve, easier to explain to stakeholders and auditors, easier to debug when something goes wrong, and less likely to have overfit to quirks of the training data in ways that only show up later. Complexity should be justified by a real, validated performance gain, not chosen by default because it sounds more sophisticated.
