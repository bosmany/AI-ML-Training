Chapter 5:
OOP, Files & Errors

This is the Module 1 capstone — the chapter where everything from Chapters 1-4 comes together. Classes let you bundle data and behavior into one reusable unit (this is what a DataFrame or a PyTorch Model actually is). try/except lets your code survive bad data instead of crashing on it. And file I/O is how data gets from disk into your program in the first place. By the end, you'll build a Dataset class that loads, cleans, and reports on real messy data.

5.1 — Classes: Bundling Data and Behavior

Every object you've used so far — a list, a string, a dict — is an instance of a class someone else wrote. A class is a blueprint for creating your own objects that carry both data (attributes) and behavior (methods) together. self refers to "this particular instance" inside the class.

Your first class
class Model: def __init__(self, name, accuracy): self.name = name # attribute — data that belongs to THIS instance self.accuracy = accuracy def report(self): return f'{self.name}: {self.accuracy:.2%}' # method — behavior that uses the data def is_production_ready(self): return self.accuracy >= 0.9 # __init__ runs automatically when you create an instance: model_a = Model('ResNet50', 0.9423) model_b = Model('MobileNet', 0.847) print(model_a.report()) print(model_b.report()) print(f'model_a production ready? {model_a.is_production_ready()}') print(f'model_b production ready? {model_b.is_production_ready()}')
💡
This is exactly what real ML classes look like. sklearn.LinearRegression() is a class. When you call .fit(X, y), you're calling a method that reads and writes attributes on self (like the learned weights). You've been using this pattern since Chapter 1 — now you know how to build one.
5.2 — Methods Calling Other Methods

Methods on the same object can call each other through self. This lets you build up complex behavior from small, single-purpose pieces — exactly like functions in Chapter 3, but scoped to one object's data.

A class with several cooperating methods
class TrainingRun: def __init__(self, model_name, train_loss, val_loss): self.model_name = model_name self.train_loss = train_loss self.val_loss = val_loss def is_overfitting(self): return self.val_loss > self.train_loss * 1.2 def verdict(self): if self.is_overfitting(): # calling another method on self return 'Watch for overfitting' return 'On track' def summary(self): return f'{self.model_name}: train={self.train_loss:.4f}, val={self.val_loss:.4f} → {self.verdict()}' run = TrainingRun('CandidateScorer-v1', 0.12, 0.31) print(run.summary())
5.3 — try/except: Surviving Bad Data

Real data is messy. A `try` block runs code that might fail; if it raises an exception, the matching `except` block runs instead of crashing your whole program. This is the difference between a pipeline that dies on row 4,000 of 10,000, and one that skips the bad row and keeps going.

Catching a specific error type
raw_ages = ['28', '34', 'unknown', '25'] # 'unknown' will break int() valid_ages = [] errors = 0 for a in raw_ages: try: valid_ages.append(int(a)) except ValueError: errors += 1 print(f'Skipping bad value: {a!r}') print(f'Valid ages: {valid_ages}, errors: {errors}')
⚠️
Catch the specific exception, not everything. A bare except: catches EVERY error — including bugs in your own code — and hides them silently. Always name the exception type you expect: except ValueError:, except KeyError:, except FileNotFoundError:.
5.4 — raise: Creating Your Own Errors

Sometimes YOUR code should refuse bad input rather than silently accepting it. raise lets you throw an exception on purpose, with a message that explains exactly what went wrong.

Validating input before using it
def validate_accuracy(value): if not (0 <= value <= 1): raise ValueError(f'accuracy must be between 0 and 1, got {value}') return value try: print(f'Valid: {validate_accuracy(0.92)}') except ValueError as e: print(f'Error: {e}') try: print(f'Valid: {validate_accuracy(1.5)}') # 1.5 is not a valid accuracy! except ValueError as e: print(f'Error caught: {e}')
5.5 — Reading and Writing Files

open(filename, mode) gives you a file handle. Use 'w' to write, no mode (or 'r') to read. Always use with — it automatically closes the file for you, even if an error happens partway through.

Writing, then reading, a file
log_text = 'epoch,loss\n1,0.82\n2,0.61\n' with open('log.csv', 'w') as f: f.write(log_text) # file is automatically closed when this block ends with open('log.csv') as f: for line in f: # looping over a file gives you one line at a time print(line.strip()) # .strip() removes the trailing newline
💡
This console really does read and write files — Pyodide gives your Python code its own private virtual filesystem in the browser. It behaves exactly like real file I/O (nothing is saved to your actual computer, and it resets each time you reload the page).
5.6 — Parsing CSV Data

CSV (comma-separated values) is the most common format for tabular data. Python's built-in csv module handles the edge cases (commas inside quoted fields, etc.) that manual .split(',') gets wrong.

The csv module, reading rows as dicts
import csv csv_text = 'name,score\nAlice,0.92\nBob,0.75\nCarol,0.81\n' with open('scores.csv', 'w') as f: f.write(csv_text) with open('scores.csv') as f: reader = csv.DictReader(f) # uses the first line as column names automatically rows = list(reader) for row in rows: print(row) # each row is already a dict! {'name': 'Alice', 'score': '0.92'} # Remember: EVERY value from a CSV is a string — convert before math (Chapter 1!) scores = [float(row['score']) for row in rows] print(f'Average score: {sum(scores)/len(scores):.2%}')
5.7 — Parsing JSON Data

JSON is a text format that looks almost exactly like Python dicts and lists nested together — because that's exactly what it is. The json module converts between JSON text and real Python objects.

json.loads() and json.dumps()
import json json_text = '{"name": "ResNet50", "epochs": 50, "metrics": {"accuracy": 0.94, "loss": 0.12}}' config = json.loads(json_text) # JSON text -> Python dict print(type(config)) print(config['name']) print(config['metrics']['accuracy']) # nested access, same as Chapter 4 config['epochs'] = 100 back_to_text = json.dumps(config, indent=2) # Python dict -> JSON text, nicely formatted print(back_to_text)
Chapter 5 Summary
  • class / __init__ / self — bundle data (attributes) and behavior (methods) together. This is what every ML library object actually is.
  • try/except lets code survive bad data instead of crashing. Always catch specific exception types.
  • raise lets your own code refuse invalid input on purpose, with a clear error message.
  • with open(file, mode) as f: reads or writes files safely, closing them automatically.
  • csv.DictReader turns CSV rows into dicts. Remember: every value from a CSV arrives as a string.
  • json.loads() / json.dumps() convert between JSON text and Python dicts/lists.
