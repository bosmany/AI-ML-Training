Chapter 3:
Functions & Modules

Every ML library you'll ever use — sklearn.metrics.accuracy_score(), torch.nn.functional.relu() — is just a function someone wrote once and packaged so everyone could reuse it. This chapter teaches you to write your own: reusable, well-named blocks of logic that turn "code you wrote once" into "a tool you use forever."

🕐 ~50 min read 💻 5 exercises 🏗️ 1 mini project 🧪 Quiz to complete
3.1 — Why Functions? From Copy-Paste to Reusable Tools

Without functions, computing accuracy for three different models means writing the same loop three times. If you find a bug in that logic, you now have to fix it in three places — and you will forget one. A function packages logic once, under a name, so you write it once and call it forever.

✅
This is exactly what a library is. When you call accuracy_score(y_true, y_pred) from scikit-learn, you're calling a function someone else wrote. By the end of this chapter you'll understand precisely what's happening inside that call.
3.2 — def, Parameters, and return vs print

Define a function with def name(parameters):. Inside, return sends a value back to whoever called the function, so it can be stored or reused. This is completely different from print(), which only displays text on screen and gives nothing back.

return vs print — the most important distinction in this chapter
def compute_accuracy(predictions, targets): correct = 0 for p, t in zip(predictions, targets): if p == t: correct += 1 return correct / len(targets) # hands the value back to the caller preds = [1,0,1,1] targets = [1,0,0,1] acc = compute_accuracy(preds, targets) # the returned value is now stored in acc print(f'Accuracy: {acc:.2%}') # we can use acc anywhere now — format it, compare it, save it # Contrast: a function that only prints, returns nothing useful def show_accuracy(predictions, targets): print(f'Accuracy: {compute_accuracy(predictions, targets):.2%}') # no return statement — this function's result is None result = show_accuracy(preds, targets) print(f'What show_accuracy() gave back: {result}') # None — it never returned anything
⚠️
Beginner trap: A function with no return statement doesn't fail — it silently returns None. If you ever see None where you expected a number, check whether the function you called actually has a return.
3.3 — Default Parameters and Keyword Arguments

A parameter can have a default value, used only when the caller doesn't supply one. This lets a function have sensible behavior out of the box, while still being configurable.

Defaults make a function flexible without being complicated
def format_metric(name, value, decimals=4): return f'{name}: {value:.{decimals}f}' print(format_metric('loss', 0.123456)) # uses the default: 4 decimals print(format_metric('accuracy', 0.9423, decimals=2)) # overrides the default: 2 decimals print(format_metric(name='f1', value=0.88, decimals=1)) # keyword arguments — order no longer matters
3.4 — *args and **kwargs

Sometimes you don't know in advance how many arguments a function will receive. *args collects any number of extra positional arguments into a tuple. **kwargs collects any number of extra keyword arguments into a dict.

*args — accept any number of values
def average_loss(*losses): print('Received:', losses, '— a tuple, no matter how many arguments') return sum(losses) / len(losses) print(f'Average: {average_loss(0.5, 0.3, 0.4, 0.2):.4f}') # 4 values print(f'Average: {average_loss(0.9, 0.1):.4f}') # 2 values — same function # **kwargs — accept any number of named settings def train_model(name, **settings): print(f'Training {name} with settings:', settings) train_model('ResNet50', epochs=50, lr=0.001, batch_size=32)
💡
You'll see this constantly in real ML code: model.fit(X, y, epochs=10, batch_size=32, **kwargs) is exactly this pattern — accept the required arguments, then forward whatever extra settings the caller wants.
3.5 — Lambda: Small, Throwaway Functions

A lambda is a function written in one line, with no name, usually passed directly into another function. The most common use in ML code is as the key for sorting.

lambda + sorted() — ranking model results
scores = [('alice', 0.72), ('bob', 0.95), ('carol', 0.81), ('dave', 0.60)] # Sort by the SECOND item in each tuple (the score), highest first ranked = sorted(scores, key=lambda x: x[1], reverse=True) print(ranked) print(f'Top performer: {ranked[0][0]}') # A named function does the same thing, just with more words: def get_score(item): return item[1] print(sorted(scores, key=get_score, reverse=True))
3.6 — Scope: Local vs Global Variables

A variable created inside a function only exists inside that function — this is local scope. It disappears the moment the function returns, and it never affects a variable of the same name outside the function, unless you explicitly say global.

The classic scope surprise
total_processed = 0 # global variable def process_batch_wrong(batch_size): total_processed = total_processed + batch_size # this LOOKS right... print(total_processed) # process_batch_wrong(32) # <-- would raise UnboundLocalError! Python sees the assignment # and assumes total_processed is a NEW local variable for the whole function body — # so reading it on the right-hand side before it's assigned fails. def process_batch(batch_size): global total_processed # explicitly say: use the outer variable, don't make a local one total_processed += batch_size print(f'Running total: {total_processed}') process_batch(32) process_batch(16) process_batch(8) print(f'Total processed: {total_processed}')
⚠️
Best practice: avoid global when you can. It's shown here because you WILL encounter it, but the cleaner pattern is almost always to return the new value and let the caller decide what to do with it. Fewer hidden side effects, easier to test.
3.7 — Modules: import and Organizing Code Across Files

A module is just a .py file. import lets you use functions defined in another file — this is exactly how import numpy as np or from sklearn.metrics import accuracy_score work. Python also ships a large standard library of built-in modules you can import for free.

Using Python's built-in modules
import math print(math.sqrt(16)) # 4.0 — same idea as **0.5, but more explicit print(math.pi) # 3.14159... print(math.ceil(4.1)) # 5 — round up print(math.floor(4.9)) # 4 — round down # Importing just one function, so you don't need the "math." prefix: from math import sqrt print(sqrt(25)) # 5.0 # This is exactly the pattern you'll use for every ML library: # import numpy as np # from sklearn.metrics import accuracy_score # import pandas as pd
⭐
If you were writing your own file (say, metrics.py) full of functions, another file could use them with from metrics import compute_accuracy. This is how real ML projects are structured: one file per concern (data loading, model, metrics, training loop), stitched together with imports.
Chapter 3 Summary
  • def name(params): ... return value — functions package logic once so you can reuse it everywhere. return hands a value back; print just displays it.
  • Default parameters (decimals=4) make functions flexible without forcing every caller to specify everything.
  • *args collects extra positional arguments into a tuple. **kwargs collects extra keyword arguments into a dict.
  • lambda writes a tiny function inline — most often as the key for sorted().
  • Scope: variables inside a function are local by default. Use global to modify an outer variable — but prefer returning values instead when you can.
  • Modules: import math, from math import sqrt — every ML library (NumPy, Pandas, sklearn) is used exactly this way.

📋 What to Build

  • precision(preds, targets) — of everything you predicted positive (1), what fraction was actually positive
  • recall(preds, targets) — of everything that was actually positive, what fraction did you correctly predict
