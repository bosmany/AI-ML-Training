Chapter 7:
Python for DevOps & APIs

This is the Module 1 capstone. Every script that automates something real — a deployment, a data pull, a health check, a CI job — is glued together with the same handful of tools: command-line arguments so it's configurable, JSON so it can talk to other systems, environment variables so secrets and config stay out of your code, and async so it doesn't sit idle waiting on slow network calls. By the end, you'll build a Health-Check CLI that ties all of it together — then Chapters 8+ turn to the math underneath every model.

7.1 — Command-Line Arguments With argparse

A script that hard-codes its inputs only ever does one thing. argparse turns a script into a configurable tool: python score.py --name Alice --score 0.92. Instead of reading sys.argv by hand, you declare what arguments exist and argparse parses, type-converts, and validates them for you.

A parser with a required flag, a typed flag, and a boolean flag
import argparse parser = argparse.ArgumentParser(description='Score a candidate') parser.add_argument('--name', required=True) parser.add_argument('--score', type=float, required=True) parser.add_argument('--verbose', action='store_true') # parse_args() with NO argument reads real sys.argv (the actual command line). # Passing an explicit list — as below — is also how you unit-test a CLI tool. args = parser.parse_args(['--name', 'Alice', '--score', '0.92', '--verbose']) print(f'Name: {args.name}') print(f'Score: {args.score}') # already a float, not a string print(f'Verbose: {args.verbose}')
⭐
Passing an explicit list to parse_args() is the key trick for THIS console — a browser has no real command line, so every argparse exercise here builds the argument list as a Python list instead of typing it at a terminal. The behavior is identical either way.
7.2 — JSON: The Universal Data Format

Almost every API, config file, and log line you'll touch uses JSON (JavaScript Object Notation) — a text format that maps directly onto Python dicts and lists. json.dumps() converts a Python object TO a JSON string; json.loads() parses a JSON string INTO Python objects.

Round-tripping a dict through JSON
import json candidate = {'name': 'Bob', 'score': 0.75, 'skills': ['python', 'sql']} payload = json.dumps(candidate) # Python dict -> JSON string (what you'd SEND to an API) print(f'Payload: {payload}') parsed = json.loads(payload) # JSON string -> Python dict (what you'd RECEIVE) print(f'Name back: {parsed["name"]}') print(f'Type: {type(parsed)}')
7.3 — Calling APIs With requests

In real production code, you fetch data from other services over HTTP with the requests library — a real outbound network call. This can't run inside this browser sandbox reliably (blocked by CORS, and no notion of "the internet" the way a real script has), so treat the block below as reference: this is EXACTLY the pattern you'll use outside this course, and it's what returns the kind of JSON string the runnable exercise below parses.

Terminal / real script — reference only, no Run button
import requests response = requests.get('https://api.example.com/candidates', timeout=5) response.raise_for_status() # raises if status is 4xx/5xx instead of failing silently data = response.json() # .json() == json.loads(response.text), done for you print([c['name'] for c in data['results']])

Notice the shape: response.json() gives you back exactly the nested dict/list structure json.loads() would. That's why the exercise below is realistic practice even without a live network call — it hands you the JSON text an API would have returned, and you parse it the same way.

Parsing "as if" this came from response.json()
import json api_response_text = '{"results": [{"name": "Alice", "score": 0.92}, {"name": "Dave", "score": 0.60}]}' data = json.loads(api_response_text) names = [r['name'] for r in data['results']] print(f'Names: {names}')
7.4 — subprocess: Running Other Programs From Python

DevOps scripts frequently need to run a shell command and use its output — deploying with docker, checking git status, running a linter. The subprocess module spawns a real OS process, which (like requests) needs a real operating system that this sandboxed browser environment doesn't provide — reference only.

Terminal / real script — reference only, no Run button
import subprocess result = subprocess.run( ['git', 'status', '--short'], capture_output=True, text=True, check=True ) print(result.stdout) # captured stdout from the git process, as a normal Python string
⚠️
Always pass a list, never a raw shell string, unless you fully trust the input — subprocess.run(user_input, shell=True) is a classic command-injection vulnerability. ['git', 'status', arg] passes each piece as a separate, un-interpreted argument.
7.5 — Environment Variables: Config Without Hard-Coding Secrets

An API key or database URL should never be hard-coded into a script (and definitely never committed to git). The standard fix: read it from an environment variable via os.environ, with a sane default for local development.

Reading config from the environment, with a fallback
import os os.environ['APP_ENV'] = 'production' # normally set OUTSIDE Python (by your shell, Docker, CI) env = os.environ.get('APP_ENV', 'development') print(f'Environment: {env}') missing = os.environ.get('MISSING_VAR', 'default-value') # not set -> uses the fallback print(f'Missing var falls back to: {missing}')
⭐
os.environ.get(key, default) never raises — unlike os.environ[key], which throws KeyError if the variable isn't set. Always use .get() with an explicit default for optional config.
7.6 — Async Basics: Not Waiting Around

If a script calls 5 slow APIs one after another, it waits for each to finish before starting the next — even though it's just sitting idle during the network wait. async/await lets I/O-bound calls run concurrently: start all 5, then wait for all of them together with asyncio.gather().

Three "slow" calls running concurrently
import asyncio async def fetch_candidate(name, delay): await asyncio.sleep(delay) # stands in for a slow network call return f'{name} fetched' async def main(): # all three sleeps happen AT THE SAME TIME, not one after another results = await asyncio.gather( fetch_candidate('Alice', 0.01), fetch_candidate('Bob', 0.01), fetch_candidate('Carol', 0.01), ) for r in results: print(r) await main() # this console already runs inside an event loop, so 'await' works at top level
💡
asyncio.gather() preserves order in its RETURN VALUE regardless of which finishes first — results[0] is always Alice's result, results[1] is always Bob's — even though the three sleeps happen concurrently under the hood.
Chapter 7 Summary — Module 1 Complete
  • argparse turns a script into a configurable CLI tool.
  • JSON (json.dumps / json.loads) is how programs exchange structured data over APIs.
  • requests makes real HTTP calls; subprocess runs real shell commands — both are one-liners once you know the shape.
  • os.environ.get(key, default) keeps config and secrets out of your source code.
  • asyncio.gather() runs I/O-bound work concurrently instead of one call at a time.
