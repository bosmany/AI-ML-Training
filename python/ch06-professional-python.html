6.1 — Virtual Environments & pip: Isolating Dependencies

Every Python project depends on specific versions of specific packages. Install everything globally on your machine and eventually two projects will need different, incompatible versions of the same package — this is "dependency hell." A virtual environment is a private, isolated copy of Python + installed packages, scoped to one project. This is the single most important habit that separates a script kiddie's machine from a professional setup.

💻
These commands run in a real terminal, not this browser
Pyodide (the in-browser Python powering this course) is a sandboxed WebAssembly runtime — it has no real filesystem-level "processes" to isolate. The blocks below are reference material: read them, understand what each line does, and try them for real next time you're in a terminal with Python installed.
Terminal — reference only, no Run button
# create an isolated environment named .venv, scoped to this project folder python -m venv .venv # activate it — your shell prompt changes to show you're "inside" it source .venv/bin/activate # macOS / Linux .venv\Scripts\activate # Windows # now pip install/uninstall only affects THIS environment pip install pandas numpy scikit-learn # snapshot the exact versions you have — this file goes into git pip freeze > requirements.txt # anyone (or any server) can recreate your exact environment with: pip install -r requirements.txt # leave the environment deactivate

The habit that matters most: never install packages globally. Every project gets its own .venv, and every project ships a requirements.txt so its exact dependencies are reproducible on any machine — yours, a teammate's, or a production server. This is also exactly what a Dockerfile does at a higher level (Module 7), and what a Colab notebook's first cell (!pip install ...) is doing when later chapters use it.

CommandWhat it does

6.2 — Custom Exceptions: Errors That Mean Something

Chapter 5 caught built-in errors like ValueError. In real code you also want to raise errors that describe YOUR problem domain — "this candidate's score is out of range" is far more useful to whoever reads the traceback than a generic ValueError. Define a custom exception by subclassing Exception.

A domain-specific exception
class InsufficientScoreError(Exception): pass def check_score(score): if score < 0.8: raise InsufficientScoreError(f"score {score} below threshold") return "qualified" try: check_score(0.6) except InsufficientScoreError as e: print(f'Caught: {e}') print(check_score(0.9))
⭐
A custom exception is just a class — usually with just pass in its body. Its NAME is the whole point: except InsufficientScoreError lets calling code react to exactly this failure without accidentally swallowing unrelated bugs (which a bare except: or overly broad except Exception: would do).
6.3 — Context Managers: Guaranteed Cleanup

Chapter 5's with open(...) as f: automatically closes the file even if an error happens inside the block. That's a context manager. You can write your own with @contextmanager from the contextlib module: the code before yield runs on entry, the code after (inside finally) runs on exit — no matter what.

A custom context manager
from contextlib import contextmanager @contextmanager def stage(name): print(f'START {name}') try: yield finally: print(f'END {name}') with stage('validation'): print('checking data')

Why this matters for DevOps and ML alike: any resource that needs guaranteed cleanup — a database connection, a temp file, a GPU memory allocation, a timer around a training step — belongs in a context manager, not in ad-hoc code you might forget to run on the error path.

6.4 — Decorators: Wrapping Behavior Around a Function

A decorator is a function that takes a function and returns a new function that wraps it — adding behavior before/after the original runs, without touching the original's code. The @decorator_name syntax above a function definition is shorthand for func = decorator_name(func).

A logging decorator
def log_call(func): def wrapper(*args, **kwargs): print(f'Calling {func.__name__} with {args}') result = func(*args, **kwargs) print(f'Result: {result}') return result return wrapper @log_call def add(a, b): return a + b add(3, 5)
⭐
*args, **kwargs let a decorator wrap ANY function — one with 0 arguments, one with 5, one with keyword arguments — the wrapper doesn't need to know the original's exact signature. You'll see this exact pattern in Flask/FastAPI route decorators and PyTorch's @torch.no_grad().
6.5 — Generators: Lazy Sequences With yield

A normal function that builds a list computes and holds EVERY value in memory at once. A generator function — one that uses yield instead of return — produces values one at a time, on demand. For a 10-million-row dataset, the difference is "crashes with MemoryError" vs. "runs fine."

A generator that filters lazily
def high_scores(scores): for s in scores: if s >= 80: yield s scores = [92, 75, 81, 60, 88] result = list(high_scores(scores)) # list() pulls every value out of the generator print(f'Qualified: {result}') print(f'Count: {len(result)}')

Calling high_scores(scores) does NOT run the function body yet — it returns a generator object. Only iterating it (with a for loop, list(), or next()) actually executes code, one yield at a time. A generator EXPRESSION — (s for s in scores if s >= 80) — does the same thing as a one-liner, the lazy counterpart to a list comprehension.

6.6 — Logging: print() That Scales

print() is fine for a 20-line script. Real programs need levels (DEBUG/INFO/WARNING/ERROR), the ability to turn noisy output off without deleting code, and timestamps — that's the logging module.

Logging levels
import logging # force=True resets handlers each call — needed because this console reuses # the same Python session, so re-running this cell wouldn't stack duplicate handlers logging.basicConfig(level=logging.INFO, format='%(levelname)s: %(message)s', force=True) logger = logging.getLogger('app') logger.info('starting validation') logger.warning('score close to threshold') # logger.debug(...) would NOT print — level=INFO filters out DEBUG
⚠️
Always pass force=True when re-configuring logging in this console — without it, calling basicConfig() a second time in the same session is a no-op (Python only lets you configure the root logger once by default), and old handlers silently keep pointing at stale output streams.
6.7 — Testing: Trusting Your Own Code

A function you never test is a function you're hoping works. unittest (built into Python, no install needed) lets you write assertions that run automatically and report exactly what broke. assertEqual(a, b) fails loudly with a diff if a != b, instead of silently passing wrong output downstream.

A real unittest.TestCase
import unittest def add(a, b): return a + b class TestAdd(unittest.TestCase): def test_add_positive(self): self.assertEqual(add(2, 3), 5) def test_add_negative(self): self.assertEqual(add(-1, 1), 0) # TextTestRunner instead of unittest.main() — main() calls sys.exit(), which # would kill this console session runner = unittest.TextTestRunner(verbosity=0) result = runner.run(unittest.TestLoader().loadTestsFromTestCase(TestAdd)) print(f'Tests run: {result.testsRun}, Failures: {len(result.failures)}')
🚫
Never call unittest.main() in this console — it calls sys.exit() when done, which raises SystemExit and looks like a crash here. Use TextTestRunner().run(...) instead, exactly as shown above — this is what you'd also do to run tests programmatically from another script.
Chapter 6 Summary
  • Virtual environments + requirements.txt make your project's dependencies reproducible on any machine.
  • Custom exceptions (subclass Exception) give errors names that mean something in your domain.
  • Context managers (@contextmanager + try/finally) guarantee cleanup code runs, even on failure.
  • Decorators wrap a function with extra behavior using @name, without changing its source.
  • Generators (yield) produce values lazily — essential for large datasets.
  • logging replaces print() with levels or professional code. unittest replaces "I ran it once and it looked fine" with real, repeatable tests.
