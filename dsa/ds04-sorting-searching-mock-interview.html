4.1 — Sorting From Scratch: Merge Sort & Quicksort

sorted() is what you'd use in real code — but interviewers ask you to implement sorting from scratch because it tests recursion, divide-and-conquer thinking, and complexity analysis all at once. Merge sort splits the array in half, recursively sorts each half, then merges the two sorted halves back together in one linear pass.

Merge sort — divide, recurse, merge
def merge_sort(arr): if len(arr) <= 1: return arr[:] mid = len(arr) // 2 left = merge_sort(arr[:mid]) # recursively sort left half right = merge_sort(arr[mid:]) # recursively sort right half return merge(left, right) def merge(left, right): result = [] i = j = 0 while i < len(left) and j < len(right): if left[i] <= right[j]: result.append(left[i]); i += 1 else: result.append(right[j]); j += 1 result.extend(left[i:]) # append whatever's left over result.extend(right[j:]) return result original = [38, 27, 43, 3, 9, 82, 10] print(f'Before: {original}') print(f'After (merge sort): {merge_sort(original)}')
🧠
Why merge sort is O(n log n): the array is halved at each level of recursion, giving log n levels total (halving n items until you reach 1 takes log₂n steps). At EACH level, the total work across all the merges is O(n) (every element gets touched once per level). Levels × per-level work = O(log n) × O(n) = O(n log n).

Quicksort takes a different approach: pick a pivot, partition everything into "less than pivot" and "greater than pivot," then recursively sort each partition. No merge step needed — by the time both partitions are sorted, the whole array is sorted.

Quicksort — pick a pivot, partition, recurse
def quicksort(arr): if len(arr) <= 1: return arr[:] pivot = arr[len(arr) // 2] left = [x for x in arr if x < pivot] mid = [x for x in arr if x == pivot] right = [x for x in arr if x > pivot] return quicksort(left) + mid + quicksort(right) original2 = [38, 27, 43, 3, 9, 82, 10] print(f'Before: {original2}') print(f'After (quicksort): {quicksort(original2)}')
⚠️
Quicksort's average case is O(n log n), same as merge sort — but its WORST case is O(n²). If the pivot consistently ends up being the smallest or largest element (e.g. an already-sorted array with a naive "always pick the first element" pivot strategy), one partition is empty and the other has n-1 elements every time — that's n levels of recursion instead of log n, giving O(n²). Merge sort has no such worst case; it's ALWAYS O(n log n) regardless of input order — that reliability is merge sort's main advantage, at the cost of needing O(n) extra space for the merge step (quicksort can sort in-place).
4.2 — Binary Search: Requires Sorted Input, No Exceptions

Binary search finds a target in O(log n) by repeatedly checking the middle element and eliminating HALF the remaining search space based on whether the target is smaller or larger — but that elimination logic is only valid if the array is sorted. On unsorted data, it doesn't just run slower — it can return a flatly WRONG answer, silently.

Binary search (iterative) — and what happens on unsorted input
def binary_search(arr, target): left, right = 0, len(arr) - 1 while left <= right: mid = (left + right) // 2 if arr[mid] == target: return mid elif arr[mid] < target: left = mid + 1 # target must be in the right half else: right = mid - 1 # target must be in the left half return -1 sorted_data = [2, 5, 8, 12, 16, 23, 38, 45, 56, 72, 91] print(f'binary_search(sorted_data, 23): {binary_search(sorted_data, 23)}') print(f'binary_search(sorted_data, 24): {binary_search(sorted_data, 24)}') # the EXACT same numbers, but NOT sorted unsorted_data = [23, 5, 91, 12, 2, 45, 8, 72, 16, 56, 38] wrong_result = binary_search(unsorted_data, 5) correct_index = unsorted_data.index(5) print(f'binary_search on UNSORTED data looking for 5: {wrong_result}') print(f'Actual index of 5 in the unsorted list: {correct_index}') print(f'Binary search gave the right answer: {wrong_result == correct_index}')
🚫
On the unsorted array, binary_search(unsorted_data, 5) returns -1 — claiming 5 isn't there — even though 5 IS in the list at index 1. This is worse than crashing: it's a silent, confident, WRONG answer that looks like a valid result. This is exactly why "is this data actually sorted?" is one of the first clarifying questions worth asking before reaching for binary search in an interview.
4.3 — The Big-O Cheat Sheet (Genuinely Worth Memorizing)

Every complexity class below has shown up somewhere in DSA 1-4 already — this table just collects them, ordered best to worst, with one canonical example each.

ComplexityNameCanonical example
O(1)ConstantAccessing a list element by index — arr[5]; a hash map lookup
O(log n)LogarithmicBinary search — each step eliminates half the remaining search space
O(n)LinearScanning every element once — linear search, summing a list, a single hash-map pass
O(n log n)LinearithmicEfficient comparison sorting — merge sort (always), quicksort (average case)
O(n²)QuadraticNested loop comparing every pair — naive duplicate check, bubble sort, quicksort's worst case
O(2ⁿ)ExponentialGenerating all subsets of a set (DSA 2 backtracking) — naive un-memoized recursive Fibonacci
⭐
Memorization trick: each row roughly matches a code SHAPE you can recognize on sight. O(1): no loop. O(log n): a loop that HALVES its range each time. O(n): one loop over the input. O(n log n): a loop combined with a halving structure (or a call to a good sort). O(n²): a loop nested inside a loop, both over the input. O(2ⁿ): recursion that branches into 2 (or more) further calls per call, without memoization.
DSA 4 Summary
  • Merge sort is always O(n log n) (log n levels × O(n) work per level) at the cost of O(n) extra space; quicksort is O(n log n) average but O(n²) worst case, in exchange for in-place sorting.
  • Binary search is O(log n) but REQUIRES sorted input — on unsorted data it doesn't just slow down, it can return a confidently wrong answer.
  • The Big-O cheat sheet (O(1) → O(log n) → O(n) → O(n log n) → O(n²) → O(2ⁿ)) maps directly onto recognizable code shapes — no loop, halving loop, single loop, sort, nested loop, unmemoized branching recursion.
  • Everything in DSA 1-4 — two pointers, sliding window, hash maps, recursion, trees, graphs, backtracking, DP, greedy, sorting, searching — is the actual content of most coding-interview rounds.

💼 The Interview Process Itself — Interview Prep

12 questions about the meta-skill that no amount of algorithm knowledge substitutes for: how to actually PERFORM in a live coding interview. Click a question to reveal the answer. Not graded — this is the difference between knowing DSA and passing an interview with it.

How do you "think out loud" without it turning into meaningless rambling?▶
Narrate DECISIONS, not every keystroke: "I'm going to use a hash map here because I need O(1) lookups" is useful; "now I'm typing the word def" is not. A good rhythm is: state the plan for the next few lines, write them, then briefly confirm they did what you expected before moving on. Silence for 20-30 seconds while you think is fine and normal — silence for 3 minutes with no signal of what you're doing is what actually worries interviewers.
How should you handle genuinely not knowing something, instead of guessing or freezing?▶
Say so plainly and pivot to what you DO know: "I don't remember the exact name for this technique, but the idea I want is X" is far stronger than confidently stating something wrong, or going silent. Interviewers are explicitly listening for how you handle not knowing something — it's one of the few moments that reliably reveals whether you're someone who can be trusted to say "I'm not sure" honestly on a real team, versus someone who bluffs.
What clarifying questions are actually worth asking before you start coding?▶
Ones that could change your approach: What's the expected input size (does it affect whether O(n²) is acceptable)? Can the input contain duplicates, negative numbers, or be empty? Is the input already sorted (this chapter's binary search question)? What should happen on an invalid/edge-case input — error, or a specific sentinel value? Asking something that DOESN'T affect your approach (like variable naming preferences) wastes time and signals you don't know what actually matters yet.
Beyond "did the code run," what do interviewers actually score you on?▶
Problem-solving approach (did you break the problem down sensibly, or jump straight to code with no plan), communication (could someone follow your reasoning without reading the code), correctness on edge cases (empty input, single element, duplicates — not just the happy path), time/space complexity awareness (can you state it, and does it match what you actually wrote), and code quality (clear variable names, no unnecessary complexity) — roughly in that order of weight for most companies.
How do you communicate a "brute force first, then optimize" approach without it sounding like you don't know the good answer?▶
Frame it as a deliberate strategy, not a fallback: "Let me start with the brute-force O(n²) approach to make sure I understand the problem correctly, then I'll optimize it" — said BEFORE you write the brute force, not as an apology after. This proves you know it's suboptimal (so it doesn't read as your ceiling), gives you a working solution as a safety net if time runs out, and often reveals the insight needed for the optimization while you're implementing the simple version.
What's actually different between a whiteboard interview and a live-coding-tool (e.g. CoderPad, HackerRank) interview?▶
Whiteboard: no syntax checking, no running the code, so pseudocode-adjacent syntax is fine and expected — focus entirely on logic and clear structure, and TALK THROUGH what the code does since the interviewer can't run it to check. Live-coding tools: the code usually needs to actually run (sometimes against real test cases), so correct syntax matters and you should actually execute it against a few cases before declaring done — but resist the urge to over-focus on getting every semicolon right at the expense of explaining your reasoning, which still matters just as much.
You've been stuck for 5 minutes with no progress. What do you do?▶
Say out loud that you're stuck and what specifically is blocking you ("I'm not sure how to handle the case where two intervals share an endpoint") — this is a legitimate move, not a failure, and it's usually the fastest way to get a useful hint. Fall back to a smaller/simpler version of the problem to build momentum (solve it for a fixed tiny input by hand, or drop a constraint temporarily), or fall back to the brute-force approach you already know works. Staying silent while stuck is the worst option — it gives the interviewer nothing to work with and no way to help.
How do you actually test your own code before declaring it done, in an interview setting?▶
Walk through your code by hand on the ORIGINAL example first, then explicitly check edge cases out loud: empty input, a single element, all-identical elements, the largest/smallest possible values, and — for this module's topics specifically — an already-sorted vs. reverse-sorted input for a sorting problem, or a graph with an isolated node for a graph problem. Doing this UNPROMPTED, before the interviewer asks "have you tested edge cases?", is a strong positive signal.
The interviewer gives you a hint. How should you respond?▶
Take it and use it — a hint is not a penalty, and most interview rubrics explicitly expect some hints to be given, especially for harder problems. Acknowledge it genuinely ("oh, that's a good point — let me reconsider") rather than pretending you were already about to say that, and show you actually integrated it into your thinking rather than just changing the code without explaining why. Ignoring a hint, or taking it personally, reads far worse than needing one in the first place.
How should you manage your time across a single interview problem (say, a 30-45 minute slot)?▶
Roughly: a few minutes clarifying the problem and stating an approach, the bulk of the time actually coding, and a few minutes at the end for testing and complexity discussion — don't let clarification eat 15 minutes, and don't skip testing to save 2 minutes. If you notice you're past the halfway mark with no working approach yet, that's the moment to fall back to brute force explicitly (see above) rather than continuing to search for the optimal solution and risking finishing with nothing.
What does "exceeds expectations" typically look like versus "meets expectations," for the same correctly-solved problem?▶
"Meets" usually means: correct solution, reasonable complexity, adequate communication, found with some interviewer guidance. "Exceeds" usually adds: reached the optimal solution with minimal or no hints, proactively identified and tested edge cases before being asked, clearly articulated complexity trade-offs between multiple valid approaches (not just the one you wrote), and communicated clearly enough that the interviewer never had to ask "wait, why did you do that?" These are relative to the SPECIFIC role level being interviewed for — an "exceeds" bar for an entry-level role is lower than for a senior one.
How should you close out a problem once your code appears to work?▶
State the time and space complexity of your final solution explicitly (don't wait to be asked), briefly mention any edge cases you handled and any you're intentionally NOT handling (and why, if that's a reasonable scope call), and ask if they'd like you to test any additional cases or discuss further optimizations. This closing summary is a small thing that reliably reads as "this person has done this before" — it's the same instinct as a good PR description: don't make the reviewer reconstruct what you did and why.
