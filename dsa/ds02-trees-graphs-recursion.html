DSA Bonus 2:
Trees, Graphs & Recursion

Once arrays/strings/hashing are solid, the next interview gate is almost always trees and graphs — and both are built on top of one foundational skill: recursion. This chapter builds that foundation from scratch (base case, recursive case, and the real cost of NOT memoizing), then uses it to implement real binary tree traversals, real graph BFS/DFS, and real backtracking — all runnable, all with printed output so you can see exactly what order things happen in.

🕐 ~60 min read 💻 5 exercises 🏗️ Capstone Project 🧪 5-question quiz 💼 12 Interview Prep Q&A
2.1 — Recursion: Base Case + Recursive Case

A recursive function calls itself on a SMALLER version of the same problem, until it hits a base case simple enough to answer directly without recursing further. Every recursive function needs both pieces: skip the base case and it recurses forever (crashing with a RecursionError); get the "smaller version" wrong and it never actually shrinks toward the base case.

factorial: the canonical base case + recursive case
def factorial(n): if n <= 1: # base case: smallest version, answered directly return 1 return n * factorial(n - 1) # recursive case: smaller problem + combine print(f'factorial(5): {factorial(5)}') print(f'factorial(10): {factorial(10)}')

Naive recursive Fibonacci is the classic warning story: fib(n) = fib(n-1) + fib(n-2) looks clean, but it recomputes the SAME smaller sub-problems over and over — fib(28) gets computed dozens of times while computing fib(30). Memoization (caching each result the first time it's computed) fixes this by making every sub-problem get solved exactly once.

Naive vs memoized fibonacci — a real, timed comparison
import time def fib_naive(n): if n <= 1: return n return fib_naive(n - 1) + fib_naive(n - 2) # recomputes everything, every time memo = {} def fib_memo(n): if n <= 1: return n if n in memo: return memo[n] # already solved -> O(1) lookup, no recursion memo[n] = fib_memo(n - 1) + fib_memo(n - 2) return memo[n] start = time.perf_counter() naive_result = fib_naive(30) t_naive = time.perf_counter() - start start = time.perf_counter() memo_result = fib_memo(30) t_memo = time.perf_counter() - start print(f'fib_naive(30) = {naive_result}, took {t_naive:.4f}s') print(f'fib_memo(30) = {memo_result}, took {t_memo:.6f}s') print(f'Naive was slower: {t_naive > t_memo}')
⭐
Naive fib(30) makes about 2.7 million recursive calls; memoized fib(30) makes 31. Naive Fibonacci is O(2ⁿ) — the call count roughly doubles for every +1 to n. Memoized Fibonacci is O(n) — each of the n sub-problems is solved exactly once. This is the single most important lesson in this chapter, and it's the entire idea behind Dynamic Programming (DSA 3).
2.2 — Binary Trees: Traversal Order Is the Whole Game

A binary tree node holds a value plus (at most) a left and right child. The three DEPTH-FIRST traversal orders — inorder, preorder, postorder — differ only in WHEN you visit the current node relative to its children. BFS (level-order) is different in kind: it visits the tree level by level using a queue, not recursion.

Node class + all four traversal orders on the same tree
from collections import deque class Node: def __init__(self, value, left=None, right=None): self.value = value self.left = left self.right = right # 50 # / \ # 30 70 # / \ / \ # 20 40 60 80 root = Node(50, Node(30, Node(20), Node(40)), Node(70, Node(60), Node(80))) def inorder(node, out): # left, SELF, right if node is None: return inorder(node.left, out); out.append(node.value); inorder(node.right, out) def preorder(node, out): # SELF, left, right if node is None: return out.append(node.value); preorder(node.left, out); preorder(node.right, out) def postorder(node, out): # left, right, SELF if node is None: return postorder(node.left, out); postorder(node.right, out); out.append(node.value) def bfs_level_order(root): if root is None: return [] result = [] q = deque([root]) while q: node = q.popleft() result.append(node.value) if node.left: q.append(node.left) if node.right: q.append(node.right) return result in_order, pre_order, post_order = [], [], [] inorder(root, in_order); preorder(root, pre_order); postorder(root, post_order) print(f'Inorder: {in_order}') print(f'Preorder: {pre_order}') print(f'Postorder: {post_order}') print(f'Level-order (BFS): {bfs_level_order(root)}')
🧠
Inorder on a Binary SEARCH Tree always produces values in sorted order — that's why it's called "inorder." Preorder is the natural order for CLONING a tree (write self before children, so a fresh copy can rebuild top-down). Postorder is the natural order for DELETING a tree (delete children before the parent that references them).
2.3 — Graphs: Adjacency Lists, and DFS vs BFS on the SAME Graph

A tree is a special graph (connected, no cycles, one path between any two nodes). General graphs can have cycles and multiple paths, and are commonly represented as an adjacency list — a dict mapping each node to a list of its neighbors. DFS (depth-first, usually recursive) dives as deep as possible down one path before backtracking. BFS (breadth-first, iterative with a queue) explores everything one step away before anything two steps away. Run both on the exact same graph and watch the visit order genuinely differ.

DFS (recursive) vs BFS (iterative, queue) on the same graph
from collections import deque graph = { 'A': ['B', 'C'], 'B': ['A', 'D', 'E'], 'C': ['A', 'F'], 'D': ['B'], 'E': ['B', 'F'], 'F': ['C', 'E'], } def dfs_recursive(graph, start, visited=None, order=None): if visited is None: visited = set(); order = [] visited.add(start) order.append(start) for neighbor in graph[start]: if neighbor not in visited: dfs_recursive(graph, neighbor, visited, order) # dive deep first return order def bfs_iterative(graph, start): visited = {start} order = [] q = deque([start]) while q: node = q.popleft() # FIFO -> visit in the order discovered order.append(node) for neighbor in graph[node]: if neighbor not in visited: visited.add(neighbor) q.append(neighbor) return order print(f'DFS from A: {dfs_recursive(graph, "A")}') print(f'BFS from A: {bfs_iterative(graph, "A")}')
⚠️
DFS from A: ['A', 'B', 'D', 'E', 'F', 'C'] — BFS from A: ['A', 'B', 'C', 'D', 'E', 'F']. Same graph, same start node, genuinely different order — DFS commits to a full path down through B before ever backing up to check C; BFS visits every direct neighbor of A (B, C) before going any deeper. This is not a minor implementation detail — it changes what each algorithm is good for (2.3 continues in Interview Prep).
2.4 — Backtracking: Recursion That Explores, Then Undoes

Backtracking is recursion with a twist: try a choice, recurse deeper as if it were permanent, then UNDO it before trying the next choice — so the same shared data structure gets reused across every branch of the search instead of copying it each time. It's the standard tool for "generate all X" problems, where X grows combinatorially.

Backtracking: all subsets, all permutations
def all_subsets(elements): result = [] def backtrack(start, current): result.append(current[:]) # snapshot -> every prefix IS a valid subset for i in range(start, len(elements)): current.append(elements[i]) # choose backtrack(i + 1, current) # explore current.pop() # un-choose (the "backtrack") backtrack(0, []) return result def all_permutations(s): result = [] def backtrack(current, remaining): if not remaining: result.append(current); return for i in range(len(remaining)): backtrack(current + remaining[i], remaining[:i] + remaining[i+1:]) backtrack('', s) return result subsets = all_subsets(['a', 'b', 'c']) perms = all_permutations('abc') print(f'Subsets: {subsets}') print(f'Number of subsets: {len(subsets)}') print(f'Permutations: {perms}') print(f'Number of permutations: {len(perms)}')
⭐
3 elements produce 2³ = 8 subsets and 3! = 6 permutations — this growth is exactly why backtracking problems are usually only tractable for small inputs. An interviewer asking a backtracking question is rarely testing whether you know the growth is exponential (you should say so up front) — they're testing whether you can correctly generate the full search space without duplicates or omissions.
DSA 2 Summary
  • Recursion needs a base case (stops it) and a recursive case that shrinks toward that base case — skipping either breaks it.
  • Memoization turns exponential re-computation (naive Fibonacci) into linear work by caching each sub-problem's answer the first time it's solved.
  • Tree traversals (inorder/preorder/postorder) differ only in when you visit the current node relative to its children; BFS level-order is a different algorithm entirely, using a queue instead of recursion.
  • DFS (recursive, dives deep) and BFS (iterative, queue, expands outward) can produce genuinely different visit orders on the same graph — the choice between them is a real design decision, not a style preference.
  • Backtracking is recursion that tries a choice, explores it fully, then undoes it before trying the next — the standard pattern for "generate all X" problems.

💼 Trees, Graphs & Recursion — Interview Prep

12 questions on the angles interviewers actually probe for this topic: BFS vs DFS judgment calls, recursion's real costs, and tree/graph terminology precision. Click a question to reveal the answer. Not graded — practice saying these out loud, not just recognizing them.

Concretely, when do you pick BFS over DFS, and vice versa?▶
Pick BFS when you need the shortest path / fewest steps in an unweighted graph (this chapter's capstone), or when "nearest things first" matters (e.g. friends-of-friends up to distance 2). Pick DFS when you need to explore a full path before concluding anything about it (maze-solving, topological sort, detecting cycles via parent-tracking), when memory is tight (DFS's stack depth is bounded by the tree/graph's depth, while BFS's queue can hold an entire "level" which may be much wider), or when recursion naturally matches the problem's structure (tree traversals).
Why does Python (and most languages) impose a recursion depth limit, and what's the practical risk?▶
Every recursive call pushes a new frame onto the call stack, and the call stack has a fixed, finite size (Python's default limit is 1000 frames, adjustable via sys.setrecursionlimit, but the OS-level stack itself is still finite). A recursive function without a properly-shrinking base case, or one applied to a very deep structure (e.g. a naive recursive sum over a 100,000-element list), will hit RecursionError: maximum recursion depth exceeded — a real production risk if input size isn't bounded, which is exactly why many "recursive" algorithms are reimplemented iteratively (with an explicit stack/queue) for production use on unbounded input.
Does Python optimize tail-recursive calls the way some other languages do?▶
No. Some languages (Scheme, some functional languages) perform "tail-call optimization," reusing the current stack frame when a recursive call is the very last operation, effectively turning tail recursion into a loop with no extra stack growth. CPython deliberately does NOT do this — every recursive call in Python always adds a new stack frame, no matter how it's structured — so writing "tail-recursive-style" Python doesn't save you from recursion depth limits the way it would in those other languages.
Precisely, what makes a tree a special case of a graph, and where does the terminology diverge?▶
A tree is a connected, acyclic graph with exactly one path between any two nodes (and, for a rooted tree, one designated root with a clear parent/child direction away from it). Terminology diverges from there: trees use "root," "parent," "child," "leaf," "depth," "height" — vocabulary that assumes a single entry point and no cycles. General graphs use "node/vertex" and "edge" without assuming any of that, and can be directed or undirected, weighted or unweighted, connected or disconnected, with or without cycles — a tree algorithm (like the traversals in 2.2) can usually be generalized to graphs only after explicitly handling cycles (tracking visited nodes), which trees don't need since they have none.
What's the time and space complexity of BFS/DFS on a graph with V vertices and E edges?▶
Both are O(V + E) time with an adjacency list: every vertex is visited once (O(V)), and every edge is examined once from each direction it's stored (O(E)). Space is O(V) for the visited set/dict, plus O(V) worst case for the queue (BFS) or the recursion stack (DFS) — e.g. BFS's queue can hold an entire graph "layer," which in a star-shaped graph could be nearly all V nodes at once.
Why does adjacency-LIST representation usually beat adjacency-MATRIX for sparse graphs, and when is the reverse true?▶
An adjacency matrix is O(V²) space regardless of how many edges actually exist, while an adjacency list is O(V + E) space — for a SPARSE graph (E much smaller than V²), the list representation is dramatically more memory-efficient, and iterating a node's actual neighbors is O(degree) instead of O(V) (scanning a full matrix row, most of which is zeros). The matrix wins when the graph is DENSE (E close to V²) and you need O(1) "are these two nodes directly connected?" lookups frequently — the list needs O(degree) to answer that same question.
How do you detect a cycle in an UNDIRECTED graph, and why can't you just check "have I visited this node before"?▶
In an undirected graph, every edge is stored in BOTH directions (A's list includes B, and B's list includes A), so a plain DFS will immediately "revisit" the node it just came FROM — that's not a cycle, it's just walking back over the same single edge. The fix (used in this chapter's capstone) is tracking each node's PARENT in the DFS call and only flagging a cycle when you reach an already-visited node that ISN'T your immediate parent — that's a genuine second path back to an ancestor, which is what a cycle actually is.
How does cycle detection differ for a DIRECTED graph?▶
Parent-tracking isn't enough, because in a directed graph reaching an already-visited node via a DIFFERENT path (not back the way you came) doesn't necessarily mean a cycle — it might just be a shared descendant reached two different ways (which is fine in a DAG). The correct approach tracks nodes in the CURRENT recursion path specifically (often called the "recursion stack" or "gray" set, distinct from the permanent "visited"/"black" set) — a cycle exists only if you reach a node that's still in that active path, not merely one you've visited at some point in the past.
Why is memoized Fibonacci O(n) time, precisely — walk through the accounting.▶
There are only n distinct sub-problems: fib(0), fib(1), ..., fib(n). The memo dict guarantees each one is actually COMPUTED (the expensive branch, not the cache-hit branch) exactly once — every other call to fib_memo(k) for an already-solved k is an O(1) dict lookup, not a fresh recursive branch. So total work is n computations × O(1) work each (one addition, one dict write) = O(n), plus O(n) for the dict lookups along the way — no exponential blowup because no sub-problem is ever solved twice.
When is an iterative solution preferable to a recursive one, even when both are correct?▶
When input size isn't bounded and could be large enough to risk hitting the recursion limit (processing a very long linked list recursively, for instance); when the overhead of function calls matters for performance-critical code (each call has real cost beyond just "the work inside it"); or when the recursive structure doesn't naturally map to the problem (forcing recursion onto an inherently sequential process just adds complexity). Recursion is usually preferable when the problem is naturally self-similar (trees, divide-and-conquer, backtracking) and the depth is bounded by something small like tree height rather than raw input size.
How would you convert a recursive DFS into an iterative one, and why would you want to?▶
Replace the call stack with an explicit stack (a list, using append/pop from the end) that you manage yourself: push the start node, then loop — pop a node, process it, push its unvisited neighbors — which is structurally almost identical to the BFS code in this chapter except using a stack (LIFO) instead of a queue (FIFO). You'd want to do this to sidestep recursion depth limits on very deep/large graphs, or when you need finer control over the traversal (e.g. pausing and resuming it) that a plain recursive call doesn't easily allow.
In a coding interview, how do you talk through choosing between BFS, DFS, and backtracking when you're not sure which fits?▶
Ask yourself out loud: "Do I need the SHORTEST path, or just A path?" (shortest + unweighted → BFS). "Do I need to explore every possible complete solution, generating all of them?" (→ backtracking, and say the complexity is likely exponential up front). "Am I processing a naturally nested/recursive structure where I just need to visit everything once?" (→ DFS, often the simplest to code). Narrating this decision process is itself a strong signal — it shows you have a repeatable framework instead of guessing, which matters as much as landing on the right answer.
