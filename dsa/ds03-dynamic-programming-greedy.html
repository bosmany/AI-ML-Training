DSA Bonus 3:
Dynamic Programming & Greedy Algorithms

DP is the topic candidates fear most — and it's really just DSA 2's memoization idea, generalized and made systematic. This chapter builds that bridge explicitly: from the memoized Fibonacci you already understand, to a proper 1D DP table, to a proper 2D DP table, ending with greedy algorithms and — critically — a REAL, runnable example of a greedy approach visibly failing where DP succeeds, so the distinction is demonstrated, not just asserted.

🕐 ~65 min read 💻 5 exercises 🏗️ Capstone Project 🧪 5-question quiz 💼 12 Interview Prep Q&A
3.1 — What DP Actually Is: Optimal Substructure + Overlapping Subproblems

Dynamic Programming applies to problems with two properties: optimal substructure (the best solution to the whole problem is built from best solutions to smaller sub-problems — exactly like factorial(n) = n * factorial(n-1)), and overlapping subproblems (those smaller sub-problems repeat, so caching pays off — exactly like naive Fibonacci recomputing fib(28) dozens of times, from DSA 2). fib_memo from DSA 2 already WAS a dynamic programming solution — a top-down one (recursion + a cache). This chapter's new piece is bottom-up DP: build the answer iteratively from the smallest sub-problem up, storing every intermediate result in an explicit dp array — no recursion, no call stack risk, and it makes the "table filling in" visible.

Climbing stairs: 1D DP, bottom-up, dp array printed at every step
# How many distinct ways to climb n stairs, moving 1 or 2 steps at a time? def climbing_stairs(n): dp = [0] * (n + 1) dp[0] = 1 # 1 way to "climb" 0 stairs: do nothing if n >= 1: dp[1] = 1 # 1 way to climb 1 stair: a single 1-step for i in range(2, n + 1): dp[i] = dp[i - 1] + dp[i - 2] # last move was a 1-step, or a 2-step return dp n = 8 dp = climbing_stairs(n) for i in range(n + 1): print(f'dp[{i}] (ways to reach step {i}) = {dp[i]}') print(f'Total distinct ways to climb {n} stairs: {dp[n]}')
🧠
Notice this is EXACTLY the Fibonacci recurrence from DSA 2 (dp[i] = dp[i-1] + dp[i-2]) — climbing stairs and Fibonacci are the same underlying problem wearing a different costume. Recognizing "this is secretly Fibonacci" (or secretly some other DP shape you already know) is a real, learnable interview skill.
Coin change: minimum coins to make an amount, 1D DP
def coin_change_min(coins, amount): dp = [float('inf')] * (amount + 1) dp[0] = 0 # 0 coins needed to make amount 0 for a in range(1, amount + 1): for c in coins: if c <= a and dp[a - c] + 1 < dp[a]: dp[a] = dp[a - c] + 1 # best way to make 'a': use coin c, plus best way to make (a-c) return dp coins = [1, 5, 10, 25] amount = 63 dp2 = coin_change_min(coins, amount) print(f'Min coins to make {amount}: {dp2[amount]}') print(f'dp[60]: {dp2[60]}, dp[61]: {dp2[61]}, dp[62]: {dp2[62]}, dp[63]: {dp2[63]}')
3.2 — 2D DP: When One Index Isn't Enough

Some problems need TWO indices to describe a sub-problem — Longest Common Subsequence (LCS) is the classic example: "what's the longest sequence that appears in both s1[0:i] AND s2[0:j]?" needs both i and j, so the DP table is 2D. The recurrence: if the current characters match, extend the diagonal's answer by 1; if they don't, take the best of "drop this character from s1" or "drop this character from s2."

Longest Common Subsequence — full 2D dp table printed
def lcs(s1, s2): m, n = len(s1), len(s2) dp = [[0] * (n + 1) for _ in range(m + 1)] for i in range(1, m + 1): for j in range(1, n + 1): if s1[i - 1] == s2[j - 1]: dp[i][j] = dp[i - 1][j - 1] + 1 # chars match -> extend the diagonal else: dp[i][j] = max(dp[i - 1][j], dp[i][j - 1]) # best of dropping either char return dp, dp[m][n] s1, s2 = 'INTERVIEW', 'INTEGER' table, length = lcs(s1, s2) for row in table: print(row) print(f"LCS length of '{s1}' and '{s2}': {length}")
⭐
LCS's time and space complexity is O(m·n) — one cell per (i, j) pair, constant work per cell. This is the general shape of 2D DP: if your sub-problem needs two indices to describe, expect an O(m·n) table, not O(m+n). LCS itself powers real tools you've used — it's the core algorithm behind diff / "track changes" style comparisons.
3.3 — Greedy Algorithms: Fast, Simple, and Not Always Correct

A greedy algorithm makes the locally-best choice at each step and never reconsiders it — no table, no backtracking, just one pass. It's much simpler and faster than DP when it works — but it doesn't always find the truly optimal answer, and knowing WHICH problems it's safe for is the actual skill.

Activity selection: greedy (earliest finish time) IS optimal here
def activity_selection(activities): # activities: list of (start, end) — pick the MAX-COUNT non-overlapping set sorted_acts = sorted(activities, key=lambda a: a[1]) # sort by END time selected = [sorted_acts[0]] last_end = sorted_acts[0][1] for start, end in sorted_acts[1:]: if start >= last_end: # doesn't overlap the last one we picked selected.append((start, end)) last_end = end return selected activities = [(1,4), (3,5), (0,6), (5,7), (3,9), (5,9), (6,10), (8,11), (8,12), (2,14), (12,16)] chosen = activity_selection(activities) print(f'Activities chosen: {chosen}') print(f'Number chosen: {len(chosen)}')

Now the critical counterexample. 0/1 knapsack looks superficially similar to activity selection (pick items to maximize value under a constraint), but greedy — even the "smartest" greedy heuristic, picking items by value-per-weight ratio — provably does NOT always find the optimal answer. Below, the DP solution and the greedy solution are run on the EXACT same data, and they disagree.

0/1 Knapsack: greedy-by-ratio visibly FAILS where DP succeeds
def knapsack_01_dp(weights, values, capacity): n = len(weights) dp = [[0] * (capacity + 1) for _ in range(n + 1)] for i in range(1, n + 1): for w in range(capacity + 1): dp[i][w] = dp[i - 1][w] # option 1: don't take item i if weights[i - 1] <= w: dp[i][w] = max(dp[i][w], dp[i - 1][w - weights[i - 1]] + values[i - 1]) # option 2: take it return dp[n][capacity] def knapsack_greedy_by_ratio(weights, values, capacity): items = sorted(zip(weights, values), key=lambda wv: wv[1] / wv[0], reverse=True) total_value = 0 remaining = capacity for w, v in items: if w <= remaining: # greedily take the best ratio that still fits total_value += v remaining -= w return total_value weights = [10, 20, 30] values = [60, 100, 120] capacity = 50 dp_best = knapsack_01_dp(weights, values, capacity) greedy_best = knapsack_greedy_by_ratio(weights, values, capacity) print(f'DP optimal value: {dp_best}') print(f'Greedy-by-ratio value: {greedy_best}') print(f'Greedy found the true optimum: {dp_best == greedy_best}')
🚫
DP optimal value: 220 — Greedy-by-ratio value: 160. Greedy picks items 1 and 2 by ratio (6.0 and 5.0 value/weight) for 160, leaving item 3 unused since it no longer fits. But items 2+3 together give 100+120=220 using the exact same 50 capacity. Greedy can't "look ahead" or reconsider — once it commits to item 1, it's stuck with whatever fits around that choice, even though skipping item 1 entirely was the better call. DP considers every combination systematically and never makes this mistake.
⚠️
Why does greedy work fine on activity selection but fail on knapsack? Activity selection has a special property (an exchange argument proves the earliest-finishing activity is always safe to include in SOME optimal solution) that 0/1 knapsack simply doesn't have — taking the best ratio item first can lock out a better combination later, with no way to undo it. There's no shortcut for knowing this in advance besides recognizing the problem type — which is exactly what repetition across problems builds.
DSA 3 Summary
  • DP applies when a problem has optimal substructure (best answer built from best sub-answers) AND overlapping subproblems (those sub-answers repeat) — memoized Fibonacci from DSA 2 already was DP, just top-down.
  • Bottom-up (tabulation) builds the answer iteratively into an explicit dp array/table — no recursion, no stack-depth risk, and the table fill is directly observable.
  • 1D DP (climbing stairs, coin change) needs one index per sub-problem; 2D DP (LCS) needs two, giving an O(m·n) table.
  • Greedy is faster and simpler than DP but only correct for problems with a provable "safe choice" property (activity selection) — it can and does fail on problems that look similar but lack that property (0/1 knapsack).
  • The real interview skill isn't memorizing solutions — it's recognizing WHICH shape a new problem has: is there a safe greedy choice, or do I need to consider combinations systematically with DP?

🏗️ Capstone: Maximize Value of a Fixed Interview Day

An interview day has fixed candidate slots — each with a start time, end time, and a "value" (how promising the candidate is). Two slots that overlap can't both be used (only one interviewer, one room). This is weighted interval scheduling: unlike DSA 3's plain activity selection (which just maximizes COUNT), here different candidates are worth different amounts, so greedy-by-earliest-finish is no longer guaranteed optimal — this needs real DP.

weighted interval scheduling 1D DP with reconstruction optimal substructure

📋 What to Build

  • Sort slots by end time (same first move as greedy activity selection)
  • dp[i] = best total value achievable using the first i sorted slots; for slot i, choose max(include it + dp[last compatible slot], exclude it) = dp[i-1]
  • "Last compatible slot" for slot i = the latest earlier slot whose end time is <= slot i's start time
  • Reconstruct WHICH slots were chosen by walking your include/exclude decisions backward
  • Report: the full dp table, the maximum total value, which candidates were scheduled, how many interviews that is, and how much total value was left unscheduled

💼 Dynamic Programming & Greedy — Interview Prep

12 questions on the angles interviewers actually probe: recognizing DP problems, complexity trade-offs, and precisely WHY greedy fails where it fails. Click a question to reveal the answer. Not graded — practice saying these out loud, not just recognizing them.

How do you recognize, from a fresh problem statement, that it's a DP problem?▶
Look for language like "maximum/minimum," "count the number of ways," or "is it possible to" combined with a sense that the answer for a larger input depends on answers to smaller versions of the same input (optimal substructure) — and, critically, that a brute-force recursive solution would re-solve the SAME smaller sub-problem multiple times (overlapping subproblems). If a brute-force recursive attempt has no repeated sub-problems, DP won't help — that's a strong signal you're looking at plain recursion or backtracking instead.
What's the standard process for going from "this is a DP problem" to a working solution?▶
1) Define the sub-problem precisely in words (e.g. "dp[i] = the minimum coins needed to make amount i"). 2) Find the recurrence — how does dp[i] relate to smaller dp values? 3) Identify the base case(s). 4) Decide the fill order (usually smallest sub-problem to largest). 5) Decide top-down (recursion + cache) or bottom-up (explicit table) — bottom-up is usually preferred in interviews since it avoids recursion-depth concerns and makes the table inspectable.
Time/space complexity: memoization (top-down) vs tabulation (bottom-up) — are they actually different?▶
Asymptotically, usually the same — both solve each distinct sub-problem once. The practical differences: memoization can be FASTER in wall-clock time on inputs where many sub-problems are never actually needed (it only computes what recursion actually visits), while tabulation typically computes every sub-problem in the table regardless of whether the final answer needs all of them. Memoization also adds function-call overhead and risks recursion depth limits (DSA 2) that tabulation doesn't have.
How would you reduce the space complexity of a 1D DP solution like climbing stairs or coin change?▶
If dp[i] only ever depends on a FIXED small number of previous entries (climbing stairs only needs dp[i-1] and dp[i-2]), you don't need the whole array — just keep the last few values in plain variables and slide them forward each iteration, dropping space from O(n) to O(1). This "rolling variables" trick doesn't work for every DP (coin change genuinely needs the full array, since any earlier coin combination might be reused), so check the recurrence's actual dependencies before assuming you can shrink it.
Is there ever a version of the knapsack problem where greedy IS provably optimal?▶
Yes — the FRACTIONAL knapsack problem (you're allowed to take a FRACTION of an item, not just all-or-nothing) is solved optimally by the exact same greedy-by-ratio approach that fails on 0/1 knapsack. The reason: with fractions allowed, you can always "top off" the last item to exactly fill remaining capacity, so there's no scenario where a slightly-worse-ratio combination beats the greedy choice — the all-or-nothing constraint is specifically what breaks greedy's guarantee in the 0/1 version.
What's the "exchange argument" that proves greedy is optimal for activity selection?▶
Take any optimal solution that doesn't include the earliest-finishing activity. You can always SWAP its first-chosen activity for the earliest-finishing one without reducing the count — the earliest-finishing activity ends no later, so it can't conflict with anything the rest of that optimal solution schedules afterward. Since the swap never makes things worse, an optimal solution containing the earliest-finishing activity always exists, which justifies greedily picking it every time. This style of argument (show a greedy choice can always be swapped in for free) is the standard way of PROVING a greedy algorithm correct, rather than just trusting it looks right.
How does this chapter's capstone (weighted interval scheduling) differ from plain activity selection, and why does that difference force DP?▶
Plain activity selection maximizes COUNT, where every activity is worth the same "1 unit" — greedy-by-earliest-finish is provably safe there. Weighted interval scheduling maximizes total VALUE, where a high-value activity might be worth sacrificing several lower-value ones for — greedy-by-earliest-finish has no mechanism to account for value at all, so it can (and does, on real inputs) pick a worse combination. The value dimension is exactly what breaks the exchange argument that made plain activity selection greedy-safe.
If you're not sure whether greedy is safe for a new problem, what should you do in an interview?▶
Say so explicitly, and try to construct a small counterexample by hand (2-4 items) before committing to greedy — this chapter's knapsack counterexample is exactly that kind of small, hand-constructable case. If you can't quickly find a counterexample AND can articulate why the greedy choice is always safe (an exchange-argument-style justification), greedy is probably fine. If you're unsure either way, defaulting to DP is the safer answer to actually ship correct — you can always mention "greedy might work here and would be faster, but I'd want to verify the exchange property first."
What's a real-world application of Longest Common Subsequence beyond interview questions?▶
Version control diff tools (git diff, "track changes" in documents) use LCS-family algorithms to find the minimal set of insertions/deletions that transform one version of a file into another — the LCS is literally the "unchanged" content, and everything not in it is what gets shown as added/removed. Bioinformatics also uses LCS-family algorithms (sequence alignment) to compare DNA/protein sequences.
How would you extend the coin_change_min DP to also RECONSTRUCT which coins were used, not just the count?▶
Track a second array alongside dp — e.g. choice[a] = c, the coin that achieved dp[a]'s best value — updated at the same time dp[a] is updated. To reconstruct, start at amount, repeatedly look up choice[amount], append that coin, and subtract it from the remaining amount, until you reach 0. This "parallel choice array" pattern is exactly what this chapter's capstone does to reconstruct which candidates were scheduled, not just the maximum value.
Why is it worth explicitly stating the recurrence relation out loud before coding, in an interview?▶
Because DP bugs are almost always recurrence bugs, not syntax bugs — an off-by-one in the recurrence, a missing base case, or a wrong "which sub-problems does this depend on" produces code that runs without error but returns the wrong number. Stating "dp[i][j] means X, and it equals Y based on dp[i-1][j] and dp[i][j-1]" out loud BEFORE writing code lets the interviewer catch a wrong recurrence early, rather than debugging silently-wrong output later — and it's the single strongest signal that you actually understand the problem rather than pattern-matching a memorized solution.
Why does DP guarantee correctness where greedy might not, even though DP considers "the same" choices greedy does?▶
Greedy commits to one choice per step and never revisits it. DP effectively considers BOTH "include this" and "exclude this" at every step (that's exactly the dp[i][w] = max(dp[i-1][w], ...) pattern in knapsack) and lets the table capture the best outcome under every possible prior state — so it never accidentally locks itself out of a better combination the way greedy can. The cost of that guarantee is doing genuinely more work (considering combinations, not just one path) — which is precisely the time/space trade-off DP makes for correctness.
