DSA Bonus 1:
Arrays, Strings & Hashing

Before any interviewer looks at your ML knowledge, most companies run you through a coding-interview gate — and that gate is almost always built from a small set of recurring patterns, not exotic algorithms. This chapter covers the three patterns that show up constantly in "easy" and "medium" interview questions: two pointers, sliding window, and hash maps — plus enough informal Big-O intuition to reason about which one to reach for. Every idea here is demonstrated with real, runnable code, not just described.

🕐 ~55 min read 💻 5 exercises 🏗️ Capstone Project 🧪 5-question quiz 💼 12 Interview Prep Q&A
1.1 — Big-O Intuition: Why Growth Rate Beats Raw Speed

Big-O describes how an algorithm's work grows as input size n grows — NOT how fast it runs on one specific input. A cleverly-optimized O(n²) algorithm can beat a naive O(n) one on a small input, but as n gets large, the ALGORITHM with the better growth rate always wins eventually, no matter how much "constant factor" speed the other one has. This is the single most important intuition for interviews: when an interviewer asks "can you do better than that?", they are almost always asking about growth rate, not micro-optimization.

You do this......that many times, roughlyComplexity
Look up one value by index1 (constant)O(1)
Scan every element oncenO(n)
Scan every PAIR of elements (nested loop)n × nO(n²)
O(n²) vs O(n): a real, timed comparison — same task, same data
import time def has_duplicate_On2(arr): # nested loop: checks every PAIR -> O(n^2) n = len(arr) for i in range(n): for j in range(i + 1, n): if arr[i] == arr[j]: return True return False def has_duplicate_On(arr): # one pass with a hash set -> O(n) seen = set() for x in arr: if x in seen: return True seen.add(x) return False # worst case for both: no duplicate exists, so every check runs to the end data = list(range(3000)) start = time.perf_counter() has_duplicate_On2(data) t_n2 = time.perf_counter() - start start = time.perf_counter() has_duplicate_On(data) t_n = time.perf_counter() - start print(f'O(n^2) version: {t_n2:.4f}s') print(f'O(n) version: {t_n:.4f}s') print(f'O(n^2) was slower: {t_n2 > t_n}')
⭐
3,000 elements means the O(n²) version does roughly 4.5 million comparisons, while the O(n) version does 3,000. Double the input to 6,000 and the O(n) version does 2x the work — but the O(n²) version does roughly 4x the work. That's the growth-rate gap interviewers are testing for when they ask about complexity.
1.2 — Two Pointers: One Pass, Two Positions

The two-pointer pattern uses two index variables that move through a sequence — often from opposite ends inward — instead of nested loops. It works whenever moving one pointer lets you safely rule out a whole range of possibilities at once, which is exactly what a SORTED array or a symmetric check (like a palindrome) gives you.

Two pointers: reverse, palindrome check, two-sum on sorted array
def reverse_string(s): chars = list(s) left, right = 0, len(chars) - 1 while left < right: chars[left], chars[right] = chars[right], chars[left] left += 1 right -= 1 return ''.join(chars) def is_palindrome(s): # ignore case and anything that isn't a letter/digit cleaned = [c.lower() for c in s if c.isalnum()] left, right = 0, len(cleaned) - 1 while left < right: if cleaned[left] != cleaned[right]: return False left += 1 right -= 1 return True def two_sum_sorted(arr, target): # arr MUST already be sorted for this logic to be correct left, right = 0, len(arr) - 1 while left < right: s = arr[left] + arr[right] if s == target: return (left, right) elif s < target: left += 1 # sum too small -> need a bigger left value else: right -= 1 # sum too big -> need a smaller right value return None print(f"reverse_string('hello'): {reverse_string('hello')}") print(f"is_palindrome('A man, a plan, a canal: Panama'): {is_palindrome('A man, a plan, a canal: Panama')}") print(f"is_palindrome('race a car'): {is_palindrome('race a car')}") print(f"two_sum_sorted([2, 7, 11, 15], 9): {two_sum_sorted([2, 7, 11, 15], 9)}")
⚠️
The #1 off-by-one trap: the loop condition must be left < right, not left <= right. With <=, a single middle element in an odd-length sequence gets compared against itself, and (worse) the two pointers can cross and start reading from the wrong end. Trace it by hand on a 3-element array before trusting it.
1.3 — Sliding Window: Tracking a Moving Range Efficiently

A sliding window is a contiguous range [left, right] that expands and shrinks as you scan once through the data — instead of re-scanning a fresh sub-range from scratch for every possible starting point (which is what makes the naive version O(n²) or worse). Reach for this pattern whenever the question is about a contiguous subarray or substring satisfying some condition.

Sliding window: longest unique substring, max sum of a fixed-size window
def longest_unique_substring(s): seen = {} # char -> most recent index seen at left = 0 best_len = 0 best_sub = '' for right, ch in enumerate(s): if ch in seen and seen[ch] >= left: left = seen[ch] + 1 # jump left past the earlier duplicate seen[ch] = right if right - left + 1 > best_len: best_len = right - left + 1 best_sub = s[left:right + 1] return best_len, best_sub def max_sum_subarray_k(arr, k): # fixed-size window of length k -> slide it, don't re-sum from scratch window_sum = sum(arr[:k]) best = window_sum for i in range(k, len(arr)): window_sum += arr[i] - arr[i - k] # add new right edge, drop old left edge best = max(best, window_sum) return best length, sub = longest_unique_substring('abcabcbb') print(f"longest_unique_substring('abcabcbb'): length={length}, substring='{sub}'") print(f"max_sum_subarray_k([2, 1, 5, 1, 3, 2], 3): {max_sum_subarray_k([2, 1, 5, 1, 3, 2], 3)}")
⭐
Notice max_sum_subarray_k never re-sums the whole window. It updates the running sum by one addition and one subtraction per step — that's what turns an O(n·k) naive re-sum-every-window approach into O(n). "Slide, don't re-scan" is the core idea behind every sliding-window solution.
1.4 — Hash Maps: Trading Memory for O(1) Lookup

A Python dict (hash map) gives average-case O(1) lookup, insert, and membership-check — dramatically faster than the O(n) scan a list requires to answer "have I seen this before?" The cost is memory: you're storing an extra copy of everything you've seen so far. This time-for-space trade is one of the most common patterns in interview solutions, and is exactly why an O(n²) nested-loop solution to a problem can usually be rewritten as an O(n) single-pass solution using a hash map.

Hash maps: unsorted two-sum, first non-repeating char, group anagrams
def two_sum_unsorted(arr, target): # single pass: for each x, check if target-x was ALREADY seen seen = {} # value -> index for i, x in enumerate(arr): complement = target - x if complement in seen: return (seen[complement], i) seen[x] = i return None def first_non_repeating(s): counts = {} for ch in s: counts[ch] = counts.get(ch, 0) + 1 for ch in s: # second pass preserves ORIGINAL order if counts[ch] == 1: return ch return None def group_anagrams(words): groups = {} # sorted-letters key -> list of original words for w in words: key = ''.join(sorted(w)) # anagrams share this exact key groups.setdefault(key, []).append(w) return list(groups.values()) print(f"two_sum_unsorted([3, 2, 4], 6): {two_sum_unsorted([3, 2, 4], 6)}") print(f"first_non_repeating('swiss'): {first_non_repeating('swiss')}") print(f"group_anagrams(...): {group_anagrams(['eat','tea','tan','ate','nat','bat'])}")
🧠
Why does two_sum_unsorted check for the complement BEFORE adding the current value? Because a pair needs two DIFFERENT indices. If you added first and checked second, an element could pair with itself (e.g. target=6, x=3 would incorrectly "find" complement 3 using its own index). Checking first, inserting second, is what guarantees the two indices returned are always distinct.
DSA 1 Summary
  • Big-O describes growth rate, not raw speed — the algorithm with the better growth rate always wins as n gets large, regardless of small-input performance.
  • Two pointers turns many O(n²) nested-loop problems into O(n) by moving two positions through sorted or symmetric data — watch the < vs <= boundary condition.
  • Sliding window tracks a contiguous range in one pass by expanding/shrinking it incrementally instead of re-scanning from scratch.
  • Hash maps trade memory for O(1) average lookup, turning many O(n²) "have I seen this?" problems into O(n) single-pass solutions.
  • Recognizing WHICH pattern applies — sorted+symmetric → two pointers, contiguous range → sliding window, "have I seen this?" → hash map — is the actual interview skill, more than memorizing any one solution.

💼 Arrays, Strings & Hashing — Interview Prep

12 questions covering the complexity-analysis and pattern-recognition angles interviewers actually probe for this topic. Click a question to reveal the answer. Not graded — this is here so you can talk through your reasoning out loud, not just recognize the right answer on a multiple-choice quiz.

How do you determine the time complexity of a piece of code you're seeing for the first time?▶
Count the loops and how their bounds relate to the input size n. A single loop over n items is O(n). A loop nested inside another loop, both over roughly n items, is O(n²) — multiply, don't add, nested loop bounds. A loop whose range HALVES each iteration (like binary search) is O(log n). If a loop calls a function that itself does O(n) work per iteration, multiply that in too — an O(n) loop calling an O(n) operation each time is O(n²) overall, even though there's only one visible loop.
What's the difference between average-case and worst-case complexity for a hash map, and why does it matter?▶
A hash map's O(1) lookup is an AVERAGE case, assuming a good hash function spreads keys evenly across buckets. Worst case (many keys colliding into the same bucket) degrades to O(n) — Python's dict implementation handles this well in practice, but it's worth knowing the guarantee is probabilistic, not absolute, which is why interviewers usually accept "O(1) average / O(n) worst case" as the fully correct answer rather than just "O(1)".
How would you recognize that a problem calls for two pointers rather than a hash map?▶
Two strong signals: the input is already sorted (or you're allowed to sort it and order doesn't matter to the answer), and/or the problem has some symmetry (a palindrome check, "meet in the middle"). If the data ISN'T sorted and sorting it would lose needed information (like original indices), a hash map is usually the better fit since it doesn't require any ordering at all — that's exactly why two_sum_unsorted uses a dict instead of sorting-then-two-pointers.
Why not just sort the array first and then use two pointers for every "two-sum"-style problem?▶
Sorting costs O(n log n) up front, and — critically — it destroys the ORIGINAL indices unless you separately track them alongside each value before sorting. If the problem asks for the original positions of the two numbers (as most two-sum variants do), sorting adds both an O(n log n) cost AND extra bookkeeping complexity that a single-pass hash map avoids entirely at O(n). Sort-then-two-pointers is the right call when you don't need original order/indices, or you need the pointers technique for a reason beyond just finding one pair (e.g. finding ALL pairs, or a 3-sum problem).
What's the single most common off-by-one mistake in sliding-window code, and how do you catch it?▶
Miscounting the window's length as right - left instead of right - left + 1 (since both endpoints are inclusive), or forgetting to update the "best" tracker at the right moment (before vs. after moving a pointer). The fastest way to catch it: trace the algorithm by hand on the smallest possible non-trivial example (e.g. a 2-3 character string) and compare against what you'd compute manually — most off-by-one bugs are invisible on medium-sized test cases but immediately obvious on tiny ones.
Why does group_anagrams use ''.join(sorted(word)) as a dictionary key instead of the word itself?▶
Anagrams are different arrangements of the SAME letters — so sorting each word's letters produces an identical string for every anagram in the group (e.g. both 'eat' and 'tea' sort to 'aet'), which is exactly the property needed for a dict key: anagrams collide into the same bucket, non-anagrams don't. Using the raw word as the key would put every word in its own group, defeating the purpose.
What's the time complexity of the group_anagrams approach shown in this chapter, including the sort?▶
If there are n words with maximum length k, sorting each word costs O(k log k), and you do that for all n words, giving O(n · k log k) total — NOT simply O(n). This is a common interview follow-up: "what's the actual complexity, accounting for the sort inside the loop?" — a candidate who says "O(n)" without noticing the per-word sort cost is missing a layer.
How would you extend the two-sum pattern to a three-sum problem, and why does the approach change?▶
Sort the array first (O(n log n)), then for each element, run the classic two-pointer two-sum on the REMAINING array to find pairs that sum to target - current_element — giving O(n²) overall (O(n) outer loop × O(n) inner two-pointer scan) instead of the O(n³) brute force of checking every triple. This is a good example of why two pointers and hash maps aren't rivals to memorize separately — three-sum combines a hash-map-style "fix one, solve the rest" idea with the two-pointer inner loop.
Why does the palindrome checker filter with c.isalnum() before comparing characters?▶
Real-world palindrome questions ("Was it a car or a cat I saw?") include spaces, punctuation, and mixed case that aren't meaningful to the palindrome property itself — only the letters/digits and their order matter. Filtering to alphanumeric characters (and lowercasing) first normalizes the input so the two-pointer comparison only ever looks at the characters that actually matter, which is also a good habit generally: separate "cleaning the input" from "the actual algorithm" so each piece is simple to reason about.
If an interviewer asks you to solve a problem "without extra space," which pattern are they usually steering you toward?▶
Almost always two pointers (or in-place array manipulation), since it uses O(1) extra space — no auxiliary dict or set. This is a common interview trick: they'll first let you solve it with a hash map (which is a perfectly valid O(n) time / O(n) space answer), then ask "can you do it in O(1) space?" to see if you recognize a two-pointer reformulation exists, which usually costs you nothing in time complexity if the input can be sorted or is already ordered.
What's the time and space complexity of the sliding-window longest_unique_substring solution?▶
Time: O(n) — the right pointer visits each character once, and left only ever moves forward, so across the whole run each pointer does at most n total steps, giving O(n) not O(n²) despite the nested-looking logic. Space: O(min(n, alphabet size)) for the seen dict, since it can hold at most one entry per distinct character — for ASCII text that's a small constant, but it's worth stating precisely rather than just saying "O(1)".
How do you explain, out loud, WHY a nested-loop two-sum can be rewritten with a hash map to go from O(n²) to O(n)?▶
The nested-loop version re-scans the ENTIRE array for every single element to find its complement — that repeated scanning is exactly the wasted work. A hash map replaces "scan the array again" with "look up in a dict," which is O(1) instead of O(n) — so the same O(n) outer loop, combined with an O(1) inner lookup instead of an O(n) inner scan, drops the total from O(n) × O(n) = O(n²) down to O(n) × O(1) = O(n). Framing it as "what expensive repeated work can a dict replace with a cheap lookup" is the general lens for spotting this kind of optimization in ANY problem, not just two-sum.
