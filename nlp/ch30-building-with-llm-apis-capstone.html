Chapter 30:
Building with LLM APIs

This is the Module 6 capstone. Everything from Chapters 27-29 — vectorizing text, next-token prediction, sampling — comes together into the patterns that power almost every real LLM application today: turning text into comparable vectors (embeddings), grounding a model's answer in retrieved facts (RAG), and letting a model decide which function to call (tool-calling agents). You'll simulate all three with deterministic, hand-built stand-ins so every result is exactly checkable, while seeing exactly where a real hosted LLM API call would go in production.

30.1 — Calling a Real LLM API

Every hosted LLM (Claude, GPT, and others) exposes an API: send a list of messages, get back generated text. Both major SDKs follow the same basic shape — a client object, a .create()-style call, a model name string, and a list of role/content messages. This needs a real API key and a real outbound network call, neither of which this sandboxed browser can provide — reference only, same reasoning as requests in Chapter 7.

import anthropic client = anthropic.Anthropic(api_key="sk-ant-...") # normally read from an env var, never hard-coded (Ch 7) response = client.messages.create( model="claude-sonnet-4-5", max_tokens=1024, messages=[ {"role": "user", "content": "Summarize the tradeoffs of RAG vs fine-tuning in 2 sentences."} ] ) print(response.content[0].text)

Both SDKs boil down to the same three ingredients: an authenticated client, a model name, and a list of role-tagged messages ("user", "assistant", sometimes "system"). Once you've called one hosted LLM API, reading another provider's docs is mostly just learning where the same three ingredients live in their SDK.
30.2 — Embeddings & Semantic Search (Runnable)

A real embedding model (e.g. text-embedding-3-small, or Claude/GPT's underlying representations) converts text into a dense numeric vector that captures meaning — similar meaning, similar vector. In production you'd call an embeddings API to get these vectors. Here, we reuse Chapter 27's hand-built TF-IDF vectors as a deterministic, fully-runnable stand-in — the numbers are cruder (word-overlap based, not learned semantics), but the RETRIEVAL MATH that follows — embed the query the same way, rank documents by cosine similarity — is identical to what you'd do with real embeddings.

Semantic search: embed a query, rank documents by similarity
import numpy as np, re STOPWORDS = {'a','an','the','and','with','is','in','on','of','to','are','she','he','it','for'} def tokenize(text): return re.findall(r"[a-z0-9]+", text.lower()) def remove_stop(tokens): return [t for t in tokens if t not in STOPWORDS] def tf(tokens, vocab): length = len(tokens) return np.array([tokens.count(w)/length for w in vocab]) def idf(vocab, doc_tokens): N = len(doc_tokens) return np.array([np.log(N / sum(1 for dt in doc_tokens if w in dt)) for w in vocab]) def cosine_sim(a, b): denom = np.linalg.norm(a) * np.linalg.norm(b) return 0.0 if denom == 0 else float(np.dot(a, b) / denom) def build_tfidf(corpus): doc_tokens = [remove_stop(tokenize(d)) for d in corpus] vocab = sorted(set(w for dt in doc_tokens for w in dt)) idf_vec = idf(vocab, doc_tokens) tfidf_vecs = [tf(dt, vocab) * idf_vec for dt in doc_tokens] return vocab, idf_vec, tfidf_vecs def embed_query(query, vocab, idf_vec): # stand-in for: real_client.embeddings.create(...) return tf(remove_stop(tokenize(query)), vocab) * idf_vec corpus = [ "Alice loves machine learning and python", "Bob loves deep learning and pytorch", "Carol works with python and data science", "Dave enjoys machine learning research", ] vocab, idf_vec, tfidf_vecs = build_tfidf(corpus) query = "python programming" q_vec = embed_query(query, vocab, idf_vec) sims = [cosine_sim(q_vec, dv) for dv in tfidf_vecs] best_idx = int(np.argmax(sims)) print(f'Sims: {[round(s,4) for s in sims]}') print(f'Best match: {corpus[best_idx]}')
⭐
"python programming" scores highest against the Alice doc, which mentions "python" (the Carol doc also mentions python, but Alice's doc also happens to share more corpus-rare terms). Swap embed_query()'s implementation for a real embeddings API call and this entire pipeline — build vectors, rank by cosine similarity — is unchanged. That's the point: the MATH doesn't care where the vectors came from.
30.3 — RAG: Retrieval-Augmented Generation (Runnable)

RAG grounds an LLM's answer in real, retrieved text instead of relying purely on what the model memorized during training: (1) embed the user's query, (2) retrieve the most relevant document(s) via similarity search (exactly 30.2's pipeline), (3) build a prompt that includes the retrieved text as context, (4) send that prompt to an LLM to generate the final answer. Here we simulate step 4 with a simple deterministic template function — standing in for exactly where a real client.messages.create(...) call would go.

Retrieve, build a prompt, simulate generation
def retrieve(query, corpus, vocab, idf_vec, tfidf_vecs, top_k=1): q_vec = embed_query(query, vocab, idf_vec) sims = [cosine_sim(q_vec, dv) for dv in tfidf_vecs] ranked = sorted(range(len(corpus)), key=lambda i: sims[i], reverse=True) return [(corpus[i], sims[i]) for i in ranked[:top_k]] def mock_generate(query, retrieved_docs): # STAND-IN for: client.messages.create(model=..., messages=[{"role":"user","content":prompt}]) # where prompt = f"Context: {context}\n\nQuestion: {query}\n\nAnswer using only the context above." context = " ".join(d for d, _ in retrieved_docs) return f"Based on the retrieved context ('{context}'), here's what I found relevant to '{query}': {retrieved_docs[0][0]}" rag_query = "Tell me about machine learning" retrieved = retrieve(rag_query, corpus, vocab, idf_vec, tfidf_vecs, top_k=1) answer = mock_generate(rag_query, retrieved) print(f'Retrieved: {retrieved[0][0]}') print(f'Similarity: {retrieved[0][1]:.4f}') print(answer)
🚫
The point of this section is the PIPELINE, not real text generation — mock_generate() is deliberately a dumb template, not an LLM. In production, step 4 is a real API call (30.1) whose prompt includes the retrieved context verbatim, which is exactly why RAG reduces hallucination: the model is answering FROM supplied text, not purely from memorized training data.
30.4 — Tool-Calling Agents (Runnable)

A tool-calling agent lets an LLM decide, from a user's natural-language request, WHICH of several available Python functions to call, then executes that function and returns the result. In production, the LLM itself reasons about which tool fits the request (using the tool definitions in its context). Here, we simulate that reasoning step with simple keyword-matching — the point is the AGENT LOOP pattern (decide → call → return), not real language understanding.

A tiny mock agent choosing between two tools
def get_weather(city): fixed = {"hubli": "28C, sunny", "pune": "24C, cloudy", "delhi": "31C, hazy", "mumbai": "30C, humid"} return fixed.get(city.lower(), "unknown city") def calculate(expression): allowed = set("0123456789+-*/(). ") if not set(expression) <= allowed: return "invalid expression" return eval(expression, {"__builtins__": {}}, {}) def agent(query): # STAND-IN for: a real LLM reasoning over tool definitions ("function calling") to pick one q = query.lower() if "weather" in q: for city in ["hubli","pune","delhi","mumbai"]: if city in q: return f"[tool: get_weather] {get_weather(city)}" return "[tool: get_weather] city not recognized" elif "calculate:" in q: expr = query.split(":", 1)[1].strip() return f"[tool: calculate] {expr} = {calculate(expr)}" else: return "[no tool matched]" print(agent("What is the weather in Hubli?")) print(agent("Calculate: 12 * 4 + 7"))
💡
Real tool-calling APIs (Anthropic's tool use, OpenAI's function calling) work the same shape — you describe available tools (name, parameters, description) in the request, the model's response tells you which tool it wants called and with what arguments, YOUR code executes the actual function, then you send the result back to the model to continue. The "decide which tool" step is the one thing our keyword-matching agent() is standing in for.
Chapter 30 Summary
  • Real LLM API calls (Anthropic, OpenAI) share the same shape: client, model name, list of role-tagged messages.
  • Embeddings + cosine similarity power semantic search — the retrieval math is identical whether vectors come from TF-IDF or a real embedding model.
  • RAG retrieves relevant context, then includes it in the prompt sent to an LLM — grounding answers in real text reduces hallucination.
  • Tool-calling agents let an LLM select and trigger real functions (weather lookups, calculators, database queries) based on a user's request.

💬 Interview Prep: NLP & LLM Engineering

12 commonly-asked interview questions covering tokenization, embeddings, attention, transformers, RAG, prompting, and evaluation. Click a question to reveal the answer. This section is supplementary reading — it doesn't count toward chapter completion or XP.

1What is the difference between stemming and lemmatization?▾
Stemming chops word endings using crude rules (e.g. "running" → "run", but also "universe" → "univers") without knowledge of grammar — fast but sometimes produces non-words. Lemmatization uses a vocabulary and morphological analysis to return the dictionary base form ("better" → "good", "running" → "run") — slower but linguistically correct. Lemmatization requires knowing the word's part of speech to work reliably.
2How does attention differ from RNN-based sequence models?▾
An RNN processes tokens one at a time, carrying a fixed-size hidden state forward — information from early tokens has to survive being repeatedly compressed through every later step, which is why RNNs struggle with long-range dependencies (vanishing gradients). Attention lets every token directly look at every other token in a single step, weighted by learned relevance — no information bottleneck from sequential compression, and it parallelizes across the whole sequence instead of processing token-by-token.
3What is RAG and why does it help reduce hallucination?▾
Retrieval-Augmented Generation retrieves relevant documents (via embedding similarity search) and inserts them into the prompt as context before the model generates an answer. Because the model can ground its answer in the supplied text instead of relying purely on facts memorized (and possibly misremembered) during training, RAG substantially reduces confidently-stated wrong answers — though it doesn't eliminate hallucination entirely, since the model can still misread or over-generalize from the retrieved context.
4What's the difference between fine-tuning and prompting/in-context learning?▾
Fine-tuning updates the model's actual weights using labeled examples — the change is permanent and baked into the model. Prompting/in-context learning (zero-shot or few-shot) supplies instructions or examples directly in the input text at inference time — no weights change, and the "adaptation" only lasts for that one request. Prompting is cheaper and faster to iterate on; fine-tuning is better when you need consistent behavior baked in, or a task too specialized for prompting alone to handle reliably.
5What is temperature in LLM sampling, and what happens at temperature=0?▾
Temperature scales the logits before softmax (dividing by T) to reshape how sharp or flat the resulting probability distribution is. Low temperature sharpens the distribution toward the highest-probability token (more deterministic, more repetitive); high temperature flattens it (more random/varied, more likely to produce incoherent output). At temperature=0, sampling becomes equivalent to always picking the single highest-probability token — deterministic greedy decoding, with no randomness at all.
6What's the difference between static word embeddings (like word2vec) and contextual embeddings (like BERT)?▾
A static embedding (word2vec, GloVe) assigns ONE fixed vector per word, regardless of context — "bank" gets the same vector in "river bank" and "savings bank". A contextual embedding (from a transformer like BERT) produces a DIFFERENT vector for the same word depending on the surrounding sentence, because attention lets the representation incorporate context. Contextual embeddings handle polysemy (words with multiple meanings) far better.
7Why do LLMs use subword tokenization (like BPE) instead of word-level tokenization?▾
Word-level tokenization needs a huge vocabulary to cover every possible word, and still fails completely on any word it's never seen (out-of-vocabulary). Subword tokenization (Byte-Pair Encoding, SentencePiece) keeps common words as single tokens but splits rare or unseen words into smaller, previously-seen pieces — so the model never truly hits an "unknown word", and the vocabulary size stays manageable (tens of thousands of entries instead of needing every possible word form).
8What is the vanishing gradient problem, and how did LSTMs address it before transformers existed?▾
In a plain RNN, gradients are multiplied repeatedly through each time step during backpropagation; over many steps these products can shrink toward zero, so early tokens' influence on the loss (and thus their gradient signal) effectively vanishes — the network can't learn long-range dependencies. LSTMs introduced gated memory cells (forget/input/output gates) with a more direct additive path for the cell state, letting gradients flow across many more time steps largely unchanged. Transformers later sidestepped the problem differently — attention connects any two positions directly, with no sequential chain of multiplications between them at all.
9What's the difference between encoder-only, decoder-only, and encoder-decoder transformer architectures?▾
Encoder-only (BERT) sees the whole input at once (bidirectional attention) and is suited to understanding tasks — classification, NER, embeddings. Decoder-only (GPT, Claude, most modern LLMs) attends only to earlier tokens (causal/masked attention) and generates text one token at a time — suited to open-ended generation. Encoder-decoder (T5, the original Transformer) encodes an input sequence fully, then a decoder generates an output sequence attending back to that encoding — suited to sequence-to-sequence tasks like translation or summarization.
10How would you evaluate the quality of an LLM's output?▾
Automated overlap metrics like BLEU/ROUGE compare generated text against a reference by n-gram overlap — fast and cheap, but they penalize valid paraphrases and don't capture factual correctness or coherence well. Human evaluation is the gold standard for subjective quality but slow and expensive to scale. LLM-as-judge (using a strong LLM to score another model's outputs against a rubric) has become a common middle ground — faster than humans, more semantically aware than BLEU/ROUGE, but introduces its own biases and needs validation against human judgment.
11What is a hallucination in the context of LLMs, and name two mitigation strategies.▾
A hallucination is when a model generates text that is fluent and confident-sounding but factually incorrect or unsupported — invented citations, wrong dates, functions that don't exist. Two common mitigations: (1) RAG — ground answers in retrieved, verifiable source text rather than relying purely on parametric memory; (2) lowering temperature and/or prompting the model to say "I don't know" when uncertain, reducing confident guessing. Neither eliminates hallucination completely — both are risk-reduction, not a guarantee.
12What's the difference between zero-shot, few-shot, and fine-tuned approaches to a new task?▾
Zero-shot: describe the task in the prompt with no examples, relying entirely on the model's pretrained knowledge. Few-shot: include a handful of input/output examples directly in the prompt to demonstrate the desired format and judgment, still with no weight changes. Fine-tuned: continue training the model's weights on a labeled dataset specific to the task, producing a specialized model that no longer needs the task explained at inference time. Moving from zero-shot → few-shot → fine-tuning generally trades increasing setup cost and data requirements for increasing task-specific reliability.
