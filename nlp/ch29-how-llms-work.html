Chapter 29:
How LLMs Actually Work

Strip away the scale — billions of parameters, trillions of training tokens, attention over thousands of positions — and every large language model is doing one repeated thing: given all the tokens so far, predict a probability distribution over what comes next, then sample from it. This chapter builds that core loop for real, from scratch, at a scale small enough to run instantly in your browser: a character-level language model you train, sample from, and control with temperature and top-k — the exact same mechanics as GPT-4 or Claude, just tiny.

🕐 ~55 min read 💻 5 exercises 🏗️ 1 mini project 🧪 Quiz to complete
29.1 — Tokenization for LLMs: Words vs. Subwords

Chapter 27 tokenized on whole words. Real LLMs almost always use subword tokenization instead (algorithms like BPE — Byte-Pair Encoding — or SentencePiece): common words stay as one token, but rare or unfamiliar words get split into smaller, frequently-seen pieces. "unbelievable" might become ["un", "believ", "able"] rather than one rare whole-word token or 12 individual characters.

ApproachVocabulary sizeHandles unseen words?
Word-level (Ch 27)Huge (every word is one entry)No — unseen word = unknown token
Character-levelTiny (~100 entries)Always, but sequences get very long
Subword / BPEMedium (10k-100k entries)Yes — falls back to smaller known pieces
💡
Subword tokenization is why LLMs never truly "fail" on a weird word — a made-up word like "quantumify" still decomposes into familiar pieces the model has seen before, just combined in a new way. This chapter won't implement BPE (that's a distinct algorithm on its own), but the char-level model you're about to build sits at one extreme end of this same spectrum — and shares the same "predict the next unit" objective as a full subword-based LLM.
29.2 — Next-Token Prediction: The One Job Every LLM Has

Every LLM is trained on exactly one objective: given a sequence of tokens so far, output a probability distribution over every possible next token. Training adjusts the model's weights so that the probability it assigns to the ACTUAL next token (from real text) goes up. That's it — no other objective. Everything else (chatting, coding, reasoning) emerges from doing this one task extremely well over enormous amounts of text.

Conceptual: what a next-token distribution looks like
# Given the prompt "The cat sat on the", a trained model outputs a distribution # like this over its ENTIRE vocabulary (shown here for just 5 candidate words): next_token_probs = { 'mat': 0.42, 'floor': 0.18, 'chair': 0.09, 'roof': 0.02, 'moon': 0.001, } most_likely = max(next_token_probs, key=next_token_probs.get) print(f'Most likely next word: {most_likely}')
⭐
"mat" outranks "moon" not because the model "understands" cats — it's because "the cat sat on the mat" appeared (or near-equivalents did) far more often in training text than "the cat sat on the moon". Next-token prediction is fundamentally a statistical pattern learned from data, however sophisticated the resulting behavior looks.
29.3 — A Real Character-Level Bigram Language Model

Let's build the smallest possible version of "predict the next token from what came before": a character-level bigram model. It looks at ONE character and predicts a probability distribution over the NEXT character, learned purely by counting how often each character-pair appeared in a training text, then normalizing counts into probabilities.

Training: counting character pairs, then normalizing to probabilities
training_text = "she sells seashells by the seashore" chars = sorted(set(training_text)) print(f'Vocabulary ({len(chars)} characters): {chars}') # count how often each character is followed by each other character counts = {c: {c2: 0 for c2 in chars} for c in chars} for i in range(len(training_text) - 1): a, b = training_text[i], training_text[i+1] counts[a][b] += 1 # normalize each row into a probability distribution probs = {} for c in chars: total = sum(counts[c].values()) probs[c] = {c2: counts[c][c2]/total for c2 in chars} # show only the non-zero next-character probabilities after 's' after_s = {k: round(v,4) for k,v in probs['s'].items() if v > 0} print(f"P(next | 's'): {after_s}")
💡
This "count then normalize" recipe is the entire model — there's no gradient descent, no neural network here, because a bigram model's entire "knowledge" fits in one table of conditional probabilities. Real LLMs replace this table with a neural network's output layer, but the underlying question — "given the recent context, what's the probability of each possible next token?" — is identical.
Generating new text by sampling from the model
import random def generate(start_char, length, probs, chars, seed): rng = random.Random(seed) # fixed seed -> fully reproducible output result = [start_char] current = start_char for _ in range(length - 1): weights = [probs[current][c] for c in chars] current = rng.choices(chars, weights=weights, k=1)[0] # sample, weighted by probability result.append(current) return ''.join(result) # probs, chars from the training step above generated = generate('s', 40, probs, chars, seed=42) print(f'Generated: {generated}')
⚠️
"she shellllls by sellshe the thorellshel" looks eerily plausible, letter by letter, yet is meaningless overall — that's exactly what a model with a one-character memory produces: locally plausible, globally incoherent. Real LLMs use attention (Chapter 25's Transformers Explained territory) to condition on THOUSANDS of previous tokens instead of just one previous character, which is most of why their output holds together over long passages.
29.4 — Sampling Strategies: Temperature and Top-k

A model's raw output before probabilities are logits — unbounded scores, one per vocabulary entry, turned into a probability distribution with softmax. Two knobs control HOW that distribution gets sampled from: temperature reshapes the distribution's sharpness before sampling; top-k restricts sampling to only the k highest-probability candidates.

Temperature: reshaping the distribution before sampling
import numpy as np def softmax(x): e = np.exp(x - np.max(x)) # subtract max first for numerical stability return e / e.sum() def softmax_temp(logits, T): return softmax(logits / T) # dividing logits by T before softmax IS temperature scaling vocab_words = ['the', 'cat', 'sat', 'dog', 'ran'] logits = np.array([2.0, 1.0, 0.5, 1.5, 0.2]) print(f'T=1.0 (baseline): {[round(float(x),4) for x in softmax_temp(logits, 1.0)]}') print(f'T=0.5 (sharper): {[round(float(x),4) for x in softmax_temp(logits, 0.5)]}') print(f'T=2.0 (flatter): {[round(float(x),4) for x in softmax_temp(logits, 2.0)]}')
⭐
Low temperature (<1) sharpens the distribution toward the top candidate — more deterministic, more repetitive. High temperature (>1) flattens it — more random, more "creative", but more likely to go off the rails. As T → 0, the distribution collapses onto a single spike at the highest-logit token — sampling becomes equivalent to always picking the single most likely token (greedy/argmax decoding).
Top-k: keep only the k most likely candidates, renormalize
def top_k_filter(probs, k): idx_sorted = np.argsort(probs)[::-1] # indices, highest probability first keep_idx = idx_sorted[:k] filtered = np.zeros_like(probs) filtered[keep_idx] = probs[keep_idx] # zero out everything outside the top k return filtered / filtered.sum() # renormalize so probabilities sum to 1 again p = softmax_temp(logits, 1.0) print(f'Top-3 filtered: {[round(float(x),4) for x in top_k_filter(p, 3)]}') print(f'Top-2 filtered: {[round(float(x),4) for x in top_k_filter(p, 2)]}')
🚫
Forgetting to renormalize after top-k filtering is a real bug — zeroing out the low-probability entries without dividing by the new (smaller) sum leaves you with numbers that no longer add up to 1, which isn't a valid probability distribution to sample from.
29.5 — Prompting: Zero-Shot, Few-Shot, and Chain-of-Thought

Since a next-token predictor's whole behavior comes from the text it's conditioned on, HOW you phrase that text (the prompt) dramatically changes its output — no weight updates required. Three patterns come up constantly:

  • Zero-shot: just ask directly — "Classify this review as positive or negative: 'The food was cold.'" No examples given; the model relies entirely on what it already learned.
  • Few-shot: show a handful of input/output examples in the prompt before the real question — "'Great service!' → positive. 'Rude staff.' → negative. 'The food was cold.' → ?" The examples steer the model toward the exact format and judgment style you want, with no retraining.
  • Chain-of-thought: ask the model to reason step-by-step before giving a final answer — "Think through this step by step, then answer." Especially for arithmetic or multi-step logic, spelling out intermediate reasoning tends to produce more accurate final answers than jumping straight to a conclusion.
💡
None of this changes the model's weights — it's the exact same next-token-prediction mechanics from 29.2, just with a cleverer prompt steering which tokens become likely. Chapter 30 puts this into practice against a real hosted LLM API.
Chapter 29 Summary
  • Subword tokenization (BPE/SentencePiece) balances vocabulary size against never hitting a truly "unknown" word.
  • Next-token prediction — a probability distribution over the next token given everything before it — is the ONE training objective behind every LLM.
  • A character-level bigram model learns this by counting character-pairs and normalizing into a probability table — the simplest possible real example.
  • Temperature reshapes distribution sharpness before sampling; top-k restricts sampling to the k most likely candidates.
  • Zero-shot, few-shot, and chain-of-thought prompting steer a frozen model's output purely through how the input is phrased.
